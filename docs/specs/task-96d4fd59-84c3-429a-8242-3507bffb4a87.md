# M4 — Albums & labels API

Expose REST endpoints to manage albums and labels and to browse their photos with the standard
filters/sorting/pagination.

## Context
Read `docs/ARCHITECTURE.md` §5, §13 (filters everywhere). Depends on the albums/labels
repositories and the photo list machinery.

## Requirements
- Albums (editor/admin for mutations; all roles read):
  `GET /api/v1/albums` (list with counts + cover), `POST /api/v1/albums`,
  `GET /api/v1/albums/{uid}`, `PATCH /api/v1/albums/{uid}` (title/description/cover/order_by/
  private), `DELETE /api/v1/albums/{uid}`,
  `POST /api/v1/albums/{uid}/photos` (add), `DELETE /api/v1/albums/{uid}/photos` (remove),
  `PATCH /api/v1/albums/{uid}/order` (reorder), and listing an album's photos via the shared
  photo-list endpoint scoped by album (with filters/sort/pagination).
- Labels (same role rules): `GET /api/v1/labels` (with counts), `POST`, `PATCH`, `DELETE`,
  attach/detach labels to a photo, and list photos by label (filters/sort/pagination).
- Validate inputs; 404 on missing; enforce roles. Keep responses consistent with the photo-list
  shape so the frontend reuses the grid.

## Quality gate (mandatory)
- Use the **golang-developer** skill. `make check` MUST pass.
- Integration tests (test DB): album create/update/delete, add/remove/reorder photos, scoped
  listing with filters/sort/pagination; label CRUD + attach/detach + listing; role enforcement
  (viewer read-only).
