# Replace Default Chrome With the App's Own Look

Two places where un-themed default UI shows straight through the app's skin.

## Part 1 — Form controls are blinding white

Every input and select renders pure white on the app's navy background. On the library that means four empty white rectangles (Rok / Album / Štítek / Osoba) that are the loudest thing on a page whose whole job is showing photographs — the eye goes to the empty filters, not the pictures. Same story on the photo detail ("Přidat do alba", "Přidat štítek") and on the login card.

**Read this before touching anything.** `docs/UX_AUDIT.md` records a deliberate REJECTION here: `color-scheme: dark` was tried and rejected because Bootswatch Superhero paints fields white regardless of theme, so the NATIVE bits — the select arrow, the date-picker glyph — render invisible against white. The controls are pinned to `color-scheme: light` on purpose. Do NOT simply flip that back; you would re-break exactly what that pin fixed.

The job is to give form controls a real dark surface consistent with the token layer AND keep every native glyph visible — both together. If you conclude the two cannot both hold, STOP and report why rather than shipping invisible select arrows.

### Requirements

- Inputs, selects and textareas read as part of the dark UI, not as holes punched in it. Their visual weight sits BELOW the photographs.
- Every native affordance stays visible and usable: select arrow, date-picker glyph, text caret, autofill styling, placeholder text.
- Placeholder and value text keep readable contrast. This is not licence to make text muddy — the audit already flags muted `text-secondary` contrast as an open gap; do not make it worse.
- Focus rings survive exactly as they are. `--kk-focus-ring-color` (theme info cyan, 3px, 2px offset) was chosen to read against both the navy page and an arbitrary photo thumbnail. The focus outline is never removed, and it stays an `outline` (not a box-shadow) so it survives `overflow:hidden`.
- Disabled and invalid states stay distinguishable from resting ones.
- Superhero stays the base theme — a binding rule in `CLAUDE.md`. `tokens.css` is a layer on top and "never redefines Bootstrap's own `--bs-*` variables except where a component opts in, so upgrading Bootswatch stays safe". If this genuinely requires opting in at the `--bs-*` level, do it narrowly and comment why. Do not reach for `!important` as a first resort.
- Do NOT disturb the two other documented Superhero traps: the navy re-pin of `--bs-body-bg`/`--bs-body-color` (dropping it gives charcoal and loses the identity), and `data-bs-theme="dark"` staying on (without it Superhero leaves tertiary/secondary surfaces light → white-on-white invisible labels). Also leave `--kk-surface-raised` alone: Superhero's own card colour `#4e5d6c` was rejected as the raised surface because it is also `secondary`, which makes an `outline-secondary` button on it invisible.

## Part 2 — Four native browser confirm dialogs

`window.confirm` is still used in exactly four places: `pages/SavedSearchesPage.tsx:51`, `pages/LabelsPage.tsx:50`, `pages/ImportPage.tsx:276`, `pages/AlbumDetailPage.tsx:149`. Each pops the browser's own grey OS dialog into the middle of an otherwise styled app. This is item #2 on `docs/UX_AUDIT.md`'s own backlog and is still open.

### Requirements

- Add ONE shared confirm dialog component and use it at all four call sites. `/trash` already has "the only page with a properly styled confirm Modal" and the audit names it the model to follow — follow it rather than inventing a second pattern.
- The dialog names what will happen, in the app's voice: the confirm button carries the action ("Smazat album"), never "OK". An action keeps the same name through the whole flow.
- Destructive confirms look destructive, and the destructive button is not the one a stray Enter press hits by default.
- Keyboard and focus behave: focus moves into the dialog, Escape cancels, focus returns to where it came from on close.
- No hardcoded user-facing strings.

## Constraints

- All user-facing text via i18n, Czech + English parity (`i18n.test.ts` enforces it).
- Icons only from `bootstrap-icons` via the `Icon` component; decorative ones `aria-hidden`.
- Use `web/src/styles/tokens.css`: `--kk-radius-lg` for panels/modals, `--kk-space-*`, and elevation `--kk-shadow-*` — note elevation here is a PAIR, drop shadow plus an `inset 0 1px 0` top highlight ring, because a plain dark shadow is nearly invisible against dark navy. Motion via `--kk-duration-*` with `--kk-ease-standard`.
- Keep the 44px touch floor under `@media (pointer: coarse)`.
- Under `prefers-reduced-motion`, decorative transforms are zeroed but spinners and progress bars KEEP animating — they carry the meaning that something is happening. Do not kill motion wholesale.
- `docs/UX_AUDIT.md` predates `tokens.css`; where they disagree, tokens win (its `h1.h3` advice is superseded — no `h3`/`h5`/`fs-*` utilities on headings).
