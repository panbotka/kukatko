# M2 — Persistent job queue

Implement a durable, Postgres-backed job queue that survives restarts — the core robustness
improvement over photo-sorter's in-memory jobs.

## Context
Read `docs/ARCHITECTURE.md` §8 (async jobs & box offline), §5.2 (`jobs` table). Jobs must
survive restarts, retry with backoff, dedupe, and degrade gracefully when the embeddings box is
offline (those jobs simply wait). Use the existing migration runner + pgx layer.

## Requirements
- Migration for `jobs`: id BIGSERIAL PK, type TEXT, state TEXT (queued/running/done/failed/dead),
  priority INT, payload JSONB, attempts INT, max_attempts INT (default 5), last_error TEXT,
  run_after TIMESTAMPTZ, locked_by TEXT, locked_at TIMESTAMPTZ, created_at, updated_at.
  Index on (state, run_after, priority). **Dedup**: partial unique index on
  (type, (payload->>'photo_uid')) WHERE state IN ('queued','running').
- `internal/jobs` package:
  - `Enqueue(ctx, type, payload, opts)` — idempotent w.r.t. the dedup key (no duplicate active
    job for the same type+photo).
  - `Claim(ctx, workerID, types...)` — atomically pick the next runnable job using
    `SELECT ... FOR UPDATE SKIP LOCKED` (respect run_after, priority DESC, FIFO), mark running +
    locked_by/locked_at.
  - `Complete`, `Fail(err)` (increment attempts; if attempts<max → requeue with **exponential
    backoff** via run_after; else state=dead + last_error), `Heartbeat`/stale-lock recovery
    (running jobs whose lock is stale get requeued).
  - Query helpers: counts by state/type, list dead-letter, requeue a dead job.
- No execution loop here (that is the worker-runtime task); this task is the storage + claim
  semantics + tests.

## Quality gate (mandatory)
- Use the **golang-developer** skill. `make check` MUST pass.
- Integration tests (test DB): enqueue + dedup (no duplicate active job), claim ordering
  (priority/run_after/FIFO), SKIP LOCKED concurrency (two claimers never get the same job),
  retry/backoff increments + dead-letter after max attempts, stale-lock recovery.
