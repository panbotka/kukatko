# Make bulk upload progress clear and usable on mobile

Uploading a large batch — e.g. 100 photos after an event — is currently hard to follow,
especially on a phone. The upload page (`web/src/pages/UploadPage.tsx`) renders every file as a
row in a Bootstrap `ListGroup` (`web/src/components/upload/UploadItem.tsx`) with its own per-file
progress bar, plus a single text "counts" summary line (created / duplicate / error / pending).
There is **no overall progress indicator**, so with 100 files the user sees a giant scrolling list
and cannot tell at a glance how far along the batch is or whether it is still working. Uploads run
≤3 concurrently via `web/src/hooks/useUploadQueue.ts`, each file a separate `POST /api/v1/upload`
request whose XHR reports per-file progress; the backend already returns a per-file result
(created / duplicate / error / warnings).

Redesign the upload progress presentation so a big batch is legible on both desktop and mobile.

## Requirements

- **A prominent overall progress header, always visible while a batch is running.** It shows: how
  many files are done out of the total (e.g. "37 / 100"), a single overall progress bar
  reflecting the whole batch (a file that is mid-upload contributes its partial fraction so the
  bar moves smoothly, not just in whole-file steps), and a compact live breakdown of the outcome
  counts (uploaded / duplicates / errors / remaining). This header stays pinned/visible as the
  per-file list scrolls, and on mobile it is the thing the user actually watches.
- **The per-file list must stay usable at 100+ items.** Virtualize or otherwise bound the list so
  the page stays responsive on a phone (the app already uses `react-virtuoso` for long lists), and
  make failures easy to find — e.g. let the user filter/jump to just the errored files, or float
  the errored rows to the top, so a handful of failures in a batch of 100 are not lost in the
  scroll. A "retry failed" affordance for the whole batch must remain (it already exists in the
  hook).
- **Clear terminal state.** When the batch finishes, the header switches to a clear completed
  summary (e.g. "98 uploaded, 2 duplicates, 0 errors") with the existing link to view the library,
  and a one-tap way to retry if anything failed.
- **Mobile-first layout.** On a narrow viewport the overall header, the counts, and the actions
  (retry, view library, clear) must all be reachable without horizontal scrolling and without the
  per-file rows overwhelming the screen. Buttons and touch targets sized for a phone.
- Preserve all existing behaviour: drag-drop and the camera-capture input, the ≤3 concurrency,
  per-file retry and remove, near-duplicate warnings, and the post-upload batch album/label
  assignment (`UploadOrganize`). Do not change the upload API contract or the concurrency model;
  this is a presentation/UX task.

## Verification

`make check` must pass. Extend the upload tests to assert: the overall header reports the correct
done/total and aggregate counts as items move through queued → uploading → created/duplicate/error;
the overall bar reflects partial in-flight progress; errored files can be surfaced/filtered; and the
completed summary renders with a working retry-failed action. Update `docs/FRONTEND.md`.
