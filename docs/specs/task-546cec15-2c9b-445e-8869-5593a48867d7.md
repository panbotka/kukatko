# Fix: selecting the first photo flickers the whole grid

In multi-select, picking the FIRST photo (selection goes from empty to one) makes every visible thumbnail in the grid blink/fade at once. Read `docs/FRONTEND.md` (photo grid / multi-select) first.

## Root cause (confirmed)

- `web/src/components/library/PhotoTile.tsx` (~lines 172-200) chooses the tile's ROOT element based on `selectFirst`: `<button>` when selection-first is active, otherwise `<Link>`.
- In `web/src/components/library/PhotoGrid.tsx` (~lines 199-202), for the library `selectFirst` reduces to "any photo selected" — so going 0→1 selected flips `selectFirst` for EVERY visible tile at once.
- Changing a tile's root element type (`<a>` ↔ `<button>`) makes React unmount the whole tile subtree and mount a new one — a brand-new `<img>` per tile. Each new image mounts unloaded and re-runs the opacity-0→1 fade (`web/src/components/FadeInImage.tsx` + the `.kk-media-img` transition in `web/src/styles/app.css` ~508-521) → the whole grid blinks together. It happens again in reverse when deselecting the last item.
- The grid itself does not remount (stable `computeItemKey`); it is the per-tile element swap that rebuilds the images.

## Requirements

- Make the tile keep a STABLE root element (and a stable `FadeInImage`/`<img>` subtree) regardless of whether selection-first mode is on, so toggling selection from empty↔non-empty does not unmount/remount tiles and does not re-trigger the image fade-in across the grid.
  - Approach (implementer's choice, but preserve behavior): render one consistent wrapper element and switch click behavior (navigate vs toggle-select) via handlers/role/attributes instead of swapping `<Link>` for `<button>`; or restructure so the image subtree is hoisted out of the element that changes. Keep keyboard accessibility and semantics reasonable (a selectable tile stays keyboard-operable and screen-reader sensible), and keep normal navigation (click opens the photo) working when not selecting.
- No visible fade/blink of already-loaded thumbnails when the first photo is selected or the last is deselected. Already-loaded images must stay loaded (no `src` reset / no remount).
- Keep the existing selection behavior intact (hover-select, click-to-select, range/anchor). This is a rendering-stability fix, not a behavior change.

## Tests

- Vitest: selecting the first photo does not remount tiles — assert the tile root element type / image element identity is stable across an empty→non-empty selection transition (or that `selectFirst` no longer swaps the element type). Assert selection still works (clicking selects; navigation still works when not selecting). Where feasible, assert already-loaded images are not reset.
