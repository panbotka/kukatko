# M6 — Trash purge + trash UI

Implement the scheduled hard-delete of archived photos after the retention period and a trash UI
to review, restore, or permanently delete.

## Context
Read `docs/ARCHITECTURE.md` §5 (archived_at), §14 (trash.retention_days). Photos are soft-deleted
via `archived_at`; nothing currently hard-deletes them. Depends on photos repository, storage,
scheduler, and FK cascades.

## Requirements
- **Purge scheduler**: a periodic job (within `kukatko serve`) that hard-deletes photos whose
  `archived_at` is older than `trash.retention_days`: remove DB rows (cascading embeddings/faces/
  markers/album_photos/photo_labels/phashes/edits/favorites), delete originals + cached
  thumbnails from disk, and (if configured) the corresponding S3 objects. Idempotent; logs counts.
- **Manual controls** (admin/editor): `POST /api/v1/photos/{uid}/purge` (permanent delete now)
  and `POST /api/v1/trash/empty` (purge all archived now), with explicit confirmation semantics.
- **Trash API**: list archived photos (reuse photo-list shape, `archived=true`).
- **Trash UI**: a page listing archived photos with **restore** (unarchive) and **permanently
  delete** actions (single + bulk), showing how long until auto-purge. i18n (cs/en), role-gated.
- No dangling files: purge must leave no orphan originals/thumbnails/S3 objects.

## Quality gate (mandatory)
- Use the **golang-developer** skill for backend; `make check` MUST pass (incl. frontend ESLint/
  Vitest).
- Integration tests (test DB): retention purge deletes only expired archived photos and all
  dependent rows + files (no orphans); manual purge/empty; restore unarchives. Vitest: trash UI
  restore/delete flows + countdown rendering. Mock API for frontend.
