# Maintainer-triggered purge of old audit logs

Give a maintainer a way to delete old audit-log entries (retention), e.g. "delete audit logs older than 1 year". No purge exists today — the audit log is append-only. Read `docs/PACKAGES.md` (audit), `docs/API.md` (audit), and `docs/OPERATIONS.md` first.

## Backend

- **Store method:** add `PurgeOlderThan(ctx, cutoff time.Time) (int, error)` to the `audit.Store` in `internal/audit` — a single `DELETE FROM audit_log WHERE created_at < $1`, returning the number of rows deleted (backed by the existing `idx_audit_log_created_at` index).
- **Endpoint (maintainer-only):** expose it via a maintainer-gated route in `internal/maintenanceapi` (the established home for destructive ops, already wired with `RequireMaintainer` in `cmd/kukatko/maintenance.go`) — e.g. `POST /maintenance/audit/purge` taking a cutoff in the body (accept a retention period such as `older_than_days`, or an explicit ISO cutoff date — pick one, validate it, reject non-positive/missing). Return the deleted count. Inject the audit store (or a purge function) into the maintenanceapi config.
- **Self-audit the purge:** write one audit record for the purge action itself (e.g. `audit.purge`) with the applied cutoff, the deleted count, and the acting maintainer — this record is recent so it survives the purge. Deleting the audit trail is sensitive; the purge must be traceable.

## Frontend

- Add a maintainer control to `web/src/pages/MaintenancePage.tsx` (maintainer-gated): a "Purge audit log" card offering a retention choice (presets e.g. 3 months / 6 months / 1 year / 2 years, or a custom period), a **confirmation step** (destructive), a Run button, and a result message showing how many entries were deleted (reuse the page's existing async-action + `Alert` feedback pattern). Add a `web/src/services/` function calling the endpoint.
- **i18n:** keys in `web/src/i18n/locales/{cs,en}/common.json`, Czech default.

## Tests

- Backend: integration test seeding audit rows with varied `created_at`; assert `PurgeOlderThan(cutoff)` deletes only older rows and returns the count; assert the endpoint requires maintainer (editor/admin 403, maintainer 200), validates the cutoff, and writes the self-audit record.
- Frontend: Vitest — the purge card requires confirmation, calls the service with the chosen retention, shows the deleted count, and is hidden for non-maintainers.
