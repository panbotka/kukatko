# Fix photo detail: page crashes after saving metadata

Saving the "Popis a místo" panel on the photo detail page appears to fail with an error. The PATCH actually succeeds (200, data persisted) — the page then crashes, so the user reads it as a failed save.

## Root cause (already diagnosed — verify, then fix)

- `internal/photoapi/update.go:78-79` — `handleUpdate` writes the bare `photos.Photo`.
- `internal/photoapi/http.go:390-393` — `handleDetail` writes the richer `photoDetail` (`photoView` + `Files`, `Albums`, `Labels`, `Uploader`).
- So the PATCH response is missing `albums`, `labels`, `files`, `is_favorite`, `uploader`.
- `web/src/services/photos.ts:417-430` casts that body to `PhotoDetail`; `MetadataPanel.tsx:184` feeds it to `setPhoto` in `PhotoDetailPage`, replacing the complete detail.
- `OrganizePanel.tsx:75,81` then does `photo.albums.map(...)` / `photo.labels.map(...)` on `undefined` → `TypeError`, blank page.

Same crash also hits the location-clear action and any other caller of `updatePhoto`.

## Requirements

- `PATCH /api/v1/photos/{uid}` returns the **same body shape** as `GET /api/v1/photos/{uid}` (full `photoDetail`, including `albums`, `labels`, `files`, `is_favorite`, `uploader`).
- Fix on the **backend**: extract the `ListFiles` + `annotate` + `photoMemberships` + `resolveUploader` block from `http.go:370-393` into a shared helper and call it from `handleUpdate`. Do not paper over it in the frontend.
- After saving description and/or coordinates the detail page stays alive, shows the saved values, and the "Uspořádání" panel still lists albums and labels.
- Also fix these silent data-loss bugs in the same form (each is a separate, verifiable requirement):
  - `MetadataPanel.tsx:26-37` `toLocalInput` truncates capture time to minutes (`00:33:39` → `00:33:00`) and flips `taken_at_source` from `exif` to `manual` on every save. Preserve seconds; do not mark the field manual when the user did not change it.
  - `web/src/lib/coordinates.ts:171` round-trips coordinates through a 6-decimal text field, rewriting `16.7083583333333` → `16.708358` on every save. Do not re-send an unchanged coordinate.
  - If the coordinate field holds unparseable text, `buildPatch()` returns `null` and the whole form (description included) refuses to save with a generic error. Report a field-level validation error on the coordinate field instead and let the other fields save.

## Tests

- Go: an integration test asserting the PATCH response carries `albums`/`labels`/`files` — the current tests only assert scalar fields on the bare `Photo` and never PATCH `description`/`notes`.
- Frontend: `MetadataPanel.test.tsx` currently mocks `updatePhoto` with a complete fake `PhotoDetail`, which is exactly why this slipped through. Make the mock return what the API really returns, and assert the page survives a save.
