# Photo detail — make the IPTC credit fields editable

The `photos` table carries `subject`, `artist`, `copyright`, `license`, `keywords` and `scan`, the API
accepts them on `PATCH /api/v1/photos/{uid}`, and they get filled from EXIF and from the PhotoPrism
import. What is missing is any way for a user to write or correct them — which matters most exactly where
the automatic sources are weakest: scanned and historical photos, where the photographer, the year and
the description live in someone's memory, not in the file.

**Before starting, confirm the columns exist and that PATCH accepts them.** If not, the storage task has
not run yet — add what you need rather than blocking, following the same shape (TEXT NOT NULL DEFAULT '',
`scan` BOOLEAN NOT NULL DEFAULT false).

## Requirements

Extend `web/src/components/photo/MetadataPanel.tsx` — the existing editable form on the photo detail page
that already PATCHes title / description / ai_note / taken_at / notes / location and is gated on
`canWrite`:

- New inputs: **Předmět** (`subject`), **Umělec** (`artist`), **Autorská práva** (`copyright`),
  **Licence** (`license`), **Klíčová slova** (`keywords`), and a checkbox **Sken** (`scan`) — "this is a
  scan of a physical photo, not a camera capture".
- They save through the **same PATCH and the same save action the form already has**. Do not add a second
  save button, a second form, or a separate request.
- `keywords` is stored as one comma-separated string. Edit it as **chips**: the field shows the current
  keywords as removable chips, typing a value and pressing Enter or comma adds one, backspace on an empty
  input removes the last. On save, join back to a trimmed, de-duplicated, comma-separated string. Reuse
  the existing chip/badge convention (`badge rounded-pill` + `ENTITY_STYLE` in
  `web/src/components/entityStyle.ts`) rather than inventing a new look — but these are **not** labels and
  must not be linked to `/labels/:uid`.
- The credit fields are secondary to title/description. Put them in a **collapsed sub-section** inside the
  panel (reuse the existing chevron disclosure pattern from `TechnicalDetails`), so the common case — type
  a title, type a description — does not get buried under six more inputs.
- Viewers (read-only role) see no editing UI here, exactly as with the existing fields.

## Behaviour

- Client-side validation mirrors the server: trim whitespace, and respect the server's length caps
  (roughly 1000 chars for subject/copyright/license, 2000 for keywords, 255 for artist). Show the existing
  form's error style on a rejected save — do not invent a new error surface.
- Saving an empty value clears the field (empty string), it does not leave the old value in place.
- A failed PATCH keeps the user's typed values in the form — never silently discard input on error.

## i18n

Every new string goes into **both** `web/src/i18n/locales/cs/common.json` and `.../en/common.json`
(namespace `common`). Czech is the default locale and comes first.

## Verification

- Vitest: the sub-section is collapsed on first render and expands on click; editing each field and saving
  sends one PATCH carrying the new values; keyword chips add on Enter and on comma, remove on click, and
  de-duplicate; a viewer sees no inputs; a failed PATCH keeps the typed values and shows the error.
- `make check` must pass.
- Update `docs/FRONTEND.md` (MetadataPanel) and `README.md` — being able to credit and describe scanned
  photos is a user-visible feature.

## Out of scope

- Bulk editing these fields across a selection (`internal/bulk` and `BulkEditModal` stay untouched).
- Turning keywords into labels — they are a separate field by design.
- Editing machine-derived fields (software, colour profile, codec, camera serial, original name,
  projection) — those stay read-only.
- Writing metadata back into the original file on disk.
