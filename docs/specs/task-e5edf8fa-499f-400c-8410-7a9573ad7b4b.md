# Inline album and label creation

Wherever a text field lets the user add photos to an album or a label, it must also let them create a new one. Today the bulk edit dialog can only pick from albums and labels that already exist.

## Requirements

- Typing a name that matches no existing album (or label) surfaces a "Create <name>" entry at the end of the multi-select's option list.
- Choosing that entry creates the album or label and then adds the selected photos to it, as one user action.
- The newly created entry appears selected in the field immediately, with no page reload.
- Creation is offered only to users who may write.
- The behaviour is a property of the album/label multi-select itself, so it appears everywhere that field is used, not only in one dialog.

## Implementation notes

- The bulk photo endpoint requires an already-existing album or label identifier and rejects unknown ones. The frontend therefore creates the entity first, then submits the bulk operation with the returned identifier. Do not change the bulk endpoint's contract to accept names.

## Edge cases

- A name differing from an existing entry only by surrounding whitespace or letter case offers the existing entry instead of creating a duplicate.
- An empty or whitespace-only name offers no create entry.
- A creation failure (duplicate name, permission denied) surfaces the server's message and leaves the field's other selections intact, so the user can retry without redoing their work.
- If the entity is created but the subsequent bulk add fails, the user is told the album/label exists and only the assignment failed.

## Tests

- Component tests: the create entry appears only for an unmatched non-empty name and never for a case-insensitive match; the entity is created and then the photos assigned; a failed creation preserves prior selections; the control is absent for a viewer.
