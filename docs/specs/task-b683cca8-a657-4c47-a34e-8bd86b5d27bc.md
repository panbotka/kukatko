# People detail: "photos where this person may appear" (candidates) section

Add a section to the People/Subject detail page (`/people/{uid}`, `web/src/pages/SubjectPage.tsx`) that surfaces photos where the system thinks the subject appears by face similarity but which are **not yet tagged** with them — letting the user confirm or reject each. This is the existing **candidates** feature (`POST /subjects/{uid}/candidates`), the mirror of the `Outliers` section already embedded on this page. Read `docs/API.md` (candidates endpoint) and `docs/FRONTEND.md` (faces/candidates + SubjectPage) first.

## What already exists (reuse it — do NOT rebuild)

- Service `searchCandidates(subjectUid, {threshold?, limit?})` and types (`Candidate`, `CandidateResult`, `CandidateAction`, `FaceBox`) in `web/src/services/faces.ts`.
- Review lifecycle hook `web/src/hooks/useCandidateReview.ts` (+ pure model `web/src/lib/candidateReview.ts`): optimistic confirm/reject, buckets, `buildAssignRequest` → `assignFace`, `buildRejection` → `rejectFace`.
- Presentational components `web/src/components/faces/CandidateCard.tsx` and `CandidateFaceImage.tsx` (self-contained: take a `Candidate` + review callbacks; render the full frame with the face boxed, match %, vote count, ✓/✗ controls).
- The full workspace `/faces` (`web/src/pages/FacesPage.tsx`) does this globally; the embedded `Outliers` component (`web/src/components/people/Outliers.tsx`, rendered in `SubjectPage.tsx`) is the pattern to mirror for an on-page, per-subject section.

## Requirements

- Add a new section to `SubjectPage.tsx`, below the gallery (near/after the existing `Outliers` section), gated by `canWrite` (editor+) exactly like `Outliers`. Suggested heading e.g. "Možná je i zde" / "Návrhy podle podobnosti" (add i18n keys, Czech default + English).
- The candidate search is **expensive** — do NOT run it automatically on page load. Trigger it explicitly: render the section with a "Najít návrhy" button that runs `searchCandidates(subject.uid, {})` (sensible default threshold/limit) on click; show a loading state while it runs and allow re-running. A single button with defaults is acceptable for v1; a small threshold/limit control is optional.
- Render results as candidate cards reusing `CandidateCard` + `useCandidateReview`, each with **confirm** (assign the face to this subject) and **reject** ("not this person") actions via the existing `assignFace` / `rejectFace` write paths — the same optimistic lifecycle as `/faces`. Respect the candidate `action`/`marker_uid` classification when confirming (assign to an existing marker vs create one) — `useCandidateReview`/`buildAssignRequest` already handle this.
- Handle the response summary states gracefully: `reason` cases (`no_faces` — the subject has no tagged faces yet to search from; `no_embeddings` — the box was offline) must show a clear message rather than an empty void. Optionally show the applied `min_match_count`/`threshold` context (as `/faces` does), but keep the embedded section simpler than the full workspace.
- After a candidate is confirmed, it should leave the list; the subject gallery/counters should reflect the new assignment on next load — reuse the page's existing reload mechanism if straightforward.
- Optionally offer a link to the full `/faces?subject={uid}` workspace (it pre-fills and auto-runs) for power users — but the on-page section is the deliverable.

## Tests

- Vitest: the section renders the trigger button, runs the mocked `searchCandidates`, renders candidate cards, confirms/rejects a candidate (asserting the correct write path is called and the card leaves the list), and shows the `no_faces`/`no_embeddings` messages. The section is hidden for non-write users.
