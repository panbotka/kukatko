# API Tokens

Kukátko can only be authenticated with a browser cookie. `authenticateRequest` reads the `kukatko_session` cookie and nothing else — there is no `Authorization` header path, no API key, no personal access token. The only non-cookie credential is a per-session `download_token` accepted via `?t=` on three media endpoints. A non-interactive client — a CLI, a script, an agent — therefore has no way in short of storing a username and password and replaying a cookie.

Add long-lived API tokens.

## Credential

- Format `kkt_<id>_<secret>`, where `secret` carries at least 256 bits from a cryptographically secure source. The `<id>` prefix indexes the row so verification is a single lookup, not a table scan.
- Store only a hash of the secret. The plaintext is returned exactly once, at creation, and never again.
- **Hash with SHA-256, not bcrypt.** Passwords use bcrypt, and copying that here would be wrong: bcrypt is deliberately slow and a token is verified on *every* API request, not once per login. Bcrypt's cost defends low-entropy secrets against dictionary attack; a 256-bit random secret has no dictionary. Leave a comment saying so, or the next reader will "fix" it.

## Model

- A token belongs to a user and inherits that user's role. Do not introduce a second permission system: the existing `RequireAuth` / `RequireWrite` / `RequireAdmin` middleware must keep working unchanged.
- Fields: a human-readable name, `expires_at`, `last_used_at`, `revoked_at`.
- Update `last_used_at` at most once a minute, mirroring the guard the sliding session already uses, so a busy client does not write on every request.
- A migration creates the table with the usual `ON DELETE CASCADE` from the owning user.

## Authentication

- `authenticateRequest` gains one branch: accept `Authorization: Bearer <token>` alongside the cookie. The cookie path must not change — the frontend depends on it.
- A revoked, expired, or unknown token is a `401`, never a `403`, and the response body must not reveal which of the three it was.
- Constant-time comparison of the hash.

## Endpoints

- `POST /api/v1/auth/tokens` — create; returns the secret once.
- `GET /api/v1/auth/tokens` — list the caller's own tokens, never the secrets.
- `DELETE /api/v1/auth/tokens/{id}` — revoke. A user may only revoke their own tokens; an admin may revoke anyone's.
- Every mutation writes an audit entry through `internal/audit` in the same transaction as the change, per the project rule.
- Rate-limit token creation, reusing the existing limiter.
