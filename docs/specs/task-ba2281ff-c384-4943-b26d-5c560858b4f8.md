# Trash retention: 1-year default + admin manual purge by age

Two backend changes to the trash subsystem: raise the automatic retention default to one year, and add an admin-only endpoint to manually purge archived photos older than a caller-specified number of days.

## Background
Photos have a three-state lifecycle: active → archived (trash, `archived_at` set) → purged (permanent, irreversible). A background goroutine in `internal/trash` (`RunPurge`, every 6h) calls `PurgeExpired`, which hard-deletes archived photos whose `archived_at` is older than the retention cutoff. Retention is configured by `trash.retention_days` (`TrashConfig.RetentionDays` in `internal/config/config.go`), default currently **30**, documented in `config.example.yaml` and `docs/OPERATIONS.md`. A value ≤ 0 disables the scheduled purge.

Existing admin trash endpoints live in `internal/photoapi/trash.go` with routes registered in `internal/photoapi/http.go`:
- `POST /api/v1/photos/{uid}/purge?confirm=true` — permanent single-photo purge, `RequireAdmin`.
- `POST /api/v1/trash/empty?confirm=true` — permanently empties the whole trash, `RequireAdmin`.
Both require `confirm=true` (else 400). Purge cascades DB rows and deletes originals, metadata sidecars, cached thumbnails and remote backup objects (see `internal/trash` `purgeOne`/`deleteArtifacts`). The store already has `ListArchivedUIDs` and the retention path uses a cutoff timestamp.

## Requirements

### 1. Change the automatic retention default to 365 days (1 year)
- Change the default of `trash.retention_days` from 30 to **365** in `internal/config/config.go` (`setDefaults`) and in `config.example.yaml`.
- Update `docs/OPERATIONS.md` where the retention default is documented (30 → 365, "one year").
- Update any config test asserting the old default (e.g. `internal/config/config_test.go`).
- Keep existing semantics: `trash.retention_days` ≤ 0 still disables the scheduled purge.

### 2. New admin endpoint: purge archived photos older than N days
- Add `POST /api/v1/trash/purge-older` — **RequireAdmin**, consistent with the existing empty-trash endpoint.
- Input: the number of days `N`, passed as a query parameter alongside a required `confirm=true` (match the existing trash endpoints' style, e.g. `?days=180&confirm=true`). Without `confirm=true` return 400.
- Validate `N`: integer ≥ 0. Reject negative or non-numeric with 400. `N = 0` means "everything currently in trash" (all archived photos) — allowed, equivalent to empty-trash; document this.
- Behavior: permanently purge every archived photo whose `archived_at` is older than `now - N days`, reusing the **exact same** purge path as the existing retention/empty-trash purge (same cascade + artifact deletion in `internal/trash`). Do NOT write a second purge implementation — add a store query that lists archived UIDs older than an explicit cutoff (a variant of the existing `ListArchivedUIDs`/`PurgeExpired` age logic) and feed those UIDs through the shared purge routine.
- Audit: record the purge in the audit trail like the existing manual purge/empty-trash, attributed to the **calling admin** (not the system retention actor). Manual admin purges must be distinguishable in the audit log from automatic retention purges (which use the empty system actor with `source=retention`).
- Response: return a summary including the count of photos purged, following the existing endpoint's status-code convention (200 with body, or 204). If the purge backend is nil/unconfigured, return 503 like the existing purge handler. Do not invent new error handling for offline storage/backup — mirror the existing purge handler.

## Tests
- Integration test against the real test DB (repo convention): archive several photos with different `archived_at` ages, call `purge-older` with a cutoff, assert only the older ones are purged and newer ones remain; assert `confirm=true` is enforced (400 without it); assert RBAC (viewer/editor rejected, admin succeeds); assert an audit entry is written attributed to the admin.
- Update/extend config tests for the new 365 default.
