# Fix: archiving/purging a stack member silently hides live photos

Archiving or permanently purging one member of a stack breaks the stack's
invariants and makes OTHER, still-live members disappear from every default view.

## Root cause (verified)
- `internal/photos/store.go` `setArchivedRow` (~lines 304-326) only sets
  `archived_at`; it does NOT clear the row's `stack_uid` / `stack_primary`, nor
  re-elect a new primary. `internal/photos/audit.go` `DeleteAudited` (purge, ~58-70)
  is a bare `DELETE FROM photos`.
- The repair that restores stack invariants (`repairStackTx`,
  `internal/photos/store_stacks.go:355`) is invoked ONLY from explicit stack
  operations (store_stacks.go:206, :261 — Create/Unstack), never from archive,
  purge, or merge-archival.
- The default visibility gate `stackClauses` (`internal/photos/store_list.go:458`)
  is `(stack_uid IS NULL OR stack_primary)`, applied to List/Count/Search/FilterUIDs/
  YearBuckets/TimelineBuckets. Combined with `archived_at IS NULL`.

## Failure scenario
Stack S = {A (primary), B (live, non-primary)}. Archive A: A leaves the default
grid (correct — it's in trash), but B is now a live, non-primary member whose primary
is gone, so `(stack_uid IS NULL OR stack_primary)` excludes B from every default
view — B vanishes though it is not archived, reachable only via the stack-variants
view opened from the now-gone primary. Purging A makes it permanent: B keeps a
`stack_uid` to a one-member primary-less stack, and `ListStackCandidates` excludes
`stack_uid IS NOT NULL`, so B can never be re-stacked or re-surfaced. The scheduled
retention purge (`internal/trash/trash.go:274-289`) and dupmerge's copy-archival
(`internal/dupmerge/apply.go:152-159`) hit the same gap.

## Requirements
- When a photo with a non-null `stack_uid` is archived, purged, or archived as a
  merge copy, run the stack-repair logic (`repairStackTx`: unstack the leaving row
  and re-elect a new primary, or dissolve a remnant that drops below 2 members) in
  the SAME transaction as the mutation, so the stack never loses its primary and no
  live member is left hidden.
- Cover all three paths: `ArchiveAudited`, `DeleteAudited` (purge), the retention
  purge in `internal/trash`, and dupmerge copy-archival.
- Unarchiving does not need to re-join a stack; just don't leave live members hidden.

## Testing
- Integration tests (real test DB) for: archive the primary of a 2-member stack →
  the other member stays visible in the default list; purge it → the remnant is a
  normal standalone photo (visible, re-stackable); same via the retention purge and
  dupmerge. `make check` must pass.
