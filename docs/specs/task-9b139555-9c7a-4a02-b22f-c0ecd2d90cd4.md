# Fill the IPTC/XMP metadata columns from the file itself

The `photos` table has columns for `subject`, `artist`, `copyright`, `license`, `keywords`,
`software`, `color_profile`, `image_codec`, `camera_serial`, `original_name` and `projection`, but
nothing fills them: `internal/exif` currently maps only date, GPS, camera/lens and exposure out of the
exiftool output. Everything else lands in the raw `exif` JSONB blob and is never read again.

Teach the extractor to pull these fields out, wire them into ingest, and backfill the photos that are
already in the library.

**Before starting, confirm those columns exist.** If they do not, the migration task that adds them has
not run yet — in that case add them first, following the same shape (all TEXT NOT NULL DEFAULT '',
`scan` BOOLEAN NOT NULL DEFAULT false), rather than blocking.

## Requirements

### Extraction — `internal/exif`

Extend the `Metadata` struct and the exiftool mapping (`exiftool.go` runs `exiftool -json -n`) to
capture, with fallbacks in the listed order — first non-empty wins:

- **subject** ← `Subject`, `Headline`, `XPSubject`, `ObjectName`
- **artist** ← `Artist`, `Creator`, `By-line`, `XPAuthor`
- **copyright** ← `Copyright`, `Rights`, `CopyrightNotice`
- **license** ← `License`, `UsageTerms`, `WebStatement`
- **keywords** ← `Keywords`, `Subject` (XMP `dc:subject`), `XPKeywords` — exiftool returns these as
  either a string or a list; normalise to a **comma-separated string**, trimmed, de-duplicated,
  order preserved
- **software** ← `Software`, `CreatorTool`, `ProcessingSoftware`
- **camera_serial** ← `SerialNumber`, `BodySerialNumber`, `InternalSerialNumber`
- **color_profile** ← `ICCProfileName`, `ProfileDescription`, `ColorSpace`
- **image_codec** ← `Compression`, `FileType`/`FileTypeExtension`, or derive from the MIME type;
  normalise to a short lowercase token (`jpeg`, `heic`, `png`, `webp`, `avif`, `tiff`, `raw`)
- **projection** ← `ProjectionType` (XMP GPano) — empty for ordinary photos

Notes:
- `Subject` legitimately feeds **both** subject and keywords depending on the writer (IPTC headline vs
  XMP `dc:subject` list). Rule: if the value is a **list**, treat it as keywords; if it is a **scalar
  string**, treat it as subject. Cover this with a test — it is the one genuinely tricky mapping here.
- Values must be trimmed; drop values that are empty, `"unknown"`, or `"0"` after trimming.
- Keep the pure-Go fallback (`fallback.go`) in sync for whatever it can reach; fields it cannot read
  stay empty rather than wrong.
- Do **not** infer `scan` — it stays a manual flag the user sets.
- The full raw tag document keeps going into the `exif` JSONB blob exactly as today.

### Ingest

`internal/ingest` writes the newly extracted fields on upload, alongside the fields it already writes.
`original_name` = the file name as uploaded.

### Backfill

Photos already in the library have empty columns and their originals are still on disk / in object
storage. Provide a backfill that re-runs extraction over existing photos and updates **only** the
metadata columns:

- Follow the established pattern in `internal/processapi` (admin-only `/process/*` backfills) and the
  existing worker handlers — a job type that re-extracts metadata for one photo, plus a backfill that
  enqueues it for photos that need it.
- The backfill must be **idempotent and resumable** — safe to run twice, picks up where it stopped.
- It must **never** overwrite a non-empty user-edited value with an empty extracted one, and must never
  touch `title`, `description`, `notes`, `ai_note`, `taken_at`, `lat`/`lng`, or any curation data.
- Streams/opens originals through the existing storage abstraction — it must work for both local `FS`
  and R2 originals.

## Edge cases

- Video files: exiftool metadata is thin; do not regress the existing ffprobe-derived
  `video_codec`/`audio_codec`/`fps`/`duration_ms`. `image_codec` stays empty for videos.
- A file whose original is missing from storage: the backfill logs and skips it, it does not fail the
  whole run.
- exiftool absent on the host: the pure-Go fallback path must still produce a valid (thinner) result —
  do not panic, do not fail the upload.

## Verification

- Unit tests in `internal/exif` with table-driven cases over realistic exiftool JSON documents:
  scalar-vs-list `Subject`, keyword list normalisation and de-duplication, each fallback chain, junk
  values dropped.
- Integration test: upload a fixture with IPTC/XMP tags → the columns are populated; run the backfill
  over a photo with empty columns → they get filled; run it a second time → nothing changes and no
  user-edited value is clobbered.
- `make check` must pass.
- Document the new job type / backfill endpoint in `docs/PACKAGES.md` + `docs/API.md`, and the CLI/
  config surface (if any) in `docs/OPERATIONS.md`.

## Out of scope

- Frontend display and editing (separate tasks).
- PhotoPrism import mapping (separate task).
- Turning keywords into labels — keywords stay a standalone field.
