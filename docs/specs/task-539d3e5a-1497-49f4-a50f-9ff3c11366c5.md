# Download many photos as a ZIP

After an event a user wants "give me all these photos" in one go. Today each photo has its own
"download original" link but there is no way to download many at once. Add a bulk download that
streams the originals of a selection (or a whole album) as a single ZIP.

## Current state (for context)

- Originals live in `internal/storage` behind a `Store` interface; `Open(ctx, relPath) io.ReadCloser`
  streams an original from local FS or R2 (there is also `Materialize`/`URL`). A single photo is
  downloaded today via the media/download route.
- The library has a selection mode (`SelectionBar`, multi-select) and album pages
  (`AlbumDetailPage`) that already know a set of photo UIDs. Photo rows carry `file_name`.

## Requirements

- **Backend:** add an endpoint that takes a set of photo UIDs (and/or an album UID) and streams a
  ZIP of their **original** files. It must **stream** — write each entry into an `archive/zip`
  writer straight onto the HTTP response using `storage.Open`, never buffering whole files or the
  whole archive in memory (large libraries, `CGO_ENABLED=0`, must not blow RAM). Include videos and
  every media type; use each photo's original `file_name` as the entry name and de-duplicate
  colliding names (append ` (2)`, ` (3)`…). Set the right `Content-Type`/`Content-Disposition` with
  a sensible archive name (e.g. the album title or `kukatko-photos-<date>.zip`; a date component may
  be passed in by the caller since the server avoids wall-clock in some paths — follow existing
  conventions). Skip and report entries whose original is missing rather than aborting the whole
  archive.
- Respect RBAC and visibility exactly like the single-photo original download (whoever may download
  one may download many); do not leak archived/private photos the requester could not otherwise get.
- Guard against unbounded requests: cap the number of files per ZIP with a clear error above the cap
  (pick a sane limit and document it), or state why no cap is needed.
- **Frontend:** add a "Download ZIP" action to the library selection bar (acts on the current
  selection) and to the album page (downloads the whole album). It triggers the browser download,
  shows a pending state while the archive streams, and surfaces an error if the request fails or is
  over the cap. Touch-friendly.

## Verification

`make check` must pass. Backend: test that the endpoint streams a valid ZIP containing an entry per
requested photo with de-duplicated names, that a missing original is skipped (not fatal), that the
per-request cap is enforced, and that RBAC/visibility is honoured. Frontend: test that the selection
bar and album actions call the endpoint with the right UIDs and reflect pending/error states.
Document the endpoint in `docs/API.md` and the UI in `docs/FRONTEND.md`.
