# M3 — Face matching, assignment & suggestions

Connect detected faces to markers/subjects via IoU, expose assignment actions, and suggest likely
identities using vector similarity.

## Context
Read `docs/ARCHITECTURE.md` §7. Depends on: faces table + vector search, subjects/markers
repositories. Coordinates are normalized [x,y,w,h] (0..1). Mirror photo-sorter's IoU matching
(threshold 0.1) and suggestion approach.

## Requirements
- **IoU matching**: for a face, find the best-overlapping existing marker (IoU ≥ 0.1); cache the
  match on the face row. Helper to compute IoU between normalized boxes.
- **Assignment API** (editor/admin), state-machine style:
  - `create_marker` (no matching marker): create a face marker on the photo + assign subject.
  - `assign_person` (marker exists, unassigned): assign subject to marker.
  - `unassign_person`: clear assignment.
  Keep the `faces` cache columns and marker.reviewed consistent. Auto-create subject by name if
  needed (find-or-create by slug).
- **Suggestions API**: for an unnamed face, return up to ~5 likely subjects via nearest face
  embeddings (HNSW cosine), excluding faces already assigned to a different person and faces below
  a minimum size; include distance/confidence. Provide a distance threshold fallback.
- `GET /api/v1/photos/{uid}/faces` returns faces with bbox, assignment, and suggestions for the
  detail UI.

## Quality gate (mandatory)
- Use the **golang-developer** skill. `make check` MUST pass.
- Unit tests: IoU computation (overlap/no-overlap/contained), suggestion filtering rules.
- Integration tests (test DB): assign/unassign updates markers + faces cache; find-or-create
  subject; suggestions return nearest unassigned candidates excluding other people; size filter.
