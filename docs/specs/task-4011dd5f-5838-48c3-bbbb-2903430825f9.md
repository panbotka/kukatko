# Fix the full-page flicker on next/prev photo navigation

On the photo detail view, clicking the right (next) or left (previous) arrow makes the whole page blank to a loading spinner and re-render — an ugly flicker. Navigation is already correct client-side SPA routing; the flicker comes from the data-loading effect resetting to a loading state on every photo change. Make moving between neighboring photos smooth, with no full-page spinner.

## Where it is
- `web/src/pages/PhotoDetailPage.tsx`.
- The arrows are `<Link to={neighborTo(...)} replace>` (on-image) and keyboard handlers call `navigate(neighborTo(...), { replace: true })` — these are fine, they are SPA navigation, not a browser reload.
- The problem: the page reads the photo `uid` from the route param, and the data-loading `useEffect` keyed on `[uid]` calls `setState({ status: 'loading' })` at the top on every `uid` change. That drops the component into the full-page spinner branch, blanking everything before the new photo + edit data are fetched (and neighbors/faces re-resolve).

## Requirements
- Navigating to a neighboring photo must NOT blank the page to the full-page spinner. Keep the current content visible while the next photo loads, then swap in place.
- Suggested approaches (pick what fits the existing code cleanly):
  - Don't reset to `status: 'loading'` when there is already a loaded photo on screen; instead fetch in the background and swap when ready (optionally show a subtle inline/overlay loading indicator on the image only, not a full-page spinner).
  - And/or seed the new photo's basic data from the already-loaded neighbor list (`usePhotoNeighbors`) so the image and core metadata render immediately, then reconcile with the fetched detail.
- Preserve correctness: the abort-on-change behavior must still cancel superseded in-flight requests; faces, edit state, and neighbors must correspond to the currently displayed photo (no stale mismatch — e.g. showing photo A's faces over photo B). The very first load of the page (no photo yet) may still show the normal loading state.
- Keep the URL/back behavior intact (still uses `replace` navigation and the existing detail query string).

## Edge cases
- Rapidly pressing the arrow several times must not show stale data or leak requests; the latest target wins.
- A fetch error for a neighbor should surface as an inline error state, not leave stale content silently — but must not regress the smooth case.
