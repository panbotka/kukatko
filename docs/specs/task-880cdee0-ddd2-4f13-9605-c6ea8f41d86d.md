# Storage Interface: Drop The Filesystem Assumption

`internal/storage` exposes a `Storage` interface whose one concrete implementation is `FS`. The interface leaks the filesystem through `AbsPath(relPath string) string`, which hands callers a real path on disk. That single method is what prevents an object-storage backend from ever existing. This task removes it. It is a pure refactor: behaviour must not change.

## Requirements

- Remove `AbsPath` from the `Storage` interface and replace it with two methods:
  - `URL(relPath string) string` — the address at which a client can fetch the object. For the filesystem backend this is the existing application route that streams the file.
  - `Materialize(ctx context.Context, relPath string) (path string, cleanup func(), err error)` — yields a real local file for tools that can only read a filename, together with a cleanup function the caller must always call.
- The `FS` implementation returns the existing on-disk path from `Materialize` with a no-op cleanup. It must not copy the file. Local development and the test suite keep their current zero-copy behaviour and their current speed.
- Update every caller of `AbsPath`. They shell out to `exiftool`, `ffprobe`, `ffmpeg`, `heif-convert`, or `vipsthumbnail`, none of which can read an `io.Reader`. The known call sites live in `internal/thumb`, `internal/thumbjob`, `internal/facejob`, `internal/photoedit`, `internal/photoapi` (video), and `internal/imgconvert` — verify the list yourself rather than trusting it.
- Every caller must release the materialized file, including on the error path. A leaked temp file on a small VPS disk is a real failure.
- `internal/ingest` already runs EXIF extraction and probing against a locally staged temp file before the original is stored. Do not route those through `Materialize`; leave them alone.
- Do not add an S3 backend in this task. Do not change any configuration key. Do not change any HTTP response.

## Verification

`make check` must pass, and so must `make test-integration`. Add a unit test asserting that `FS.Materialize` returns the original path and does not copy, and that its cleanup is safe to call twice. Update `docs/PACKAGES.md`.
