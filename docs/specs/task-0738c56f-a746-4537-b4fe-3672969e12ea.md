# Frontend: adopt the maintainer/admin role model

Update the React frontend to the new role model: add **maintainer** as the top role, remove **ai**, and re-group navigation, route guards, and the user-management UI so operational areas require maintainer and governance areas require admin. Read `docs/FRONTEND.md` (role model / guards section) first.

## Target model (must mirror the backend)

Strict ladder **viewer < editor < admin < maintainer**, each inheriting the one below.
- editor = write (photos, metadata, sorting/review, albums & labels).
- admin = editor + user management + audit + trash purge.
- maintainer = admin + operations: import, maintenance, system status, backup, restore, jobs, processing.

## Requirements

- `web/src/services/auth.ts`: change the `Role` union to `'viewer' | 'editor' | 'admin' | 'maintainer'` (drop `'ai'`). Update `ROLE_RANK` to `viewer:0, editor:1, admin:2, maintainer:3`. `canWrite` = editor+. Replace `canImport` with a maintainer check (imports need maintainer). Add helpers as needed: `isMaintainer`; keep `isAdmin` meaning admin-or-higher (admin OR maintainer) via `roleAtLeast('admin')`. Any operational capability (import/maintenance/system/backup/restore/jobs/process) maps to maintainer.
- `web/src/auth/AuthContext.tsx` / `AuthProvider.tsx`: expose the new capabilities (e.g. `isMaintainer`; keep `isAdmin` as admin-or-higher; `canImport`→maintainer). `useAuth()` consumers must get correct values.
- `web/src/auth/ProtectedRoute.tsx`: `RequireRole` is already generic via `roleAtLeast`. Ensure `RequireImport` (and operational guards) require maintainer. Add a maintainer guard if helpful.
- `web/src/App.tsx`: route guards —
  - Operational routes (`/import`, `/maintenance`, `/system`, and any backup/restore/jobs/process UI routes) → require **maintainer**.
  - Governance routes (`/users`, `/audit`) → require **admin** (admin-or-higher).
  - Write routes (review, duplicates, upload, clusters, faces, expand, recognition, outliers, trash view) → editor, unchanged. Note: the destructive "empty trash" ACTION requires admin on the backend now; the trash page itself stays editor-visible, but the empty/purge control must be shown/enabled only for admin-or-higher.
- `web/src/components/Layout.tsx`: reorganise the nav. Create a **maintainer** group for operational items (import, maintenance, system, and backup/restore/jobs if they have nav entries) shown only to maintainer; keep a **governance** group (users, audit) shown to admin-or-higher. `IMPORT_ITEM` moves under the maintainer group.
- User-management UI (`web/src/pages/UsersPage.tsx` and its role selector): offer roles **viewer/editor/admin/maintainer** (drop `ai`). The **maintainer** option must only be selectable/assignable by a maintainer (mirror the backend rule — an admin cannot grant maintainer); hide or disable the maintainer option for a non-maintainer admin, and don't let a non-maintainer edit an existing maintainer's account.
- Pages/components currently gated on `isAdmin`/`canImport` for operational areas (`ImportPage`, `MaintenancePage`, `SystemStatusPage`, `web/src/components/JobQueueBadges.tsx`) must switch to the maintainer capability. Governance pages (`UsersPage`, `AuditPage`) stay admin-or-higher.
- i18n: add role labels for `maintainer` and remove `ai` in `web/src/i18n/locales/{cs,en}/common.json` — Czech default. Choose a sensible Czech label for maintainer (e.g. "Správce systému") and English "Maintainer". Update any role-label map.

## Tests

- Vitest: `auth.ts` helpers — `roleAtLeast`, `canWrite`, the maintainer/import capability, and `isAdmin` (admin-or-higher) for each of the four roles.
- Component tests where feasible: the user role selector does not offer maintainer to a non-maintainer; operational nav/pages are hidden for admin and shown for maintainer.
