# M5 — PhotoPrism import job

Import photos and metadata from PhotoPrism into Kukatko: download originals, dedup, map external
IDs, and run incrementally on subsequent passes. Then let normal jobs compute embeddings/faces.

## Context
Read `docs/ARCHITECTURE.md` §9. Depends on: PhotoPrism API client, import_runs repo, storage,
photos/albums/labels/people repositories, job queue. PhotoPrism file Hash is **SHA1**; Kukatko
uses **SHA256** (compute after download). Store `photoprism_uid` + `photoprism_file_hash` for
dedup/incremental.

## Requirements
- A `pp_import` job (and/or `kukatko import photoprism` command + admin endpoint
  `POST /api/v1/import/photoprism`) that:
  1. Reads the latest successful watermark from `import_runs`; lists photos with
     `updated:` ≥ watermark (incremental); first run pulls all.
  2. For each photo: dedup by `photoprism_uid` (already imported → update changed metadata);
     otherwise download the primary original, compute SHA256, dedup by `file_hash`, store original,
     create `photos` row with metadata (taken_at, GPS, title/description, camera/lens/EXIF) +
     `photoprism_uid` + `photoprism_file_hash`, generate thumbnails, and **enqueue image_embed +
     face_detect** jobs.
  3. Map albums, labels, and people/markers (find-or-create by name; attach to photos).
  4. Record counts + new watermark in `import_runs`; resumable/idempotent (safe to re-run).
- Handle partial failures per-photo without aborting the whole run; respect 429 backoff from the
  client. Long runs should checkpoint progress.

## Quality gate (mandatory)
- Use the **golang-developer** skill. `make check` MUST pass.
- Integration tests (test DB) with a **fake PhotoPrism**: first import creates photos + albums +
  labels + people + external IDs and enqueues embed/face jobs; re-run is idempotent (no
  duplicates); incremental run only processes changed photos; SHA256 dedup against an existing
  identical file; per-photo failure is recorded and doesn't abort the run.
