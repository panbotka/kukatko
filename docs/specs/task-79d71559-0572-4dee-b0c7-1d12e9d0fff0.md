# Remote CLI: `kukatko ctl`

Every existing CLI subcommand (`serve`, `migrate`, `import`, `backup`, `restore`, `maintenance`, `version`) acts directly on the database and filesystem. No HTTP client for Kukátko's own API exists anywhere in the tree. Add one, so a running production instance can be driven from a terminal — and through that terminal, by an AI agent.

This is cheaper in tokens than an MCP server: no tool schemas are loaded into the model's context, just a short command and a narrow result. Keep the output compact; that is the whole point.

This task builds the client foundation and the `photos` commands. Other resources follow in a separate task.

## Requirements

### Invocation

- A new `ctl` subcommand tree: `kukatko ctl photos list --year 2024`.
- One binary. When invoked through a symlink named `kukatkoctl`, the `ctl` level is implied, so `kukatkoctl photos list` works. Detect this from the program name.

### Configuration

- A client-side config at `~/.config/kukatko/ctl.yaml` holding named contexts in the style of `kubectl`: each has a server URL and a token, with one marked current. Add `ctl config` subcommands to set, list, and switch contexts.
- `KUKATKO_SERVER` and `KUKATKO_TOKEN` override the active context.
- This file has nothing to do with `internal/config`, which is server-side only and has no notion of a remote endpoint. Do not extend the server config struct.
- Never write a token to a file that is not mode `0600`. Never log a token.

### Authentication

- Send `Authorization: Bearer <token>`. (A companion task adds token support to the server; if it is not present yet, write the client against that contract and let the integration test drive an `httptest` server.)
- A `401` produces a short actionable message — that the token is missing, expired, or revoked, and how to create a new one — not a stack trace and not a raw body dump.

### Output

- A compact table by default. `-o json` emits the API's JSON unchanged for machine consumption. `-o yaml` is not required.
- Sensible exit codes: `0` success, non-zero on HTTP or transport failure.

### Commands in this task

- `ctl photos list` with the filters the API already supports (paging, sort, year, album, label, favorite, archived).
- `ctl photos get <uid>`.
- `ctl photos search <query>` with the search mode flag.

Note that the API has **no uniform list envelope**: `photos` returns `{photos, total, limit, offset, next_offset}` while other resources return bare lists. Write a decoder per resource; do not build a generic one and do not change the API to make one possible — that would break the frontend.

## Verification

`make check` must pass. Add unit tests for context resolution and the environment overrides, and tests against an `httptest` server covering both output formats, a successful list, an empty list, and the `401` message. Update `docs/OPERATIONS.md` with the command tree and the config file, and `docs/PACKAGES.md`.
