# M1 — Photo listing, detail, metadata update, archive

Expose the REST API to browse the library with filters/sorting/pagination, fetch a photo, update
its metadata, serve originals/thumbnails, and archive (trash) photos.

## Context
Read `docs/ARCHITECTURE.md` §3.2, §5, §13 (filters everywhere; back works). Depends on the photos
repository, storage, thumbnailer. Filters/sort must be expressible as query params so the frontend
can mirror them in the URL.

## Requirements
- `GET /api/v1/photos` (auth): list with **filters** (date range via taken_at, has-GPS,
  camera/lens, favorite [per-user, may be wired in the favorites task], private, archived,
  album/label scope as available) and **sorting** (newest/oldest/taken_at/added/title/size) and
  **pagination** (cursor or limit/offset). Return total/next-cursor for infinite scroll.
- `GET /api/v1/photos/{uid}`: full detail (metadata, EXIF, file info, GPS).
- `PATCH /api/v1/photos/{uid}` (editor/admin): update title/description/notes/taken_at/lat/lng.
- `POST /api/v1/photos/{uid}/archive` and `/unarchive` (editor/admin): soft delete via
  `archived_at`; archived excluded from default listing.
- Media serving: `GET /api/v1/photos/{uid}/thumb/{size}` (cached thumbnail) and
  `GET /api/v1/photos/{uid}/download` (original) — **stream** the file (do not load whole file
  into RAM), correct content-type and cache headers; download honors auth/download token.
- All list filters/sorts documented and validated; invalid params → 400.

## Quality gate (mandatory)
- Use the **golang-developer** skill. `make check` MUST pass.
- Integration tests (test DB): listing with each filter + sort + pagination correctness;
  detail; metadata update; archive hides from default list; thumbnail + original streaming
  (assert streamed, not buffered); auth/role enforcement (viewer cannot PATCH/archive).
