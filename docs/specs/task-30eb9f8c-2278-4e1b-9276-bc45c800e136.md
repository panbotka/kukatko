# Backend role model: add `maintainer`, remove `ai`

Introduce a new top-level role **maintainer** and remove the existing **ai** role, redefining the authorization model as a strict ladder. This task changes ONLY the core role model, the DB constraint/migration, user-management validation, and the MCP per-tool role checks. Endpoint re-partitioning (which HTTP surfaces require which role) is a SEPARATE task — do not change `cmd/kukatko` endpoint wiring here beyond what is needed to keep the code compiling.

Read `docs/PACKAGES.md` (auth section) before starting. The canonical role source is `internal/auth/role.go`; the DB constraint is defined in `internal/database/migrations/0002_auth.sql` and amended by `0023_role_ai.sql`.

## Target role model (strict ladder)

Roles, lowest to highest: **viewer < editor < admin < maintainer**. Each role inherits every permission of the roles below it.

- **viewer** — read-only (view photos, albums, labels).
- **editor** — viewer + write: edit photos & metadata, sorting/review, assign photos to albums & labels, manage albums & labels.
- **admin** — editor + governance: user management, audit log, emptying/purging trash.
- **maintainer** — admin + operations: imports, maintenance, system status, backup, restore, jobs, processing backfills. The most powerful role.

## Requirements

- In `internal/auth/role.go`:
  - Add `RoleMaintainer` (string `"maintainer"`). Remove `RoleAI` and every reference to it.
  - Update `Valid()` to accept `viewer/editor/admin/maintainer` and reject `ai`.
  - Redefine the permission predicates for the strict ladder. `CanWrite()` = editor, admin, or maintainer. `IsAdmin()` (governance: users/audit/trash) = admin or maintainer. Introduce a predicate for the maintainer tier (operations), e.g. `IsMaintainer()`/`CanMaintain()` = maintainer only. `CanImport()` (and any other operational capability) must now require maintainer — imports move from the old ai/admin capability up to maintainer.
  - Keep the pure `authorize(role, requirement)` design. Add whatever `requirement` value the maintainer tier needs (e.g. `requireMaintain`) and wire it through `authorize`. `requireAdmin` must be satisfied by admin OR maintainer (ladder inheritance).
- In `internal/auth/middleware.go`: add a `RequireMaintainer` middleware constructor mirroring the existing `Require*` set. Keep the existing `RequireAuth/RequireWrite/RequireAdmin/RequireImport` methods present so unrelated wiring still compiles. Their semantics follow the new model: `RequireImport` now resolves to maintainer (imports are an operations capability); `RequireAdmin` is satisfied by admin or maintainer.
- Migration `internal/database/migrations/0036_role_maintainer.sql` (verify the highest existing migration and use the next free number):
  - Drop and re-add the `users_role_check` CHECK constraint to `('viewer','editor','admin','maintainer')` — mirror the pattern in `0023_role_ai.sql`.
  - **Migrate existing data**: `UPDATE users SET role='maintainer' WHERE role='ai'` BEFORE the new constraint forbids `ai`. (This is how the botka automation account, currently `ai`, becomes a maintainer.)
  - `sessions.role` has no CHECK constraint; no change needed there. Any live `ai` session simply becomes invalid on next login — acceptable.
- **User-management validation** in `internal/auth` (service_admin.go / validation helpers):
  - Reject `ai` as an assignable role everywhere (creation & update).
  - **Only a maintainer may create or promote an account to the maintainer role**, and only a maintainer may modify (edit/disable/reset password of) an existing maintainer account. An admin managing users can assign viewer/editor/admin but must be forbidden (403 / validation error) from granting or altering the maintainer role. The acting user's role is available from the request context — thread it into the create/update validation.
  - **Bootstrap**: the first account created when the users table is empty must be created as **maintainer** (not admin), so the instance always has a root that can grant maintainer. Update `Service.Bootstrap`.
- **MCP** (`internal/mcpapi`, esp. `caller.go`): update any per-tool role checks that reference the `ai` role or the old import capability to the new model (imports/ops → maintainer).
- Grep the entire repo for the `ai` role (`RoleAI`, `"ai"` role literals, `.CanImport`, any role-rank tables) and update every occurrence. Be careful NOT to touch the unrelated label-source constant `SourceAI = "ai"` in `internal/organize` / `internal/ctl` — that is metadata provenance, not an auth role.
- Do NOT change frontend code (separate task) and do NOT re-partition `cmd/kukatko` endpoint gating (separate task) — only keep things compiling.

## Tests

- Unit tests for `role.go`: the new predicates for each of the four roles (write/admin/maintain/import truth table); `Valid()` accepts maintainer and rejects ai.
- Integration/service tests for user management: an admin cannot create or promote to maintainer (expect the right error/403); a maintainer can; bootstrap creates a maintainer; assigning `ai` is rejected.
- Migration/integration: after migrating, an `ai` user has become `maintainer`, and inserting a user with role `ai` violates the CHECK constraint.
