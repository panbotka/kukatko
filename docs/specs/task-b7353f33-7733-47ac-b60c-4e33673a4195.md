# PhotoPrism import — carry over the metadata Kukátko currently drops

`internal/ppimport` maps title, description, dates, GPS, camera/lens/exposure, albums, labels and face
markers out of PhotoPrism. It drops a whole block of metadata that PhotoPrism does hold, and Kukátko now
has columns for it. Map it across so a re-import brings the library up to parity.

**Before starting, confirm the target columns exist** on `photos`: `subject`, `artist`, `copyright`,
`license`, `keywords`, `software`, `scan`, `color_profile`, `image_codec`, `camera_serial`,
`original_name`, `projection`. If they do not, the migration task has not run yet — add them first
(all TEXT NOT NULL DEFAULT '', `scan` BOOLEAN NOT NULL DEFAULT false) rather than blocking.

## The two traps — read this first

1. **PhotoPrism's list endpoint does not carry these fields at all.** `GET /api/v1/photos?merged=true`
   returns a flattened search struct with **no `Details` object** — no Subject, Artist, Copyright,
   License, Keywords, Notes, no Altitude, no per-file Codec/ColorProfile/Orientation. They exist only on
   the per-photo detail endpoint `GET /api/v1/photos/{uid}`. This is exactly the same class of bug as
   commit `0798d7f` (markers served only on the detail endpoint). The importer already fetches the
   detail per photo for albums/labels/markers — hang this mapping off that same fetch, do not add a
   second round-trip.
2. **`photo_description` is a dead column in PhotoPrism.** It was renamed to `photo_caption`
   (`description_src` → `caption_src`) and the Go field is `gorm:"-"`, i.e. not persisted. Import the
   caption from **`Caption`**, not `Description`. Check what `internal/photoprism` and `internal/ppimport`
   currently read — if they read `Description`, that is a live bug and captions are silently being lost;
   fix it as part of this task and say so in the commit message.

## Requirements

Extend the read-only client in `internal/photoprism` with the fields it does not currently model, and map
them in `internal/ppimport`:

| PhotoPrism (JSON on the detail endpoint) | Kukátko column |
|---|---|
| `Details.Subject` | `subject` |
| `Details.Artist` | `artist` |
| `Details.Copyright` | `copyright` |
| `Details.License` | `license` |
| `Details.Keywords` (comma-separated) | `keywords` |
| `Details.Notes` | `notes` (only if Kukátko's `notes` is empty — never clobber a local note) |
| `Details.Software` | `software` |
| `Scan` | `scan` |
| `CameraSerial` | `camera_serial` |
| `OriginalName` | `original_name` |
| primary `Files[].Codec` | `image_codec` (stills only — do not touch `video_codec`/`audio_codec`, those keep coming from ffprobe) |
| primary `Files[].ColorProfile` | `color_profile` |
| primary `Files[].Projection` | `projection` |
| `Favorite` | see below |

**`Favorite` is currently dropped by the PhotoPrism importer** even though the photo-sorter importer
(`internal/psimport`) does map it. Kukátko's favourites are **per-user**, so map PhotoPrism's global
favourite onto a favourite for the user the import runs as (the same way `psimport` does it) — mirror
that existing behaviour rather than inventing a new one. If on inspection `psimport` does something that
does not translate, leave `Favorite` alone and note why in the commit message.

## Precedence rules

The import is **incremental and idempotent** and re-runs over photos that already exist. Keep the
importer's existing precedence discipline:

- PhotoPrism wins over locally-extracted EXIF for the fields it owns (this matches how camera/exposure
  are handled today).
- PhotoPrism must **never overwrite a non-empty Kukátko value with an empty PhotoPrism one**.
- Never touch curation the user did in Kukátko: rating, flag, per-user favourites already set,
  `ai_note`, albums/labels the user added locally.
- Re-running the import must be a no-op when nothing changed upstream.

## Verification

- Unit tests for the mapping with fixture PhotoPrism detail JSON — including a photo whose `Details` is
  absent/empty (older PhotoPrism rows have no details row at all), a keywords string with stray
  whitespace, and the empty-does-not-clobber-non-empty rule.
- Integration test over the fake/mock PhotoPrism client (the client is behind an interface — use it,
  do not hit the live instance in tests): first import populates the fields, second import is a no-op,
  a locally-edited value survives a re-import.
- `make check` must pass.
- Update `docs/PACKAGES.md` (`internal/ppimport`, `internal/photoprism`) with the fields now carried over.

## Out of scope

- Frontend display and editing (separate tasks).
- EXIF-based extraction of the same fields (separate task).
- Any write back to PhotoPrism — the client stays strictly read-only.
