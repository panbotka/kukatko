# Fix invisible filter labels (white text on white background)

In the library's advanced filter panel, the "Pořízeno od / do" (captured-date) labels — and other
labels in that panel — are effectively invisible: near-white text on a near-white background. The
date-range column reads as a blank white block. This makes the filters unusable.

## Root cause (verify, then fix)

The app never sets `data-bs-theme="dark"` anywhere (`web/index.html`, `web/src/main.tsx`, the app
root). Bootswatch Superhero is dark for body text but leaves several **surface tokens at light
values on `:root`**, flipping them to dark only under `[data-bs-theme=dark]`. So `--bs-body-color`
and `--bs-tertiary-bg` both resolve to a near-white `#ebebeb`. The filter panel wrapper in
`web/src/components/library/FilterBar.tsx` uses `className="card card-body bg-body-tertiary"`, which
paints a near-white background, and the `Form.Label`s inside carry no colour class so they inherit
the near-white body colour → invisible. The native date inputs compound it: `web/index.html` sets
`<meta name="color-scheme" content="dark">` while Superhero forces a white form-control background,
so the empty date placeholder and calendar glyph are painted light-on-white too.

## Requirements

- Every label and control in the library advanced filter panel is clearly legible: readable text
  on a properly dark surface consistent with the rest of the Superhero UI. The "Pořízeno od / do"
  date fields, their labels, placeholders, and the calendar-picker glyph are all visible.
- **Fix the root cause, not just this one panel.** Prefer setting `data-bs-theme="dark"` at the
  document/app root so Bootstrap's dark surface tokens apply everywhere and this whole class of
  "light token on a dark app" bug is eliminated. Align `color-scheme` with the dark theme so native
  controls (date inputs, selects) render dark-appropriately.
- **Do not regress other screens.** A root theme change touches the whole app, so verify the main
  surfaces still look correct: the library grid + filter bar (expanded), the photo detail page, the
  upload page, albums/labels pages, the login page, modals/offcanvas, and any component that
  currently relies on `.bg-body-*` or Bootstrap surface tokens. If the global change causes a
  regression that cannot be cleanly resolved, fall back to a scoped fix (give the filter panel an
  explicit dark background and light label colour, and fix the date inputs' `color-scheme`), and
  say so.
- Verify visually, not just by reading CSS — confirm the labels are actually readable against the
  panel background in the running UI (a static style harness against the real Superhero + app CSS
  is acceptable).

## Verification

`make check` must pass. Add a test or assertion guarding the fix (e.g. the theme attribute is set
on the root, or the filter panel/labels carry the intended dark-surface classes) so the regression
cannot silently return. Note in the commit / `docs/FRONTEND.md` whether the fix was global
(`data-bs-theme`) or scoped, and what screens were visually verified.
