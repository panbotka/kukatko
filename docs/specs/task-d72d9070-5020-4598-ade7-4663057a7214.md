# Bound the media decode/thumbnail pipeline against pathological input

Two robustness gaps let a single bad image wedge a worker or OOM the box.

## Issues (verified)

1. **`vipsthumbnail` is the one shell-out with no per-invocation timeout.**
   `internal/thumb/vips.go:155` (`runVips`) calls `exec.CommandContext(ctx, …)` with the
   worker's long-lived context, which carries no deadline (there is no per-job timeout
   anywhere). Every sibling bounds itself with `context.WithTimeout` — heif 30s
   (`imgconvert/heif.go:38`), raw 60s (`imgconvert/raw.go:69`), poster 60s
   (`video/poster.go:87`), probe 30s (`video/probe.go:70`), exiftool 30s
   (`exif/exiftool.go:49`). A wedged `vipsthumbnail` (pathological input / stalled R2
   temp read) hangs a worker goroutine forever; enough wedged jobs starve the whole
   background queue until restart.
   - Fix: wrap the exec in `context.WithTimeout` inside `runVips` (~60s, matching the
     poster/raw bounds) so a wedged process is killed and the job fails/retries.

2. **Full-image decode has no pixel-count guard (decompression bomb).**
   `internal/ingest/ingest.go:433` (`decodeOriginal`, run for every uploaded image for
   the pHash) and `internal/thumb/resize.go:40` (`decodeAndOrient`) call `image.Decode`
   on the raster path (JPEG/PNG/WebP/BMP/GIF/TIFF) with no dimension cap. A legitimately
   huge image (e.g. 30000×30000 panorama, or a decompression-bomb PNG/TIFF) allocates
   the whole RGBA (~3.6 GB) and is decoded TWICE in flight (ingest pHash + thumb worker)
   → can OOM the request or a worker on the 16 GB box shared with other stacks. (Behind
   `RequireWrite`, so robustness rather than an unauthenticated attack.)
   - Fix: peek `image.DecodeConfig` and reject or down-route sources whose
     `Width*Height` exceeds a configurable cap before allocating the full bitmap.

## Requirements
- `runVips` is bounded by a timeout; a hung vips fails the job instead of blocking a
  worker.
- Oversized images are rejected (or safely handled) before a full decode; add a
  configurable max-pixels bound (with a sensible default).

## Testing
- Unit tests: `runVips` respects a context deadline; an over-large `DecodeConfig` is
  rejected before decode. `make check` must pass. Add the new config key to
  `config.example.yaml` + `docs/OPERATIONS.md`.
