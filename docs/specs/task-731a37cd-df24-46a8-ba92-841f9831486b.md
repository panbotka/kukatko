# Add "person" and "favorites" filters to the library

The library filter bar (`web/src/components/library/FilterBar.tsx`) can filter by year, album,
tag/label, camera, captured-date, archived, GPS, private, rating and flag — but **not by person**,
even though people are first-class elsewhere in the app, and there is **no favorites toggle**
(a Favorites page exists, but you can't combine "favorites + album + year" in the main grid). Add
both facets so the grid can be narrowed by who is in the photo and to favorites only.

## Current state (for context)

- Filters live in URL state and are assembled by `buildPhotoQuery` (`web/src/services/photos.ts`)
  into `GET /api/v1/photos`. `buildPhotoQuery` already supports a `favorite` param (used only by
  the Favorites page today) — the backend list already honours it.
- Facets render in `FacetRow`/`AdvancedFilters` inside `FilterBar.tsx`; album and tag use a
  `SearchableSelect` combobox. Facet option data is gathered in
  `web/src/hooks/useLibraryFacets.ts` (`GET /api/v1/albums`, `/api/v1/labels`, `/api/v1/photos/years`).
- Subjects/people are served by `GET /api/v1/subjects` (`web/src/services/people.ts`); the
  photo-list query does **not** currently accept a person/subject filter.

## Requirements

- **Favorites filter:** add a control in the filter bar (a toggle or tri-state consistent with the
  existing GPS/private tri-states) that sets the already-supported `favorite` filter, wired into
  URL state like every other filter. No backend change needed — reuse the existing `favorite` param.
- **Person filter:** add a "person" facet (a `SearchableSelect` mirroring the album/tag facets)
  populated from `GET /api/v1/subjects`. Selecting one or more people narrows the grid to photos
  that contain them. This needs backend support: extend the photo-list query
  (`buildPhotoQuery` + the backend `photoapi` list handler and `internal/photos` store) to accept a
  subject/person filter and join through the faces/markers that link subjects to photos. Follow the
  existing filter param conventions; keep it URL-state driven so "Back always works".
- Both facets behave like the others: reflected in the active-filter chips, cleared by "clear all",
  survive reload via the URL, and combine with every existing filter (AND semantics).
- Respect the colour language if albums/tags/people already have distinct chip colours; otherwise
  keep the person chip visually consistent with the other facet chips.

## Verification

`make check` must pass. Backend: add a `photos`/`photoapi` test that the list query filters to
photos containing a given subject. Frontend: extend the library/filter tests to assert that
selecting a person writes the person filter to the URL and refetches with it, that Back removes it,
and that the favorites toggle sets/clears the `favorite` filter. Update `docs/API.md` (new list
filter param), `docs/FRONTEND.md`, and `docs/PACKAGES.md` if the store query changes.
