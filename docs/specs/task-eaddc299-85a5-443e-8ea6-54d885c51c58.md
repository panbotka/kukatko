# Metadata sidecars — curation that outlives the database

Everything a user builds in Kukátko — titles, descriptions, who is in the photo, which album it belongs to, the
rating — exists in exactly one place: Postgres. The originals on disk are pristine and know nothing. There is an S3
backup, and that is good, but it is a single mechanism: a backup that has been quietly failing for three months is
discovered on the day you need it.

PhotoPrism's answer is a sidecar per photo: the index can be thrown away and rebuilt from the originals plus the
sidecars. That is a genuinely different kind of safety — the curation lives *next to the photo it describes*, in a
plain-text file any tool can read, on the same storage that holds the original.

Build the export half.

## What gets written

One sidecar per photo, next to the original in storage (or in a parallel sidecar tree — decide and document; a
parallel tree is safer if originals may be on read-only or object storage, and Kukátko's originals can live in R2).

**YAML** — human-readable, diffable, and the format PhotoPrism uses, which makes the file legible to anyone who
lands on it cold. Contents: everything a human created or a machine derived that would be expensive to recompute:

- identity: the photo UID, the SHA256, the original filename and path
- descriptive: title, description, notes, ai_note, and the IPTC fields (subject, artist, copyright, license,
  keywords) **if those columns exist by the time this runs** — if they do not, write what exists and do not block
- temporal: `taken_at` and its source (plus the estimated-date flag and note, if those exist)
- spatial: lat/lng/altitude and their source, plus the cached place (country/city)
- technical: camera, lens, ISO, aperture, exposure, focal length, dimensions
- curation: album membership, labels (with source and confidence), **people** (subject name + the face bounding
  box — a marker without its box cannot be rebuilt), per-user favourites and ratings, private, archived
- the non-destructive edit (crop / rotate / brightness / contrast)

**Deliberately NOT written:** the embedding vectors and the face vectors. They are large, they are binary, and they
are cheap to recompute from the original — that is what the backfill jobs are for. Say this in the file header so
nobody "fixes" it later.

## When it gets written

- A **job type** (`sidecar`) that writes/refreshes one photo's sidecar, enqueued whenever the photo's metadata or
  curation changes. Follow the existing job-queue conventions (`internal/jobs`, a handler like `thumbjob`).
  Debounce it — a bulk edit over 500 photos must enqueue 500 cheap jobs, not write 500 files synchronously inside
  the request.
- A **backfill** over the whole library, following the `internal/processapi` `/process/*` pattern (admin-only,
  idempotent, resumable) — plus a CLI equivalent, since this is exactly the thing you want to run from a terminal
  before a risky operation.
- Writing is **atomic** (write a temp file, rename) — a half-written sidecar is worse than none. A failure logs and
  is retried by the queue; it must never fail the user's edit.
- A config switch to turn the whole thing off (default **on**), because on a large library it is real I/O.

## The format is a contract

The point of this feature is that a future `kukatko restore --from-sidecars` can rebuild the catalogue. That
importer is **not** part of this task — but the format must be sufficient for it, and that is the thing to get
right. So:

- Version the schema (a `version:` key at the top). A format you cannot evolve is a trap.
- **Write a test that round-trips**: serialise a fully-populated photo to YAML, parse it back, and assert every
  field survives. That test is what proves the format is actually sufficient, and it is what the future importer
  will be built against.
- Document the format in `docs/RESTORE.md` — that is where someone will look when the database is gone.

## Verification

- Unit tests: the serialiser writes every field; the round-trip test above; the atomic write leaves no partial
  file on a simulated failure.
- Integration: editing a photo enqueues the job and the sidecar appears with the new value; a bulk edit over N
  photos enqueues N jobs and does not block the request; the backfill is idempotent; the whole thing works with
  originals on **both** local FS and R2 (the storage abstraction — do not assume a local path).
- `make check` must pass.
- Document: `docs/RESTORE.md` (the format, in full — this is the document that matters here),
  `docs/ARCHITECTURE.md` (the design decision and what is deliberately excluded), `docs/PACKAGES.md`,
  `docs/OPERATIONS.md` + `config.example.yaml` (the switch, the backfill endpoint, the CLI command).

## Out of scope

- **Reading sidecars back / rebuilding the catalogue from them** — the next task, and it deserves its own.
- Writing metadata into the original files themselves. Originals stay untouched; that is a project invariant.
- XMP. YAML only.
