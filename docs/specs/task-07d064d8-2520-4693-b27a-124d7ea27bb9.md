# M5 — PhotoPrism API client

Implement a read-only client for a running PhotoPrism instance: authenticate, list photos
(incrementally), read albums/labels/subjects/markers/GPS, and download originals.

## Context
Read `docs/ARCHITECTURE.md` §9 (verified API details). PhotoPrism stays primary; import is
read-only and repeatable. Auth: a long-lived **app password / access token** sent as
`Authorization: Bearer` (do NOT login per request — login is rate-limited). Configure base URL +
token via config/env (add keys like `import.photoprism.{base_url,token}`; secrets via env).

## Requirements
- `internal/photoprism` client behind an **interface** (fakeable in tests):
  - `ListPhotos(ctx, params)` → `GET /api/v1/photos?count=1000&offset=N&merged=true&order=updated&
    q=updated:"<RFC3339>"` for **incremental** pulls; parse fields: UID, TakenAt, Lat/Lng/Altitude,
    Title, Description, Type, Width/Height, OriginalName, Camera/Lens/EXIF, and `Files[]`
    (UID, Hash[SHA1], Primary, Mime, Markers[]). Handle pagination (count≤1000 + offset).
  - `ListAlbums`, `ListLabels`, `ListSubjects`, plus markers via `Files[].Markers[]`.
  - `DownloadOriginal(ctx, fileHash)` → `GET /api/v1/dl/{hash}?t=<download_token>` streaming the
    original; obtain/refresh the **download_token** (it can rotate — read `X-Download-Token` from
    responses) from a session/create-session call.
- Robustness: handle **429** with exponential backoff; require `Content-Type: application/json`
  on JSON calls; sane timeouts; typed errors. Pin behavior to the configured instance (API has no
  deprecation policy).

## Quality gate (mandatory)
- Use the **golang-developer** skill. `make check` MUST pass.
- Tests against a **fake PhotoPrism server** (httptest): incremental query construction + paging;
  field parsing incl. Files[].Hash + Markers[]; download streams bytes with token; token refresh
  from `X-Download-Token`; 429 backoff. No real PhotoPrism required.
