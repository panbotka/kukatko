# Fix: the login rate-limiter map grows unbounded (RAM-exhaustion DoS)

An anonymous attacker can exhaust the box's RAM by flooding the public login endpoint
with distinct, oversized usernames.

## Root cause (verified)
- `internal/auth/handlers_auth.go:69-70` builds the limiter key as
  `normalizeUsername(req.Username) + "|" + clientIP(r)`.
- `internal/auth/ratelimit.go` backs it with `attempts map[string][]time.Time` (~16-21)
  with NO size cap; `Allow` inserts a key on every call (~44-55). Keys are pruned only
  by `Cleanup`, ticked hourly (`cmd/kukatko/serve.go`, `sessionCleanupInterval = 1h`).
- `POST /api/v1/auth/login` is public, has no outer IP rate limit, and there is NO
  username-length validation anywhere; the body cap is 1 MiB, so each key can be ~1 MB.
  A non-existent username returns before bcrypt (`service.go:53-58`), so requests are
  cheap and high-throughput.

## Failure scenario
Flood login with a distinct ~1 MB username per request. Each becomes a persistent
key in the single process-global map (unbounded, pruned only hourly). On this shared
16 GB box (Postgres + other stacks co-resident) sustained requests exhaust RAM within
minutes — no auth, no header spoofing needed. (Distinct from the known SEC-001 bypass
and SEC-002 disk findings.)

## Requirements
- Cap the username length BEFORE it is used as a limiter key or looked up — reject
  (400) or truncate at a sane bound (e.g. 64 chars). A username longer than the max is
  never a real account.
- Give `auth.Limiter` a hard maximum key count that prunes on insertion (evict oldest /
  expired), mirroring how `internal/ratelimit` bounds its bucket map with `maxBuckets`
  — so memory is bounded without relying on the hourly `Cleanup` tick.
- Do not weaken the existing per-(username,IP) throttling for legitimate use.
- Consider (optional) an IP-independent per-username failure counter, which also helps
  the known SEC-001 per-account guessing concern.

## Testing
- Unit tests: an over-long username is rejected/truncated; the limiter map size stays
  bounded under many distinct keys (insertion-time pruning). `make check` must pass.
- Add the username max-length to `docs/` if it's a new config/validation constant.
