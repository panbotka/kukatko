# User administration

Administrators have no interface for managing accounts, even though the backend already exposes admin-only user endpoints (list, create, update, disable, set password). Add the missing page. This task is frontend-only.

## Requirements

- A new admin-only route `/users`, gated exactly the way the existing admin routes (`/system`, `/import`, `/maintenance`) are, and linked from the admin group in the main navigation.
- The page lists all users in a table: username, real name, role, enabled/disabled state, note, last login, created date.
- Create a user via a dialog: username, password, role (admin / editor / viewer), real name, note. Validation errors returned by the API are shown inline next to the offending field, not as a generic alert.
- Edit a user: change role, real name and note. The username is immutable and rendered read-only.
- Enable or disable a user behind a confirmation step. An administrator may not disable their own account — the control is disabled with a short explanation of why.
- Change another user's password from a separate dialog. Never render existing password hashes anywhere.
- Deleting users is not offered; disabling is the supported way to retire an account.
- All strings go through i18n in both `cs` and `en`; `cs` is the default.
- States: a loading skeleton while fetching, a clear error alert with a retry button on failure, and an empty state that cannot normally occur (there is always at least the bootstrap admin) but must not crash.

## Implementation notes

- Follow the existing API service module pattern under `web/src/services/`.
- Read the Go admin user handlers to confirm the exact request and response payloads rather than guessing field names.

## Tests

- Component tests: the page is unreachable for a non-admin, the table renders from a stubbed response, an inline field error is shown when the API rejects a create, and the self-disable control is disabled for the logged-in admin's own row.
