# Import from photo-sorter over its HTTP API

`internal/psimport` migrates from photo-sorter by reading its **PostgreSQL database directly** (`import.photosorter.dsn`) and opening each original **from the local filesystem** (`s.open(ps.FilePath)`, `internal/psimport/photos.go:99`). Neither works against production: photo-sorter runs on the VPS (`sorter.kotrzina.cz`), its Postgres is not exposed, and the Pi cannot open a path that lives on another machine.

Switch the migration to photo-sorter's HTTP API.

## Do NOT import into the live catalogue

The user runs the real migration **by hand, together with the assistant** — no task imports production data. Write the code and test it; for a live check use the **test database** (`KUKATKO_TEST_DATABASE_URL_HOST`, DB `kukatko_test`, safe to truncate) with a scratch storage dir, never the live catalogue. Reading from the production photo-sorter is fine — the client is read-only by design.

## Blocked on

The two photo-sorter tasks that add the missing endpoints (complete photo payload + `updated_since` cursor + `?include=labels,albums,markers,files`; bulk embedding/face feeds). **Do not start until both have landed** — without them a faithful migration cannot be written, and half of it would have to be thrown away.

### Status: STILL BLOCKED — verified against production 2026-07-14

Probed `https://sorter.kotrzina.cz` with the token from `.secrets/db.env` (auth works, real
data comes back). Neither prerequisite task has landed. Implementation was **not** started.

What exists today:

| Endpoint | State |
| --- | --- |
| `GET /api/v1/photos` | Flat listing, 19 scalar fields. Paginates on `offset` + `count` (**not** `limit`/`page`/`per_page` — those are silently ignored). |
| `GET /api/v1/photos/{uid}` | Same 19 fields, no relation expansion. |
| `GET /api/v1/photos/{uid}/faces` | Face *bookkeeping* only (`embeddings_count`, `markers_count`, `faces_processed`) — **no vectors**. |
| `GET /api/v1/labels`, `/api/v1/albums`, `/api/v1/subjects` | Present. |

What the requirements need and the API does not have:

1. **Download of originals — missing entirely.** `GET /api/v1/photos/{uid}/download?original=true`
   (named verbatim in the requirements) is a hard `404`, as are `/download`, `/original`, `/file`,
   `/thumb`, `/preview`. Without a byte source there is no migration at all: this alone blocks the task.
2. **`?include=labels,albums,markers,files` — ignored.** The param changes nothing; per-photo albums,
   labels, markers and files are unreachable.
3. **`updated_since` cursor — ignored.** Returns an identical page, so "resuming on the API's cursor"
   cannot be implemented.
4. **Bulk embedding / face-vector feeds — `404`.** So "embeddings and face vectors imported 1:1,
   never recomputed" is impossible.

Two traps for whoever picks this up next:

- **`/openapi.json` and `/swagger.json` return HTTP 200 but serve the SPA's `index.html`** — the
  frontend catch-all answers every unknown path. There is no machine-readable API spec; a 200 alone
  proves nothing, check the body.
- **This spec's dedup premise is wrong.** It states "photo-sorter's `file_hash` is already SHA256".
  The live API's `hash` is **40 hex chars — SHA1** (e.g. `2e591ac93a7ecb1e97c12adf89dc34c66cfda3e9`),
  the same SHA1 PhotoPrism uses. Dedup on SHA256 of the *downloaded bytes* still works as specified,
  but the cross-source match against an already-imported PhotoPrism photo cannot key off
  photo-sorter's `hash` as SHA256. Fix this assumption before implementing.

## Requirements

- New read-only client `internal/photosorterapi` (mirroring `internal/photoprism`): base URL + token auth, typed errors, 429 backoff, paginated listings, streamed original download. The existing `internal/photosorter` (pgx reader) may stay for the integration tests, but the production path no longer uses it.
- `psimport` is rewired to the client: photos (resuming on the API's cursor), albums, labels, subjects, markers, per-photo edits, **embeddings and face vectors** (imported 1:1, never recomputed — verify the reported `model`/`dim` matches what Kukátko queries with, and fail loudly if it does not).
- Originals are downloaded over HTTP (`GET /api/v1/photos/{uid}/download?original=true`) and streamed into storage — never buffered whole in RAM, never read from a local path.
- Dedup is unchanged: SHA256 of the downloaded bytes; photo-sorter's `file_hash` is already SHA256, so a photo already imported from PhotoPrism must be recognised and enriched, not duplicated.
- Config: `import.photosorter.base_url` + `import.photosorter.token` (env `KUKATKO_IMPORT_PHOTOSORTER_BASE_URL` / `_TOKEN`, both already provisioned in `.secrets/db.env`), documented in `config.example.yaml` + `docs/OPERATIONS.md`. Keep `dsn` working for the integration tests, or remove it and say so in the docs.
- The same scoped/partial modes the PhotoPrism import has (`--album`/`--label`/`--person`/`--year`, no watermark advance — commits 14ee9c7, ccddda8) apply here too.

## Access

photo-sorter uses the **same user accounts as PhotoPrism** — `POST /api/v1/auth/login` returns a `session_id` usable as `Authorization: Bearer <session_id>`. Verified against production. That session **expires after 24 hours**, so prefer the long-lived read-only migration token once photo-sorter ships it (its own queued task). Never commit a token.

Production scale for planning: **20,310 photos, 20,092 embeddings, 112,806 faces**.
