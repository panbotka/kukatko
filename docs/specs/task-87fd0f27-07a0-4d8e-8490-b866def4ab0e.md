# Empty, no-result & error states

Replace bare "nothing here" screens and raw errors with thoughtful, consistent states that
tell the user what happened and what to do next — the last layer of polish.

## Requirements

- **A single reusable empty-state pattern:** centered icon (via the `Icon` component,
  bootstrap-icons only) + heading + one line of guidance + an optional primary action, styled
  from the design tokens. Use it consistently everywhere.
- Apply it to the key surfaces that can be empty:
  - Library with no photos ("no photos yet" + how to import/upload).
  - Album / label / people list with no entries.
  - Search / global search with **no results** (echo the query, suggest broadening).
  - Map with no geotagged photos in view.
  - Selection-dependent screens with nothing selected, where relevant.
- **Error states:** when a data view fails to load, show a compact error state with a short
  message and a **Retry** action, instead of a blank screen or a raw error string.
- **Toasts:** ensure success/error toasts are consistent in placement, styling (token surface +
  accent) and duration across the app.
- All new copy goes through i18n (cs default, en). Keep wording short, plain and friendly — no
  developer jargon in user-facing text.

## Verification

- Drive the app in a browser: trigger at least one empty state and one no-result search,
  confirm the shared pattern renders; force a failed load and confirm the error + Retry state.
- `make check` must pass.
