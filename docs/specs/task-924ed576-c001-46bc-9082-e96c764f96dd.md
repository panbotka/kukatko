# M5 — photo-sorter migration

Migrate data directly from a photo-sorter PostgreSQL database into Kukatko, transferring
embeddings and faces 1:1 (same models/dimensions) without recomputation.

## Context
Read `docs/ARCHITECTURE.md` §10. Models/dimensions are identical (CLIP 768 + InsightFace 512), so
embeddings/faces transfer directly. photo-sorter and Kukatko both use SHA256 file hashes, so dedup
is direct. Match to existing photos by `file_hash`; if a photo already exists (e.g. from PhotoPrism
import), just attach embeddings/faces and set `photosorter_uid`. Connect to photo-sorter's DB via a
read-only DSN from config/env (add `import.photosorter.dsn`).

## Requirements
- A `kukatko migrate photosorter` command + admin endpoint `POST /api/v1/import/photosorter` that
  reads photo-sorter's tables and maps into Kukatko:
  - `photos` (metadata) → create or match by `file_hash`; set `photosorter_uid`. Copy originals by
    `file_path` if not already present on disk.
  - `embeddings` (768) and `faces` (512 + bbox + det_score + cache) → insert into Kukatko's
    `embeddings`/`faces` **as-is** (1:1), preserving model/pretrained.
  - `subjects`, `markers`, `albums`/`album_photos`, `labels`/`photo_labels`, `photo_edits`,
    `photo_phashes` → map across (find-or-create subjects/albums/labels by name/slug).
  - **Do NOT** migrate photo book tables or share links.
- Idempotent: re-running matches by `photosorter_uid`/`file_hash` and does not duplicate. Record a
  run in `import_runs` (source=photosorter) with counts. Per-row failures recorded, run continues.

## Quality gate (mandatory)
- Use the **golang-developer** skill. `make check` MUST pass.
- Integration tests (test DB) with a **seeded fake photo-sorter schema** (a second test database
  or fixtures): photos matched/created with `photosorter_uid`; embeddings/faces transferred 1:1 and
  queryable via similarity; subjects/markers/albums/labels mapped; re-run idempotent; book/share
  tables ignored.
