# Import a folder from disk

Right now there is **no way to get a directory of photos into Kukátko except by dragging them into a browser**.
The CLI can serve, migrate, import from PhotoPrism, back up, restore and do maintenance — but it cannot ingest a
folder. Two thousand scanned photos sitting on a disk have no path in. For an app meant to replace PhotoPrism —
whose entire model is "index the originals directory" — that is the biggest hole in the product.

## The command

`kukatko import dir <path>` — recursively walk a directory and ingest every media file into the library.

Flags:
- `--album <uid|title>` — put everything in this album (create it if a title is given and no such album exists)
- `--labels <a,b,c>` — attach these labels to everything
- `--recursive` / `-r` — default **true**; `--no-recursive` for a single flat directory
- `--dry-run` — walk and report what *would* be imported (counts, per-file decisions: new / duplicate / skipped / unsupported), write nothing
- `--concurrency N` — bounded; default small. **This box has 16 GB of RAM shared with everything else** — thumbnailing a wide fan-out of large images will swap the machine solid. Pick a conservative default (2–4) and cap it.
- `--uploader <user>` — which user owns the imported photos; default the bootstrap admin

## Behaviour

- **Reuse `internal/ingest`.** Ingest already streams the file, hashes it (SHA256), extracts metadata, writes the
  original into storage under `YYYY/MM`, generates thumbnails and enqueues the embedding/face jobs. Do not build a
  second ingest path — call the existing one. If ingest is currently too coupled to the HTTP upload handler,
  factor the shared part out rather than duplicating it.
- **Idempotent.** SHA256 dedup already exists: re-running the import over the same folder skips what is already in
  the library and says so. Re-running after adding 10 new files imports 10 files. This must hold — a folder
  import you are afraid to re-run is useless.
- **Resumable.** A run that dies halfway (or gets Ctrl-C'd) leaves the already-imported photos in the library and
  can be re-run to finish. Record the run through `internal/importer` (which already does bookkeeping of import
  runs and high-watermarks) so `/import` in the UI and `GET /import/runs` can show it alongside the PhotoPrism and
  photo-sorter runs.
- **Originals are copied, never moved and never modified.** The source directory is left exactly as it was. (If a
  `--move` mode is ever wanted, it is a separate decision — do not add it here.)
- **Skip rules:** ignore dotfiles, `@eaDir`, `__MACOSX`, Windows/macOS junk (`Thumbs.db`, `.DS_Store`), and
  anything whose MIME is not a supported image or video. Report the skipped count by reason; do not fail the run.
- **Sidecar files** (`.xmp`, `.json`, `.aae`, `.thm`) are not media — do not import them as photos. (Reading them
  for metadata is a separate task; here, just do not choke on them.)
- **Progress**: a readable line-per-file or a periodic counter, plus a final summary (imported / duplicates /
  skipped / failed, elapsed). A per-file failure logs and continues — one corrupt JPEG must not abort a
  2000-file run. Exit non-zero if anything failed, so a script can tell.
- The embeddings sidecar being offline is fine and expected — the jobs queue up in Postgres and drain when the box
  comes back. Say so in the summary rather than looking broken.

## Edge cases

- A file with no EXIF and no parseable date: import it, leave `taken_at` NULL. Do **not** invent a date from the
  file mtime — a wrong date is worse than no date. (Deriving a date from the *filename* is fine if `internal/exif`
  already does that; it does have a `filename` source.)
- HEIC / RAW / video: these already work through `internal/imgconvert` and `internal/video` — verify the folder
  import path hits them.
- A file that is already in the library under a different name: SHA256 catches it; report it as a duplicate with
  both paths so the user can see what happened.
- Symlinks: follow or skip — pick one, do it deterministically, and document it. Do not loop forever.

## Verification

- Integration test over a fixture directory (mixed JPEG / PNG / HEIC-or-skip / a video / junk files / a nested
  subfolder / a duplicate of an already-imported photo): the right files land in the catalogue with the right
  metadata, duplicates are skipped, junk is skipped with a reason, the run is recorded, and **a second run imports
  nothing and reports zero new**.
- A `--dry-run` test that asserts nothing was written.
- `make check` must pass.
- Document: `docs/OPERATIONS.md` (the command and every flag), `README.md` (this is a headline capability —
  "point it at a folder"), `docs/PACKAGES.md` if a package appears.

## Out of scope

- A watched/auto-indexed directory (a daemon that picks up new files by itself) — worth doing later, but this task
  is the manual command it would be built on.
- Reading Google Takeout / Apple XMP sidecars for metadata (separate task).
- Moving or reorganising files on disk.
