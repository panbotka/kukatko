# Show the semantic-search hint only when the embedding service is reachable

On the main library page the search hint advertises semantic search unconditionally, even when the embedding (inference) sidecar — which powers semantic search — is offline (it is "often offline" by design). Gate the semantic-search affordance on a cached reachability flag that the backend maintains by pinging the service periodically. Read `docs/OPERATIONS.md` (embedding config) and `docs/FRONTEND.md` (FilterBar / library) first.

## Current state

- The hint renders in `web/src/components/library/FilterBar.tsx` (~lines 220-227), only on the library page (LibraryPage passes `searchHref`):
  - `library.filters.searchHint` = "Filtruje název a popis." (describes the quick `q` filter box)
  - `library.filters.fullSearchLink` = "Fulltext a sémantické hledání →" (a `<Link>` to `/search`)
- `internal/embedding` exposes `Client.Healthy(ctx) bool` (a live `GET /health`, ~5s timeout) but there is NO cached state — every call is a fresh probe.
- `internal/systemapi` `GET /system/status` pings live and is maintainer-only. There is no all-users capabilities endpoint; the only all-authenticated endpoint is `GET /auth/me` (`RequireAuth`). The `internal/wake` ticker (`Run`/`Tick`) is the template for a periodic background loop.
- The backend search endpoint already degrades gracefully to full-text when embeddings are offline (returns `degraded=true`), so correctness does not depend on this — it is purely presentational.

## Requirements

- **Cached reachability checker (backend):** add a small background service that periodically (≈ every 60s) probes the embedding service via `embedding.Client.Healthy` and stores the result in an `atomic.Bool` (mirror the `internal/wake` ticker structure). If `embedding.url` is not configured (`cfg.Embedding.URL == ""`), the flag is `false` without probing. Wire it into `cmd/kukatko/serve.go` alongside the other background services, reusing the existing `embedding.New(...)` client construction. Do not block startup on it; the value before the first probe is a safe `false` (or probe once at boot).
- **Expose the flag to any logged-in user:** add `GET /api/v1/capabilities` gated `RequireAuth`, returning a small JSON object of instance feature flags, at minimum `{ "semantic_search": <bool> }` (leave the shape open for future flags such as maps-configured). The handler reads the cached `atomic.Bool`. (Alternative acceptable approach: add a `capabilities` object to the existing `GET /auth/me` response — but a dedicated endpoint is preferred because it can be re-fetched cheaply as the box goes on/offline.)
- **Frontend gating:** add a small `web/src/services/` function + a hook/context (e.g. `useCapabilities`) that fetches `/capabilities` on mount and refreshes it periodically (≈ every 60s, or on window focus) so the hint appears/disappears as the box comes online/offline. In `FilterBar.tsx`, show the **"Fulltext a sémantické hledání →" link only when `semantic_search` is true**. Keep the plain "Filtruje název a popis." filter-help text always visible (it describes the quick filter box, unrelated to embeddings).
  - Full-text search still works when embeddings are down; hiding the link is the chosen behavior per the request. (Optionally, when semantic is unavailable you may instead show a fulltext-only variant of the link via a new i18n key — but the default is to hide the semantic link.)

## Tests

- Backend: unit-test the reachability checker (the flag flips with a fake embedding client that toggles `Healthy`; the flag is false when URL unconfigured). Handler test: `/capabilities` returns `semantic_search` reflecting the flag and requires auth (401 unauthenticated).
- Frontend: Vitest — FilterBar shows the semantic link when the capability is true and hides it when false, while always rendering the plain filter-help text; the capabilities hook fetches and updates.
