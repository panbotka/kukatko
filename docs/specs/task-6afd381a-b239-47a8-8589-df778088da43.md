# Cap search-query complexity (authenticated slow-query DoS)

A single authenticated request can force a very expensive query by packing tens of
thousands of alternatives into one `q=` token.

## Root cause (verified)
- `internal/query/parse.go:181-193` (`splitAlternatives`) imposes no bound on the
  number of `|`-alternatives; each compiles to its own `ILIKE $n` term in
  `internal/photos/store_query.go:78-98` (`queryFilterClause`).
- Nothing caps `q` length or the resulting condition count in
  `internal/photoapi/params.go:91-97` (`applyQuery`) or `parseListParams`. `q` is only
  bounded by the ~1 MB header limit.

## Failure scenario
`GET /photos?q=title:a|a|a|…|a` with tens of thousands of `|` produces a WHERE clause
with ~N `ILIKE` terms and N bound parameters, evaluated as a seq scan of tens of
thousands of ILIKEs per row — and it runs TWICE per request (List + Count). Above
~65 535 alternatives it trips PostgreSQL's parameter limit and 500s instead. Either
way a cheap request forces an expensive response.

## Requirements
- Impose a hard cap on query complexity in `parseListParams` (or the parser): a
  maximum number of filter alternatives (and/or filters, and/or a `q` string length),
  rejecting anything past the cap with a 400 and a clear error. A limit of a few
  hundred alternatives is far above any legitimate query.
- The cap must apply to both the query-language path and any equivalent param path.
- Keep normal queries unaffected.

## Testing
- Unit/integration test: a query exceeding the cap returns 400; a normal query
  works. `make check` must pass. Document the limit in `docs/API.md` if user-facing.
