# Fix: a sidecar rewrite is dropped when an edit arrives while its job is running

An edit committed to a photo while its sidecar-export job is already running is
coalesced away and never written, so the on-disk sidecar stays stale.

## Root cause (verified)
- The job dedup partial index covers BOTH queued and running:
  `internal/database/migrations/0005_jobs.sql:40-41`
  (`WHERE state IN ('queued','running')`).
- `internal/jobs/enqueuer.go:85-88` (`EnqueueSidecar`) swallows `ErrDuplicate`.
- `internal/sidecarjob/sidecarjob.go:204-231` (`Export`) reads the photo's state at
  run time.

## Failure scenario
A sidecar job for photo P is claimed and goes `running`; it reads P's state. The user
then commits another edit to P and `EnqueueSidecar(P)` runs — the insert collides with
the dedup index (because it also covers `running`) → `ErrDuplicate` → swallowed → no
rewrite scheduled. The already-running job wrote the sidecar BEFORE seeing the new
edit, so the YAML stays stale until some later edit happens to enqueue a fresh job. If
the DB is lost before that, the edit is unrecoverable — defeating the "curation
survives the database" guarantee.

## Requirements
- An edit that lands while a sidecar job for the same photo is running must result in
  a fresh sidecar write that reflects that edit. Two clean options (pick one):
  1. After `Export`, re-check whether the photo changed since the job was claimed
     (e.g. compare `updated_at` captured at claim vs current) and re-enqueue if so; or
  2. Scope the SIDECAR dedup to `state='queued'` only, so an edit during `running`
     schedules a follow-up job (verify this doesn't cause a tight rewrite loop — the
     queued-state debounce still coalesces bursts).
- Keep sidecar writes idempotent and debounced for ordinary bursts.
- Do not change the dedup semantics of other job types unless scoped specifically to
  sidecar.

## Testing
- Integration test: start (or simulate) a running sidecar job for P, commit an edit +
  enqueue, and assert a rewrite reflecting the edit eventually happens. `make check`
  must pass.
