# Kukátko as an MCP server

Kukátko has a full HTTP API, long-lived API tokens (`kkt_…`) and a `kukatko ctl` client. What it does not have is a
way for an AI agent to *use* the library directly — search it, tag it, organise it, answer questions about it.
photo-sorter exposes an MCP server for exactly this. Given how this library is actually maintained, that is not a
gimmick: "find every photo of grandma from the sixties and put them in an album" is the workflow.

## Scope

Expose an MCP server over the existing HTTP server, mounted at `/mcp`, guarded by the **existing API-token auth**
(`kkt_` Bearer) and the **existing RBAC** — an agent holding a viewer token can read and nothing else; the `ai`
role already exists for exactly this kind of automation. **Do not invent a second auth mechanism** and do not add a
bypass.

**Call the services in-process.** Do not have the MCP server make HTTP calls back into its own API — it is slower,
it doubles the failure modes, and it loses the transaction boundaries. Wire it to the same service layer the HTTP
handlers use.

## Tools to expose

Enough to be genuinely useful, not the whole API surface. Start with:

**Read**
- search photos (free text; and the query language if that task has landed — check, and use it if it is there)
- get a photo's details (metadata, albums, labels, people)
- find photos similar to a given photo
- list / get albums, labels, subjects (people)
- list a subject's photos, an album's photos, a label's photos
- library statistics (counts) — cheap and constantly useful for answering "how many…"

**Write** (write-scoped tokens only)
- create an album; add / remove photos from an album
- create a label; attach / detach a label
- set a photo's title / description / notes
- set favourite / rating
- the bulk edit (`POST /photos/bulk`) — one tool, since an agent operating on 200 photos one call at a time is
  both slow and a great way to half-apply a change

**Deliberately NOT exposed:** anything destructive or irreversible — no purge, no empty-trash, no restore, no user
administration, no backup. An agent must not be able to delete photos. Say this explicitly in the docs so nobody
"completes" the tool list later.

## Design rules

- **Tool descriptions are the interface.** An agent picks a tool by reading its description; a vague one means the
  agent calls the wrong thing. Write them as carefully as you would write the API docs — say what the tool does,
  what the arguments mean, and what comes back.
- **Return compact, structured results.** An agent's context is the scarce resource: a search tool that returns 50
  full photo objects with EXIF blobs is unusable. Return the minimum useful shape (uid, title, date, thumb URL)
  and let the agent fetch details for the ones it cares about. Paginate, and say how many more there are.
- **Never return the raw `exif` JSONB blob** from a list tool.
- Pure Go, `CGO_ENABLED=0` — that is a binding project constraint. Pick an MCP library that respects it, or
  implement the protocol directly if the available ones do not.
- Every mutating tool call goes through `internal/audit`, in the same transaction, like every other mutation — an
  agent's actions must be as traceable as a human's. This is the whole reason the audit trail exists.
- The server is **off by default** and enabled by a config key. It is a new attack surface; make it opt-in.

## Verification

- Integration tests over the MCP transport: a viewer token can read and is refused on every write; a write token
  can create an album and attach a label; a destructive operation is not exposed at all; a search returns the
  compact shape and not the EXIF blob; every mutation writes an audit row.
- A test that the server is genuinely disabled when the config key is off (the route is not mounted at all, not
  merely 403).
- `make check` must pass — including `CGO_ENABLED=0`.
- Document: `docs/API.md` or a new `docs/MCP.md` (the tool list, the auth model, and the explicit list of what is
  deliberately not exposed), `docs/OPERATIONS.md` + `config.example.yaml` (the switch, the mount path, how to mint
  a token for an agent), `docs/PACKAGES.md` + a line in the `CLAUDE.md` package map, `README.md`.

## Out of scope

- Any destructive tool.
- A hosted/remote MCP transport beyond what the existing HTTP server provides.
- Tools for the admin surface (jobs, maintenance, backup, users).
