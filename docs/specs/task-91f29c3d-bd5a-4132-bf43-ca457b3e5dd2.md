# Archive a photo from the photo detail view

Let a user archive (send to trash) the currently open photo directly from the single-photo detail page, without going back to the grid to use multi-select.

## Background
Archiving is currently only possible via multi-select bulk actions in the grid (`BatchActionBar` / `BulkEditModal`); the photo detail page (`web/src/pages/PhotoDetailPage.tsx`) has no archive control. The backend already exposes single-photo endpoints:
- `POST /api/v1/photos/{uid}/archive` — soft-delete, moves the photo to trash (sets `archived_at`). Requires write role (editor+).
- `POST /api/v1/photos/{uid}/unarchive` — restores from trash (clears `archived_at`). Requires write role (editor+).

The photo object from the detail API carries its archived state (an `archived_at` / `archived` field). Check the existing photo types and `unarchivePhoto()` in `web/src/services/photos.ts`.

## Requirements
- On the photo detail page, add an **Archive** action for editor+ users. Reuse the app's existing write-permission gating (the same guard the bulk archive button uses). Viewers must NOT see it.
- Place the control among the other per-photo actions in the detail header/toolbar (next to favorite/edit/download). Use the `Icon` component with a `bootstrap-icons` archive icon and an accessible label.
- Clicking **Archive** calls the archive endpoint for the current photo. On success, reflect the new state: update the on-page photo state to "archived" and swap the button to **Restore** (prefer keeping the user on the page over navigating away, unless the page already has an established post-mutation navigation pattern to follow).
- If the open photo is **already archived** (e.g. opened from the Trash page `/trash`), the same control shows **Restore** and calls the unarchive endpoint. After restoring, swap back to **Archive**.
- Add an `archivePhoto` service function in `web/src/services/photos.ts` if one does not exist, following the existing `unarchivePhoto()` pattern.
- Show a success and an error toast/notification, consistent with how other detail-page actions report results.
- i18n: reuse existing strings if they fit (`batch.archive` = "Archive"/"Archivovat", `trash.restore` = "Restore"/"Vrátit z koše"), otherwise add detail-scoped keys in both `web/src/i18n/locales/cs/common.json` and `.../en/common.json`. Czech: "Archivovat" / "Vrátit z koše"; English: "Archive" / "Restore".
