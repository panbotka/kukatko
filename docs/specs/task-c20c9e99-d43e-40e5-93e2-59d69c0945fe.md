# M6 — Durable audit log

Add an append-only audit log written **in the same transaction** as the mutation it records
(fixing photo-sorter's after-commit gap), plus an admin API to view it.

## Context
Read `docs/ARCHITECTURE.md` §5 (audit_log), §11, §15. Audit must not be lost if the process
crashes between a mutation commit and a separate audit insert — so the audit insert participates
in the mutation's transaction.

## Requirements
- Migration: `audit_log` (id BIGSERIAL PK, user_uid FK SET NULL, action TEXT, entity_type TEXT,
  entity_uid TEXT, metadata JSONB, ip TEXT, user_agent TEXT, created_at TIMESTAMPTZ). Index on
  (created_at), (entity_type, entity_uid).
- `internal/audit` package exposing a transaction-aware `Record(ctx, tx, entry)` so callers write
  the audit row inside the same DB transaction as the change. Provide a helper pattern other
  packages adopt for mutations (create/update/delete/bulk/assign/import).
- Wire audit into the key mutating paths added so far where feasible (photos metadata,
  archive/restore, bulk edit, album/label changes, people assignment, user management). At minimum
  provide the mechanism + cover several representative paths; document the convention so future
  tasks use it.
- Admin API: `GET /api/v1/audit` with filters (user, entity, action, date range) + pagination.

## Quality gate (mandatory)
- Use the **golang-developer** skill. `make check` MUST pass.
- Integration tests (test DB): audit row is written within the mutation transaction and **rolls
  back with it** on failure (no orphan audit, no missing audit on success); audit query filters +
  pagination; admin-only access.
