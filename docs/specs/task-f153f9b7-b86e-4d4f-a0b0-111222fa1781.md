# Maintainer announcement banner for all users

Let a maintainer publish a short message (e.g. "expected downtime tonight 22:00–23:00") from the admin area; every logged-in user then sees it as a banner at the top of the app. No such mechanism exists today — build it, mirroring existing patterns. Read `docs/PACKAGES.md`, `docs/API.md`, and `docs/FRONTEND.md` first.

## Backend (mirror `internal/savedsearch` store + `internal/organizeapi` dual-guard API)

- **Migration** `internal/database/migrations/0038_announcement.sql` (verify the next free number): a single-row `announcements` table — `message TEXT NOT NULL`, `level TEXT NOT NULL DEFAULT 'info'` (info|warning), `author_uid VARCHAR(32) REFERENCES users(uid) ON DELETE SET NULL`, `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()`. Enforce a single row (e.g. `id BOOLEAN PRIMARY KEY DEFAULT true CHECK (id)`) so publish is an upsert.
- **Package** `internal/announcement`: a `Store` over the shared pgx pool (mirror `savedsearch/store.go`, `models.go`) with `Get(ctx) (Announcement, error)` (returns a not-found sentinel or an empty announcement when none), `Set(ctx, message, level, authorUID) (Announcement, error)` (upsert), `Clear(ctx) error`.
- **API** `internal/announcementapi` (mirror `organizeapi`'s dual-guard `Config`): routes under `/announcement` — `GET /` gated `RequireAuth` (any signed-in user reads the current message), `PUT /` and `DELETE /` gated `RequireMaintainer` (publish / clear). GET with no announcement returns `200 {"message":""}` (friendlier for the banner client than 404).
- **Audit** the publish and clear in the same transaction (new actions e.g. `announcement.set` / `announcement.clear`, recording the message/level) via `internal/audit`.
- **Wiring:** `cmd/kukatko/announcement.go` `buildAnnouncementAPI(db, authAPI)` injecting `RequireAuth` + `RequireMaintainer`; register in `cmd/kukatko/serve.go` next to the peer APIs (e.g. savedsearch).

## Frontend

- **Service** `web/src/services/announcement.ts` (mirror `savedSearches.ts`): `fetchAnnouncement()`, `setAnnouncement(message, level)`, `clearAnnouncement()` with typed shapes.
- **Banner for all users:** a `useAnnouncement` hook (fetch on mount + light polling, ≈ every 60s, so a freshly published message appears without reload) and a dismissible react-bootstrap `<Alert>` rendered in `web/src/components/Layout.tsx` immediately before `<Outlet/>` (top of the content area), variant driven by `level` (info/warning), with a decorative `Icon` (e.g. `info-circle` / `exclamation-triangle`).
  - **Per-user dismiss keyed on `updated_at`:** store the dismissed announcement's `updated_at` in localStorage (mirror `web/src/lib/faceOverlayPref.ts`), so dismissing hides the current message but a newly published message (new `updated_at`) reappears — do NOT use a plain boolean.
  - **Shell caveat:** routes rendered outside `Layout` (`/photos/:uid`, `/slideshow`, `/review`, `/duplicates/compare`) will not show the banner — acceptable (immersive views); note it in the docs.
- **Compose control (maintainer):** add an "Announcement" card to `web/src/pages/SystemStatusPage.tsx` (already maintainer-gated) with a textarea + level select + Publish and Clear buttons, mirroring the existing async-action + `ActionNotice` `<Alert>` feedback pattern on that page. Gate it on `useAuth().isMaintainer`.
- **i18n:** `announcement.*` keys in `web/src/i18n/locales/{cs,en}/common.json`, Czech default.

## Tests

- Backend: integration/HTTP — `GET /announcement` readable by any authed user (viewer 200); `PUT`/`DELETE` require maintainer (editor/admin 403, maintainer 200); publish then read returns the message; set/clear write an audit row.
- Frontend: Vitest — the banner renders the fetched message for a normal user; dismiss persists (keyed on `updated_at`) and a new `updated_at` re-shows it; the compose control publishes/clears and is hidden for non-maintainers.
