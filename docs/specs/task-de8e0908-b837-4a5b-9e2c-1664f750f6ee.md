# Micro-interactions, motion & loading states

Make the app feel alive and responsive with restrained motion and modern loading states,
replacing bare spinners and abrupt content pops.

## Requirements

- **Image load-in:** grid and viewer thumbnails fade (and subtly scale) in once decoded rather
  than snapping in; a low-cost placeholder (token surface, or a blurred/low-res preview if one
  is already available) holds the tile's space to avoid layout shift.
- **Skeleton loaders:** replace full-page spinners on the main data views (library grid, album/
  people lists, detail panels) with skeleton placeholders shaped like the eventual content.
- **Transitions:** consolidate hover, focus and open/close transitions onto a small set of
  duration/easing tokens so motion feels consistent app-wide (nav active state, buttons, cards,
  panels, the batch bar).
- **Respect `prefers-reduced-motion`:** when the user prefers reduced motion, disable non-
  essential animation (fades/scales/slides) and fall back to instant state changes.
- No janky or long animations — keep everything short (~120–200ms) and GPU-friendly
  (transform/opacity), no layout-thrashing properties.
- Introduce motion tokens in `tokens.css`/`app.css` and use them; no hardcoded durations
  scattered across components.

## Verification

- Verify visually via the static harness and a throwaway instance: watch images fade in,
  confirm skeletons appear on load, and confirm animations are suppressed under
  `prefers-reduced-motion`.
- `make check` must pass.
