# Make the floating bulk-action bar responsive on mobile

When photos are selected, the floating bottom bulk-action bar (`BatchActionBar`)
holds ~10 labelled buttons. On a 360px phone they wrap into a tall multi-row block
that floats over the photo wall, and the grid does not reserve enough bottom space,
so the last rows of photos stay hidden behind the bar and cannot be scrolled clear.

## Current behavior (root cause)

- `web/src/components/organize/BatchActionBar.tsx` (around lines 201-258) renders
  ~10 always-labelled controls in one row: ✕ (clear), the selected count, "Vybrat
  vše", "Přidat do alba", "Štítky", "Oblíbené", "Archivovat", download (ZIP),
  "Seskupit" (stack), "Další úpravy".
- `web/src/styles/app.css` (around lines 388-400) styles `.kk-batch-bar` with
  `flex-wrap: wrap` and no `max-height` / internal scroll, so on a narrow screen the
  buttons wrap to roughly 3-4 rows (~12rem tall).
- The page reserves a FIXED bottom clearance: `web/src/pages/LibraryPage.tsx`
  (around line 264) sets `paddingBottom: '6rem'`, which is less than the wrapped
  bar's real height, so the bottom photos are covered. (The safe-area inset itself
  IS handled — `bottom: calc(env(safe-area-inset-bottom,0px) + …)` — so the home
  indicator is not the issue; the bar's height/coverage is.)

## Requirements

- On narrow screens (phones), the bulk-action bar must not consume a large portion
  of the screen or hide photos behind it. Collapse the actions so the bar stays a
  single compact row: e.g. icon-only buttons, a single horizontally-scrolling row,
  and/or an overflow "…" menu that holds the secondary actions. Keep the clear (✕)
  and the selected count always visible; keep the most common actions (add to album,
  labels) directly reachable; the rest may move into the overflow menu.
- The grid's bottom clearance must always equal the bar's ACTUAL rendered height so
  the last row of photos can be scrolled fully clear of the bar, regardless of how
  many actions are shown or how the bar wraps. Prefer measuring the bar's height
  rather than a hard-coded constant.
- Preserve the existing desktop layout and all actions — this is purely about the
  bar's responsive behavior and the grid clearance.
- Keep the existing safe-area handling intact.

## Implementation notes

`BatchActionBar` is currently rendered by the Library and (per other planned work)
may be rendered on more list pages, so make the responsive behavior a property of
the component itself, not of one page. For the clearance, a resize-observer / ref
measuring the bar height and feeding the grid's bottom padding is the reliable
approach; if that is too heavy, at least make the reserved padding scale with the
number of action rows on narrow screens. Keep it typed (no `any`).
