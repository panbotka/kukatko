# Audit coverage: trash purge and user administration

## Summary
Wire the existing audit trail into the two most sensitive unaudited areas: **permanent deletion** (trash purge) and **user administration**, so the audit log records who permanently deleted photos or changed user accounts, and when.

## Background (already in place â€” follow the established pattern)
- `internal/audit.Write(ctx, exec, Entry)` runs inside the mutation's `pgx.Tx` (atomic). Build the envelope with `audit.FromRequest(r, actorUID)` (actor from `auth.UserFromContext`), stamp with `meta.Entry(...)`.
- The API-token handlers in `internal/auth` already do exactly this (`CreateAPITokenAudited`/`RevokeAPITokenAudited` via `inAuditedTx` in `store_apitoken.go`), so user-admin handlers can follow their sibling code. Also see `internal/photos/audit.go` (`mutateAudited`).
- `ActionUserCreate/Update/Disable/Password` already exist and are unused. Add a **new** action constant for permanent purge (none exists yet).
- Note: reversible archive/unarchive is already audited; this task covers the **irreversible** purge and user changes.

## Requirements
Each of these writes one audit entry, atomically in the same transaction as the change, capturing the actor, target UID, and useful `details`:
- **Trash**: permanently purge a photo, empty trash, and scheduled retention purge (`PurgePhoto`, `EmptyTrash`, `PurgeExpired`). For scheduled/system-initiated purges with no HTTP actor, record a system actor (an empty or `system` actor UID is acceptable) and the purged photo UID(s)/count in `details`.
- **Users**: create, update, disable, password reset.
- Atomic with the mutation (rollback â‡’ no audit row).
- No change to endpoint responses.

## Implementation notes
- Trash: `internal/trash/` (no audit import today). HTTP-triggered purges have an actor via context; the scheduled `PurgeExpired` runs without a request â€” use a system/empty actor and still record the target(s).
- Users: `internal/auth/` user CRUD handlers/service/store (`routes.go`, `service.go`, `store.go`) â€” mirror the existing `inAuditedTx` approach already in `internal/auth`.
- Record identifying `details` for deletes (which photo(s) purged; which user changed and what fields).

## Tests
- Integration tests asserting each mutation inserts the expected `audit_log` row (action/target/actor/details), that a rolled-back mutation writes none, and that scheduled `PurgeExpired` records a system-actor entry. Follow existing audit integration test patterns.
