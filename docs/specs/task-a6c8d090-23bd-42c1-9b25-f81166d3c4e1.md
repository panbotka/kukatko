# Compare two duplicates side by side

`/duplicates` finds near-duplicate groups (pHash + embedding) and can merge a group into a keeper. What it cannot
do is help you **decide which one to keep**. The two photos differ — one is 12 MP and one is 2 MP, one has the
right date and one does not, one came from the camera and one from WhatsApp — and none of that is visible until
you open both in separate tabs and squint.

photo-sorter has a compare view for exactly this, and it is the thing that makes duplicate review actually
finishable.

## The view

From a duplicate group on `/duplicates`, open a **compare** view for two photos:

- The two photos **side by side**, as large as the viewport allows. On a narrow screen, stacked.
- **Synchronised zoom and pan** — zoom into the left photo and the right one follows. This is what lets you see
  that one is a soft JPEG re-encode of the other. Without it, the comparison is guesswork.
- A **difference table** between them, with the differing rows **visibly marked** — that is the whole point;
  identical rows are noise. Compare at minimum: dimensions and megapixels, file size, format/MIME, capture date,
  camera and lens, file name, location, and which albums / labels / people each one carries (this last one
  matters — the "worse" file is sometimes the one with all your curation on it, and merging blindly would throw
  it away).
- Three actions: **Nechat levou** / **Nechat pravou** / **Nechat obě**.
  - Keep-one routes through the **existing** `POST /duplicates/merge` (`internal/dupmerge`), which already unions
    albums, labels and people onto the keeper, fills metadata gaps and archives the copies. Do not write a second
    merge path.
  - Keep-both dismisses the pair so it is not offered again. **Persist that** — a "these are genuinely different"
    decision that evaporates on reload means the same pair comes back forever, which is exactly the failure mode
    this whole batch of work is fixing. If `internal/feedback` (from the review-features work) exists by then, put
    it there; otherwise add a small persistent dismissal for duplicate pairs.
- A **dry-run preview** of what merging would do (`POST /duplicates/merge` already supports `dry_run`) — show it
  before the destructive action, since the loser gets archived.

## Navigation

- Keyboard: `←` keep left, `→` keep right, `b` keep both, `Esc` back to the group list. Register in the existing
  `?` shortcuts overlay.
- After a decision, advance to the **next pair** without going back to the list. Duplicate review is a queue; make
  it feel like one.
- A group with more than two members: compare them pairwise against the suggested keeper, or let the user pick two
  — either is acceptable, but say which you did and make it obvious in the UI. Do not silently ignore the third
  member.

## Safety

- The loser is **archived, never deleted**. Purging is a separate, explicit act from `/trash`. State this in the
  confirmation.
- Show what the merge will carry over (albums, labels, people, metadata gaps filled) before it happens.

## i18n

Every string into **both** `web/src/i18n/locales/cs/common.json` and `.../en/common.json`.

## Verification

- Vitest: the diff table marks exactly the differing rows and leaves identical ones unmarked; synchronised zoom
  moves both images; keep-left calls merge with the right keeper; keep-both persists a dismissal and the pair is
  gone from the next load; the keyboard shortcuts fire; after a decision the next pair appears.
- Integration test for the dismissal persistence (whatever store it lands in): dismissing a pair removes it from
  `GET /duplicates`, and it stays gone across a re-scan.
- `make check` must pass.
- Document: `docs/FRONTEND.md` (the compare view), `docs/API.md` if a dismissal endpoint appears,
  `docs/PACKAGES.md` if a package changes, `README.md`.

## Out of scope

- Changing the duplicate detection algorithm.
- Permanently deleting anything.
