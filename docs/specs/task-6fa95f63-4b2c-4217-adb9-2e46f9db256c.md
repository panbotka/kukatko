# M6 — Observability

Add Prometheus metrics and consistent structured logging across the app.

## Context
Read `docs/ARCHITECTURE.md` §14 (observability). Mirror photo-sorter's Prometheus approach. Keep
it lightweight.

## Requirements
- Expose `GET /metrics` (Prometheus) — consider protecting it or binding appropriately. Instrument:
  HTTP request count/latency by route+status, job queue depth by state/type, job execution
  duration/outcome, embeddings sidecar call latency/availability, import progress counters,
  thumbnail generation duration, DB pool stats.
- **Structured logging** (slog) with consistent fields (request id, user, route, job id/type),
  configurable level; do not log secrets (mapy key, S3 keys, tokens, passwords).
- A request-ID middleware that ties logs + metrics together.

## Quality gate (mandatory)
- Use the **golang-developer** skill. `make check` MUST pass.
- Unit/integration tests: `/metrics` exposes expected metric names; HTTP middleware records
  request metrics + request id; a job run increments job metrics; secret values never appear in
  emitted log lines (test a redaction/no-secret guarantee on representative paths).
