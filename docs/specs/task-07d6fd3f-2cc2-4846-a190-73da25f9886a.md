# Album Covers And Year Ranges

`/albums` shows every album as a grey "Bez náhledu" square, because `AlbumTile` renders the cover only when `cover_photo_uid` is set, and that field is only ever assigned by hand from the album detail page. No album has one. The list also gives no sense of *when* an album is from, even though the names carry years ("Pouť 2024", "Sraz rodáků 2016").

## Requirements

### Backend

- The album list response gains an effective cover photo: the manually chosen cover when one is set, otherwise a deterministic fallback such as the album's newest photo. The same album must return the same fallback on every request — no random or unordered pick.
- The album list response gains the range of capture dates across the album's photos (earliest and latest).
- Aggregate in SQL. Do not fetch an album's photos to compute this. No schema migration should be needed; if a query turns out to be slow on a large library, add an index and record the reasoning in `docs/PERF.md`.
- Respect the caller's visibility rules: photos the user cannot see must not supply the cover or move the range.

### Frontend

- The album tile renders a thumbnail of the effective cover. The grey placeholder appears only for a genuinely empty album, and then as the shared empty state, not a bare box.
- Under the album title, show the capture range compactly, always on one line: a single year as "2006"; a span as "1998–1999"; and when every photo falls inside one month, "6/2007". An empty album shows nothing.

## Verification

`make check` must pass. Add an integration test against the real test database covering the fallback cover and the aggregated range, and a unit test for the range formatter exercising all three shapes plus the empty album. Update `docs/API.md` and `docs/PACKAGES.md`.
