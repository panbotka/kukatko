# Fix photo-sorter Migration Gaps

`docs/MIGRATION_AUDIT.md` audits the photo-sorter → Kukátko field mapping and marks every source field **MAPPED**, **WAIVED** (deliberately dropped, with a reason) or **GAP** (a real loss nobody decided on). Fix the GAPs on the photo-sorter migration path.

## Requirements

- Read `docs/MIGRATION_AUDIT.md`, section "photo-sorter → Kukátko". **If that file or section does not exist, stop and report — do not guess what the gaps are.** This task's whole input is that document.
- Fix every row marked GAP whose recommended fix lands in `internal/psimport` or `internal/photosorter`. Do NOT touch rows marked WAIVED — those are deliberate decisions (`AlbumPhoto.SortOrder`, the `Slug` columns Kukátko re-slugifies, `ps.UpdatedAt` as a watermark), and re-litigating them is out of scope.
- The migration must stay INCREMENTAL and IDEMPOTENT — re-running it must not duplicate rows or clobber Kukátko-side edits. Existing integration tests in `psimport_integration_test.go` and `fixture_integration_test.go` assert this and must keep passing.
- If a GAP is a whole photo-sorter table nobody reads, adding it is a bigger change than a field fix. Do it only if the audit recommends it AND it fits this session; otherwise say so plainly and leave the row as a GAP rather than half-migrating a table.
- If a fix needs a new column, add a migration in `internal/database/migrations` following the existing ascending numbering, and grep the WHOLE repo for other writers of that table — a new column other packages do not fill will pass `make check` and still be wrong.
- If a GAP turns out to need a product decision rather than code, do NOT invent a column. Leave the code and flip that row to WAIVED in the audit document with the reasoning.

## Widen the mapping test regardless of what the audit found

`psimport.TestBuildPhoto` in `helpers_test.go` asserts only `FileHash`, `FilePath`, `FileSize`, `Title`, `Private`, `FileOrientation`, `PhotosorterUID` — 7 of roughly 23 fields that `buildPhoto` (`psimport/photos.go` ~line 140) actually maps. The other ~16 have no test proving they survive, so a regression there is invisible. Widen this test to cover every field `buildPhoto` maps. Do this even if the audit reports zero gaps — it is the guardrail that keeps this mapping honest.

## Scope guard

If the audit lists more gaps than fit in one session, fix them highest-impact first — a whole unread table or a silently dropped column beats a cosmetic field — and leave the remaining GAP rows in the document untouched so the rest stays visible. State plainly in your final report which gaps you fixed and which you left, and why.

## Verification

- Every fixed field needs a test asserting it survives a fresh migration and a re-run.
- Integration tests run against a real test DB — `KUKATKO_TEST_DATABASE_URL` in `.secrets/db.env` (database `kukatko_test`, safe to truncate). The photo-sorter source DSN is `import.photosorter.dsn`; the reader is read-only and must stay that way — never write to the photo-sorter database.
- `make check` must pass.
- Update `docs/MIGRATION_AUDIT.md`: flip each fixed row from GAP to MAPPED. Update `docs/PACKAGES.md` if the migration's documented behaviour changed.
