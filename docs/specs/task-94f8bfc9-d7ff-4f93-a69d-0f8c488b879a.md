# Audit log: record the previous value on edits (old → new)

Audit records currently store only the new value (or just the names of changed fields). Record the **previous** value alongside the new one for edits, so the log shows e.g. a caption change `stary popisek` → `novy popisek`. Read `docs/PACKAGES.md` (audit), `docs/API.md` (audit), and `docs/FRONTEND.md` (AuditPage) first.

## Convention

- Establish a documented convention in `internal/audit`: an edit's `Details` carries a `changes` map of the form `{ "<field>": { "old": <value>, "new": <value> } }`, containing **only the fields whose value actually changed**. Add a small helper in `internal/audit` to build this map from old/new values (skipping unchanged fields), so all call sites use it consistently. (The `Entry.Details map[string]any` already exists; this is a payload convention, not a schema change.)

## Apply to these edit paths

- **Photo metadata edit** — `internal/photoapi/update.go` `handleUpdate`. It already loads the current row (`current, _ := a.store.GetByUID(...)`), so old values are in scope. For each edited field that changes (title, description, notes, ai_note, subject, keywords, artist, copyright, license, scan, taken_at, location, location_source, etc.), record old→new in the `changes` map. Keep the existing `fields` list if useful; the diff is the deliverable. Update the MCP write path (`internal/mcpapi/tools_write.go`, same `photo.update` action) for consistency if straightforward.
- **Album update** — `internal/organizeapi/albums.go` `handleAlbumUpdate`. It already loads `existing`; record old→new for the editable album fields (title, description, cover, private).
- **Subject update** — `internal/peopleapi/handlers.go` `handleUpdate`. Add a `GetSubjectByUID` before building the entry (mirror the delete handler which already loads it) to obtain the old name/type, and record old→new.
- **Label update** — `internal/organizeapi/labels.go` `handleLabelUpdate`. Add a `GetLabelByUID` before the entry to obtain the old name/priority, and record old→new.

## Out of scope

- **Bulk metadata edit** (`internal/bulk`) is intentionally excluded: it applies a single `UPDATE` across many photos without loading old rows; per-photo SELECT-before-UPDATE would double the statements per batch. Leave its existing summary details as-is (note this in the docs).

## Frontend

- In `web/src/pages/AuditPage.tsx` (the expandable details row currently dumps `JSON.stringify(details)`), render a `details.changes` map as a compact **old → new** table (field, old value, new value) when present, falling back to the current raw-JSON rendering for records without a `changes` key (legacy rows and non-edit actions). Add the `changes` shape to the `AuditRecord`/details typing in `web/src/services/audit.ts` if helpful.

## Tests

- Backend: for each of the four paths, an integration/handler test asserting that changing a field writes `details.changes[field] = {old, new}` with the correct values, and that unchanged fields are absent (and no `changes` entry when nothing changed).
- Frontend: Vitest — AuditPage renders a changes map as an old→new table and falls back to JSON for records without `changes`.
