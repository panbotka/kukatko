# Serve Media Through The Worker

With originals and thumbnails in a private R2 bucket behind a Worker, the application should stop transferring image bytes. It runs on a VPS with a small disk and a shared link; every thumbnail it proxies is wasted bandwidth.

## Requirements

- Photo payloads returned by the API gain `thumb_url` and `download_url`, produced by the storage backend's `URL` method. With the filesystem backend these keep pointing at the existing application routes, so nothing breaks locally. With the R2 backend they are short-lived signed URLs pointing at the Worker's domain.
- The frontend uses those URLs directly in `img` elements and download links. It must stop constructing media URLs from a photo UID.
- Because a signed URL expires (one hour by default), the frontend must tolerate a stale URL: an image that fails to load is retried once after refetching the photo payload, rather than showing a broken image forever. Do not paper over this with a very long TTL.
- Keep `GET /photos/{uid}/thumb/{size}` and `GET /photos/{uid}/download` as routes, now answering with a redirect to the signed URL. Existing links and bookmarks must keep working.
- Authorization still gates discovery: a user who may not see a photo must not receive a signed URL for it. The object itself is gated by the signature the Worker verifies. Unlike the earlier public-bucket design, the `private` flag and the archive remain **real security boundaries** — say so in a code comment, because the previous design deliberately gave that up and someone will remember the old rule.
- Video: playback points at the Worker, which streams Range requests straight from R2. The current implementation uses `http.ServeContent` and therefore needs a seekable local file; that requirement goes away.
- Transcoding and probing currently shell out to `ffmpeg` and `ffprobe` against a local path. Both accept an HTTP URL as input. **Verify this against the actual binaries on the target system before relying on it.** If it works, feed them the signed URL; if it does not, fall back to `Materialize` and say so in the code.

## Verification

`make check` must pass. Add an integration test asserting that with the filesystem backend the thumb route still streams bytes, and that with the R2 backend it answers a redirect to a signed URL whose signature verifies. Add a frontend test that a photo tile renders the `thumb_url` it was given rather than a constructed path, and one for the stale-URL retry. Update `docs/API.md` and `docs/FRONTEND.md`.
