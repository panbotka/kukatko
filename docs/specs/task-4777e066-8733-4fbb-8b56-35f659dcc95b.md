# Cloudflare R2 Storage Backend

Kukátko will run on a VPS whose disk cannot hold a 120 GB photo library. Originals and thumbnails move to Cloudflare R2. This task adds the backend; separate tasks add the Worker that serves the objects and the code that hands out URLs.

The `Storage` interface already offers `URL` and `Materialize` instead of `AbsPath`, and `internal/backup` already depends on minio-go v7 with a configurable, non-AWS endpoint. R2 is S3-compatible. Reuse both. Do not add another S3 client library.

## Object keys

Keys derive from values already stored in Postgres: `photos.file_path` and `photo_files.file_path` are stored verbatim rather than recomputed, so the object key can simply *be* that value. **No new column and no key migration.** Thumbnails keep their existing hash-derived cache layout, which becomes the object key directly.

Objects are **private**. A key is not a secret and does not need to be unguessable — a request without a valid signature is rejected before it reaches the object. Do not invent a random key scheme.

## Signed URLs

`URL(relPath)` returns:

```
https://<media-base-url>/<object-key>?exp=<unix-seconds>&sig=<hex HMAC-SHA256>
```

The HMAC covers the object key and the expiry. Support **two signing secrets at once** — current and previous — so the secret can be rotated without a window of broken URLs. Default TTL one hour.

## Requirements

- Implement `Storage` over minio-go. `Store` streams; never buffer a whole file in memory. `Materialize` downloads to `storage.temp_path` and its cleanup removes the file, including on the error path — a leaked temp file on a small disk is a real failure.
- Hard links do not exist in object storage and are not needed: `PutObject` is atomic, and deduplication is enforced by the unique constraint on `photos.file_hash`.
- New configuration keys: `storage.backend` (`fs` default, or `r2`); `storage.r2.endpoint`, `.region`, `.bucket`, `.access_key`, `.secret_key`; `storage.r2.media_base_url`; `storage.r2.url_signing_secret` and `.url_signing_secret_previous`; `storage.r2.url_ttl`; `storage.temp_path`. Add them to the `Config` struct, `setDefaults`, `config.example.yaml`, the config tests, and `docs/OPERATIONS.md`.
- `storage.backend: fs` stays the default so existing deployments and tests are unaffected.
- Fail fast at startup with a clear error when `r2` is selected and a required key is missing.
- Never log a signing secret or an access key.

## Verification

`make check` must pass. Add integration tests against a real S3-compatible endpoint (MinIO is acceptable) covering store, open, stat, delete, and materialize-and-cleanup, including a key containing a UTF-8 filename. Add unit tests for URL signing: a valid signature verifies, a tampered key or expiry does not, an expired URL is rejected, and a URL signed with the previous secret still verifies. Update `docs/PACKAGES.md` and `docs/ARCHITECTURE.md`.
