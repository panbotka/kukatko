# Migration Audit — photo-sorter → Kukátko

Produce a complete, field-by-field audit of what the photo-sorter migration carries into Kukátko and what it drops, and add it to `docs/MIGRATION_AUDIT.md` as the section "photo-sorter → Kukátko". This task DOCUMENTS; it does not change import behaviour.

## Why

Kukátko absorbs photo-sorter, and `internal/psimport` is the only path that library takes into the new database. No document enumerates source field → target column; `docs/PACKAGES.md` (~lines 1529-1557) describes the migration in Czech prose but is not a mapping table. The owner needs certainty that nothing is silently lost.

## The document

`docs/MIGRATION_AUDIT.md` may already exist with a "PhotoPrism → Kukátko" section. If so, ADD your section alongside it and leave the existing one untouched. If the file does not exist, create it — same structure, in Czech (it is a report for the project owner, and the repo's docs are Czech).

One table per entity. Every row: source field → target column → verdict. Exactly three verdicts:

- **MAPPED** — the value lands in a column. Note any precedence/fallback rule.
- **WAIVED** — deliberately not carried, WITH the reason (e.g. `AlbumPhoto.SortOrder` is dropped because Kukátko orders chronologically; `Album.Slug`/`Label.Slug`/`Subject.Slug` because Kukátko re-slugifies; `ps.UpdatedAt` is a watermark, not a column).
- **GAP** — a real loss nobody decided on. State what is lost, when it bites, and a recommended fix.

Coverage must be provable in BOTH directions: every source field defined in `internal/photosorter/models.go` appears in a table, and every Kukátko column the migration is meant to populate is accounted for. Open your section with a summary: how many fields audited, mapped / waived / gaps.

## Concrete leads to confirm or refute

Verify each against the code — do not take them on trust, and do not stop at them.

1. **Does photo-sorter have tables nobody reads?** `internal/photosorter` deliberately never touches photo-book and share-link tables (both are out of scope for Kukátko). Confirm by listing photo-sorter's actual schema that no OTHER table holds data worth migrating. This is the highest-value check in this task: a whole unread table is a bigger loss than any single field. The photo-sorter DSN is `import.photosorter.dsn`; the repo also has a read-only client you can lean on.
2. **`psimport.TestBuildPhoto` is thin** — it asserts only `FileHash`, `FilePath`, `FileSize`, `Title`, `Private`, `FileOrientation`, `PhotosorterUID`: 7 of roughly 23 mapped fields. So 16 mapped fields have NO test proving they survive. Record this as a standing risk with the exact list of untested fields.
3. **IPTC/XMP columns.** Kukátko's `photos` has `subject`, `artist`, `copyright`, `license`, `keywords`, `software`, `scan`, `color_profile`, `image_codec`, `camera_serial`, `original_name`, `projection` (migration `0027`). `ppimport` fills these from the PhotoPrism detail endpoint. Does `psimport` fill them at all, or does a photo-sorter photo arrive with all of them empty? If photo-sorter holds the data and the migration drops it, that is a GAP.
4. **Video columns.** `media_type`, `duration_ms`, `video_codec`, `audio_codec`, `has_audio`, `fps` (migration `0004`). Does `psimport` populate them?
5. **`Marker.Invalid` and `Marker.Reviewed`** — `psimport` is believed to preserve them while `ppimport` does not. Confirm, and record the asymmetry.
6. **`Subject` attributes** — `psimport` maps `Name`/`Type`/`Favorite`/`Private`/`Notes`. Note that Kukátko's favourites are per-user; check whether `Subject.Favorite` is genuinely a subject-level column here or the same per-user concern that made `Photo.Favorite` a waiver.
7. **`Face.photo_width`/`photo_height`/`orientation`** — bbox geometry depends on them. Confirm `convertFace` carries whatever the normalised bbox needs, and that a re-orientation cannot silently move boxes.
8. Classify: `AlbumPhoto.SortOrder`, `Album.Slug`, `Label.Slug`, `Subject.Slug`, `ps.UpdatedAt`, `faces_processed.face_count`, `PhotoLabel.source`/`uncertainty`.
