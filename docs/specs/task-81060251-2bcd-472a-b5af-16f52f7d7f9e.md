# M6 — Robustness hardening

A focused pass closing the remaining robustness gaps identified versus photo-sorter.

## Context
Read `docs/ARCHITECTURE.md` §15 (what we do differently), §11. This task verifies and fixes
cross-cutting robustness items rather than adding features.

## Requirements
- **Streaming downloads/edited exports**: ensure original and edited-image downloads stream and do
  not pin the whole decoded+encoded image in RAM (photo-sorter bug). If non-destructive edits
  (crop/rotate/brightness/contrast) are applied on download, apply them in a memory-bounded way.
- **FK cascade audit**: verify all child tables (embeddings, faces, markers, album_photos,
  photo_labels, photo_phashes, photo_edits, user_favorites) have correct `ON DELETE CASCADE`/
  `SET NULL`; add a migration to fix any gaps. Verify trash purge / hard-delete removes all
  dependent rows with no dangling references.
- **Rate-limit review**: confirm login rate limiting is effective; add sensible limits/concurrency
  caps to resource-intensive endpoints (upload, bulk, import triggers, map tile/geocode proxy).
- **Upload race**: confirm concurrent identical-SHA256 uploads return a clean duplicate (409-style)
  result rather than a 500.
- **Job recovery**: confirm in-flight jobs interrupted by restart are recovered (stale-lock
  requeue) and not lost.

## Quality gate (mandatory)
- Use the **golang-developer** skill. `make check` MUST pass.
- Integration tests (test DB): hard-delete of a photo removes all dependent rows (no orphans);
  download path is memory-bounded (assert streaming, e.g. via a large fake file without buffering);
  concurrent identical upload → duplicate not 500; interrupted job is requeued and completes;
  rate limits trigger on abusive request patterns.
