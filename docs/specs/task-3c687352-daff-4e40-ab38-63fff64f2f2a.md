# Library Filters

Photos are found primarily by year, by album, and by label. Fulltext is a special-case feature and stays on `/search`. The library page has no such facets today.

## Requirements

- The library gains a filter bar with three facets: Year, Album, and Label.
- Year is a list of the years actually present in the catalog, each with its photo count. Selecting one narrows the grid.
- Album and Label are searchable selects (type to filter), because both collections will grow.
- Facets combine with each other and with the filters the library already has.
- A quick inline filter input narrows the grid by title and description. Next to it, a clearly-labelled link leads to `/search` for real fulltext and semantic search. Do not duplicate the search modes here.
- All filter state lives in URL query params with History API, so the back button restores the exact view. This is a project rule.
- An empty result renders an empty state that names the active filters and offers to clear them.

## Backend

Counting distinct years client-side would mean downloading the whole catalog. If no aggregation endpoint exists, add `GET /api/v1/photos/years` returning each year with its photo count, respecting the caller's visibility (archived, private) exactly as the photo list does. Document it in `docs/API.md` and the owning package in `docs/PACKAGES.md`.

## Verification

`make check` must pass. Add an integration test for the years endpoint (real test DB) and a unit test asserting that selecting a facet writes it to the query string.
