# /expand — grow an album or a label with similar photos

The backend can now find photos that look like the ones already in an album or on a label
(`GET /api/v1/albums/{uid}/similar`, `GET /api/v1/labels/{uid}/similar`). Build the page that turns that into
photos actually added.

**Confirm those endpoints exist before starting.** If they do not, the backend task has not run yet.

**The good news:** unlike photo-sorter, Kukátko already has everything needed for the "add them" half —
`useSelection`, `SelectionBar`, `BulkEditControl`/`BulkEditModal` and `POST /photos/bulk` with
`add_to_albums` / `add_labels`. **Reuse them. Do not build a bespoke bulk bar for this page.**

## The page

New route **`/expand`** (editor-only), linked from the nav near albums/labels.

**Config panel:**
- Source type toggle: **Album | Štítek**.
- A source picker for the chosen type — sorted by **photo count descending** (the collections worth expanding
  are the ones that already have material), each option showing its photo count. Collections with zero photos
  are not offered.
- **Threshold slider in percent** — the UI never shows a cosine distance. Range 20–80 %, step 5, **default
  70 %** (converted to distance for the API). Bookend it **"Více výsledků" ↔ "Lepší shody"**.
- Limit — number input, 1–200, default 50.
- An explicit Search button (this is an expensive query; do not fire it on every slider drag).

**Results:**
- A summary row: source photos / how many of them have embeddings / **`min_match_count`** / results found —
  with the two explanatory lines that make the ranking legible: *"Fotka musí odpovídat alespoň {{n}} zdrojovým
  fotkám"* and *"řazeno podle počtu shod, pak podle podobnosti"*. Exposing the vote rule is what stops the page
  feeling like a black box.
- A photo grid (reuse the existing library grid/tile components and their virtualization — do not hand-roll a
  grid). Each tile shows the **similarity percentage** and, when > 1, a badge with the **number of source photos
  that matched it**.
- Clicking a tile opens the photo (the existing lightbox/detail behaviour), so a doubtful candidate can be
  inspected.

**Acting on the results:**
- The existing **selection model**: click to toggle, **Shift+click for a contiguous range**, Ctrl/Cmd+A select
  all, Esc to clear — whatever the library grid already does, this page does identically. If the library's
  selection hook is missing shift-range selection, add it there so every grid benefits, rather than forking it
  here.
- With a selection active, the existing bulk bar offers **"Add to this album" / "Add this label"** — pre-filled
  with the collection being expanded, since that is the action in 99 % of cases. It stays a normal bulk edit, so
  everything else the bulk modal can do is still available.
- After a successful add the added photos **leave the result grid** (they are now members) and the summary
  counts update. No full refetch, no scroll jump.
- **Rejecting**: a per-tile ✗ persists a rejection for that label (`POST /api/v1/feedback/label-rejections`), so
  the photo is never offered for this collection again. This is what makes repeated passes converge instead of
  re-offering the same wrong photos forever. For albums, offer ✗ only if the backend models album rejections;
  otherwise omit it rather than faking it.

## States

- Nothing searched yet: an explanatory empty state.
- The collection has photos but **none have embeddings**: say exactly that, and that embeddings are computed when
  the box is online — this is the common confusing case and a generic "no results" would send the user hunting a
  bug that is not there.
- Zero results: suggest lowering the threshold.

## i18n

Every string into **both** `web/src/i18n/locales/cs/common.json` and `.../en/common.json` (namespace `common`,
Czech is the default).

## Verification

- Vitest: percent → distance conversion; the source picker excludes empty collections and sorts by photo count;
  results render with similarity and match-count badges; selection + bulk add calls `POST /photos/bulk` with the
  right operation and the added photos leave the grid; ✗ calls the rejection endpoint and removes the tile; the
  "no embeddings" state renders its own message distinct from "no results".
- `make check` must pass.
- Document: `docs/FRONTEND.md` (the page, components, hooks) and `README.md` — user-visible feature.

## Out of scope

- Changing the album/label detail pages (a "expand this album" link from them would be nice, but it is not this
  task).
- Any new bulk-edit backend operation — `POST /photos/bulk` already does what is needed.
