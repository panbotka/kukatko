# Outliers — turn the existing detection into a usable review tool

Kukátko already detects outliers: `GET /api/v1/subjects/{uid}/outliers` (`internal/outliers`) ranks a person's
assigned faces by cosine distance from their centroid, most suspicious first, and the `Outliers` component on
the subject page lists them with a one-tap unassign. The detection is fine. **The review experience is not** —
there is no threshold, no way to see enough of the photo to judge, no bulk action, no keyboard, and it is buried
inside one tab of one page.

Fix the review side, and fix one real flaw in the algorithm.

## Algorithm fix

The centroid is currently computed over **all** of the subject's faces — **including the outliers themselves**.
A person with three badly misassigned faces has those three dragging the centroid toward them, which masks
exactly what you are trying to find.

Make the centroid **robust**: compute it, drop the faces furthest from it (a trimmed mean — e.g. discard the
top decile by distance, with a floor so a person with 4 faces does not lose half of them), recompute, and score
against the trimmed centroid. Keep it simple and deterministic; do not introduce a clustering step. Cover it
with a test that plants two obvious outliers and asserts they score **higher** with trimming than without.

Keep `MinMeaningful = 3` (below that the result is flagged `meaningful: false`) and keep the endpoint's existing
shape — this is a scoring change, not an API redesign.

Add a **threshold** parameter (minimum distance from the centroid; 0 = return everything) and a **limit**, so
the endpoint can serve the page without shipping every face of a well-tagged person. Both as query params with
sane defaults; the existing "everything, ranked" behaviour stays the default.

## The page

New route **`/outliers`** (editor-only), linked from the people section of the nav. The existing panel on the
subject page **stays** — it is the right entry point when you are already looking at a person. The new page is
the entry point when you want to sweep.

- **Person picker** (typeahead over subjects with a face count) — plus, if arriving from a subject page link,
  the person pre-selected via a URL query param. View state lives in the URL, per the project's "back always
  works" rule.
- **Threshold slider in percent**, bookended **"Zobrazit vše" ↔ "Pouze extrémní"**, default = show all. The UI
  speaks percent; convert for the API.
- Stats: total faces / average distance / how many are shown / **how many faces have no embedding** (a face
  detected while the sidecar was offline cannot be checked — say so explicitly instead of silently omitting it),
  plus the one-line explanation "seřazeno podle vzdálenosti od centroidu, nejpodezřelejší první".
- Grid of **large thumbnails with the face drawn as a rectangle — with roughly 30 % padding around the box**.
  This detail matters more than it sounds: a tight crop of a face you are asked to judge is unjudgeable; you need
  the surrounding photo to recognise the person. Reuse the existing bbox geometry helpers
  (`web/src/lib/faceGeometry.ts`).
- Each card shows the distance as a percentage and a clear "this looks wrong" affordance.

## Acting on the results

- **Per card: ✓ = "yes, this is wrong" → unassign** (through the existing assign state machine,
  `POST /photos/{uid}/faces/assign` with `unassign_person`). **✗ = "no, this really is them" → persist that**
  via `POST /api/v1/feedback/face-rejections`… — careful: that endpoint records "this face is NOT this person",
  which is the **opposite** meaning. Do not misuse it. If the feedback package has no way to record a
  *confirmation*, the honest options are (a) simply dismiss the card locally for the session, or (b) add a
  confirmation to `internal/feedback` so a face confirmed as correct is not offered as an outlier again. **(b) is
  the right answer** — an outlier list that re-offers the same false alarms every time is the exact problem this
  whole batch of work is fixing. Implement it, and exclude confirmed faces from future outlier results.
- **Bulk selection** — reuse the existing selection hooks, with shift-range and Ctrl/Cmd+A, and a bulk
  "unassign selected" action. Photo-sorter has no bulk here and it is its weakest page.
- **Keyboard** — the same vocabulary as the other review pages: arrows / `j k h l` move, `y`/`Enter` unassign,
  `n` confirm-as-correct, focus advances. Register in the `?` help overlay.
- Confirming is **optimistic and in place** — the card flips state where it stands, nothing reloads, no scroll
  jump.

## i18n

Every string into **both** `web/src/i18n/locales/cs/common.json` and `.../en/common.json`.

## Verification

- Backend: a test that plants two obvious outliers among a person's faces and asserts the trimmed centroid ranks
  them above where the untrimmed centroid would; the threshold and limit params filter correctly; a person with
  2 faces still returns `meaningful: false` and does not error; a face confirmed as correct is excluded from the
  next call.
- Vitest: the bbox renders with padding; ✓ unassigns via the assign endpoint; ✗ records the confirmation; bulk
  unassign walks the selection and reports a partial failure honestly; keyboard shortcuts work and advance focus.
- `make check` must pass.
- Document: `docs/API.md` (new params, any new feedback endpoint), `docs/PACKAGES.md` (`internal/outliers`,
  `internal/feedback`), `docs/FRONTEND.md` (the page), `README.md`.

## Out of scope

- Removing the outliers panel from the subject page — it stays.
- Deleting or archiving photos. An outlier review only ever unassigns a person from a face.
