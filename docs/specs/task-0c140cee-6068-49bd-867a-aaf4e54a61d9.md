# Places — Data Enrichment

Reverse-geocode geotagged photos into a place hierarchy (country / region / city / place name) and cache it on the photo so the library can be browsed and filtered by location. Use the existing server-side mapy.com client (`internal/mapy`, method `ReverseGeocode`) which returns a `regional_structure`. Geocoding is rate-limited, so it runs as a background queue job, not inline. This task is the data layer + job only (the browse API and UI are separate tasks).

## Requirements
- New migration under `internal/database/migrations/` using the next free number. Add cached place fields for photos: `country`, `region` (state/kraj), `city` (municipality), `place_name` (most specific label), and `geocoded_at TIMESTAMPTZ`. Decide between columns on `photos` vs a `photo_places` side table keyed by `photo_uid` (FK `ON DELETE CASCADE`); document the choice in CLAUDE.md. Index the fields used for grouping/filtering (`country`, `city`).
- Add a new job type `places` to the `jobs` allowed types (CHECK constraint / type list) and a handler package `internal/placesjob`: payload `{photo_uid}` → load the photo; if it has GPS, call `mapy.ReverseGeocode(lat,lng)`, parse `regional_structure` into country/region/city/place_name, persist them and stamp `geocoded_at`. Idempotent: skip if already geocoded and coordinates unchanged. mapy.com unavailable / rate-limited → return `worker.RetryAfter(...)` so the attempt is not burned (mirror the embedding job's offline handling). Photos without GPS are recorded as processed (so they are not retried forever).
- `BackfillPlaces(ctx)` enqueues a `places` job for every geotagged photo missing place data (dedup no-op via the queue), returning the enqueued count. Wire it into the admin `internal/processapi` as `POST /process/places` → `{enqueued}`.
- Register the `places` handler on the background worker in `serve`, building the mapy client dependency only when `maps.mapy_api_key` is configured; otherwise the job and `/process/places` are gracefully unavailable.
- Respect mapy.com rate limits: the job must not exceed the geocode credit budget — processing slowly is acceptable. Reuse the existing geocode rate-limiting approach where practical.
- Everything that touches mapy.com must be behind an interface and faked in tests. Tests: unit tests for parsing `regional_structure` → place fields (fake client), and an integration test for the place write + the "missing place data" query.

## Done criteria
- Update `CLAUDE.md`/`README.md` (new job type, schema choice, `POST /process/clusters`-style `/process/places`, config notes).
- `make check` must pass. Follow the project's strict lint and testing rules.
