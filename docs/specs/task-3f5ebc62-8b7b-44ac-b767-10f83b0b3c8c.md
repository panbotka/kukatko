# M6 — Backup restore / disaster recovery

Implement restoring the database and originals from an S3-compatible backup — the counterpart to
the backup task, so a backup is actually usable.

## Context
Read `docs/ARCHITECTURE.md` §14 (backup). The backup task writes a `pg_dump` (custom/compressed
format) to S3 (e.g. `db/kukatko-<timestamp>.dump`) and syncs originals. Same S3 config
(`backup.s3.*`) and `minio-go/v7`, path-style, streamed.

## Requirements
- `kukatko restore` command (and optionally an admin endpoint, guarded — destructive!) that:
  - Lists available DB dumps in the bucket and restores a chosen one via `pg_restore` (streamed
    from S3). Clear, explicit confirmation/flag because this overwrites data.
  - Restores/downloads missing originals from S3 to the local originals path (skip those already
    present by key+size).
  - After restore, optionally re-run migrations (idempotent) and trigger a maintenance/integrity
    check so thumbnails/derived state are consistent.
- Verify integrity: after restore, report counts (photos in DB vs originals on disk) and surface
  mismatches.
- Document the full restore procedure in `docs/` (runbook: fresh machine → install → restore →
  verify), including how to point at the bucket and the exact commands.
- Never log secret keys; handle partial/interrupted restores safely (resumable downloads).

## Quality gate (mandatory)
- Use the **golang-developer** skill. `make check` MUST pass.
- Tests against a **fake S3** (httptest/stub): list dumps, streamed `pg_restore` command
  construction (skip actual restore if `pg_restore` absent in unit suite), originals download with
  skip-existing, post-restore count/mismatch report. Integration test (test DB) for the
  verify-counts logic.
