# Places — Browse API

Expose the reverse-geocoded place hierarchy for browsing, and let the photo list be scoped to a place. Photos already carry cached place fields (country / region / city / place_name) populated by the `places` enrichment job. This task is backend API only.

## Requirements
- `GET /api/v1/places` (RequireAuth) → a hierarchy with counts aggregated over non-archived photos that have place data, e.g. `{places:[{country, count, cities:[{city, count}]}]}`. Support optional `?country=` to drill into one country's cities only. Sort by count desc, then name. Photos with no place data are excluded.
- Extend `photos.ListParams` and `photoapi.parseListParams` with `country` and `city` filters (exact match on the cached fields), so the shared `GET /photos?country=&city=` returns that place's photos honoring all other filters/sort/paging — exactly the pattern used by `?album=`/`?label=` scoping. Filters go through `buildWhere` so `Count` is correct.
- Add a photos store method that computes the aggregation efficiently (GROUP BY country, then city, with counts) behind the store interface.
- Mount the new endpoint via a `server.WithAPI` builder in `cmd/kukatko/` (new `internal/placesapi` package, or extend an existing read API — keep it cohesive).
- Integration tests: the aggregation counts (with a `?country=` drill-down) and the `country`/`city` photo-list filter, including archived exclusion.

## Done criteria
- Update `CLAUDE.md`/`README.md` (Places API + the new photo list filters).
- `make check` must pass. Follow the project's strict lint and testing rules.
