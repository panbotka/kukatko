# Album and label badges on the photo detail page

On the photo detail page (`web/src/pages/PhotoDetailPage.tsx`, route `/photos/:uid`), the albums and
labels a photo belongs to are only visible far down the page, inside the "Organize" card in the
lower-left column. A user looking at the photo has no idea what it is filed under without scrolling.

Surface them as a compact, **read-only** strip of pill badges directly between the page title row and
the photo itself. This is purely informative — all editing stays exactly where it is today.

## Requirements

- Render a badge strip in `PhotoDetailPage` **between the header row** (back link, `<h1>` title,
  rating / flag / favorite controls) **and the media row** (the photo + the faces column).
- The strip lists, in this order: every album the photo belongs to, then every label on the photo.
- Each badge uses the existing pill-badge convention already used by `OrganizePanel`
  (`web/src/components/photo/OrganizePanel.tsx`): a `badge rounded-pill` with the shared colour class
  and leading glyph from `web/src/components/entityStyle.ts` (`ENTITY_STYLE.album` — violet,
  `ENTITY_STYLE.tag` — teal) plus the `Icon` component. Do not invent a new visual style, and do not
  add a new design-system component — reuse the existing pattern so the badges look identical to the
  ones in the Organize panel.
- Each badge is a link: album badges navigate to `/albums/:uid`, label badges to `/labels/:uid` —
  the same targets the Organize panel badges already use.
- The badges carry **no controls**: no remove (`CloseButton`), no add field, no create. Adding and
  removing albums/labels keeps happening **only** in `OrganizePanel`, whose UI and behaviour must not
  change at all.
- The strip must reflect the current state: adding or removing an album/label in the Organize panel
  updates the badges immediately, without a page reload. Read from the same state the Organize panel
  already uses — do not add a second fetch of the same data.
- Badges wrap onto multiple lines on narrow viewports; the strip must not force horizontal page
  scrolling.
- No empty state: when the photo has no albums and no labels, render nothing (no heading, no
  placeholder text, no empty gap).

## Edge cases

- Photo still loading / not found: no badge strip (the existing loading and error paths are unchanged).
- A viewer (read-only role) sees exactly the same badge strip as an editor — it is informative for
  everyone.
- Very many albums/labels: the strip simply wraps; do not truncate or add a "show more".

## Verification

- Extend `web/src/pages/PhotoDetailPage.test.tsx` (Vitest + Testing Library): badges render with the
  photo's albums and labels, they link to the right detail routes, they carry no remove button, and
  nothing is rendered when the photo has neither albums nor labels.
- Any new UI string goes into **both** `web/src/i18n/locales/cs/common.json` and
  `web/src/i18n/locales/en/common.json` (namespace `common`, Czech is the default). Prefer reusing the
  existing `photo.organize.*` keys over inventing new ones; a new key is only needed if the strip
  requires an accessible label of its own.
- `make check` must pass (docs-budget + fmt + lint + typecheck + tests + frontend ESLint/Prettier/Vitest).
- Update `docs/FRONTEND.md` with the change to `PhotoDetailPage` (and any new component), per the
  project's Definition of Done.

## Out of scope

- People/face badges — only albums and labels go into this strip.
- Any change to `OrganizePanel`, `PeoplePanel`, the faces panel, or the faces visibility behaviour.
