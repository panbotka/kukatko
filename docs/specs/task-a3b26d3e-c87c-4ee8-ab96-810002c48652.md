# Bug: the people-detail photo viewer pages the wrong set

On the People/Subject detail page (`/people/{uid}`), clicking a photo opens the full-screen viewer (`/photos/{uid}`, `web/src/pages/PhotoDetailPage.tsx`) and paging left/right walks the **entire library** instead of this subject's photos. Read `docs/FRONTEND.md` (photo detail / neighbors, and the "Back always works" URL-state convention) first.

## Root cause

The subject gallery tile (`web/src/components/people/SubjectPhotoTile.tsx`) links to a bare `/photos/{uid}` with **no scope query string**. The viewer's prev/next hook (`web/src/hooks/usePhotoNeighbors.ts`) therefore falls back to the default library view and pages `GET /photos` (whole library, newest-first). Correctly-scoped galleries (e.g. `web/src/pages/AlbumDetailPage.tsx`) build a `detailQuery` (via `detailQueryString`) and pass it to their tiles, which append it to the link (see `web/src/components/library/PhotoTile.tsx`), so the viewer pages the scoped set (e.g. `GET /photos?album=<uid>`).

## Requirements

- Scope the people-detail viewer so opening a photo from this page and paging left/right navigates **only this subject's photos, in the same order shown in the subject gallery**.
- The library already has a `person` facet (`person` query param in `web/src/lib/libraryView.ts`, honored by `buildPhotoQuery` in `web/src/services/photos.ts`, so `GET /photos?person=<uid>` works). Prefer reusing it: have `SubjectPage` build a `detailQuery` scoping to `person=<subjectUid>` and thread it to `SubjectPhotoTile`, which appends it to the `/photos/{uid}` link (mirroring `PhotoTile`).
- **Verify the set/order actually match.** The subject gallery is loaded from `GET /subjects/{uid}/photos` (hook `useSubjectPhotos`, no sort params), whereas neighbors would page `GET /photos?person=<uid>&sort=newest`. Confirm that `GET /photos?person=<uid>` returns the **same membership and order** as `GET /subjects/{uid}/photos`. If they match, the query-scope fix is sufficient. If they do NOT match (different default sort or membership rules), make them consistent — either align the subject-photos endpoint/params so the two agree, or teach the viewer/neighbors a subject-scoped paging mode that pages via the subject-photos source. Do NOT ship a fix that silently pages a different order than the gallery shows.
- Keep the viewer behaviour for all other galleries unchanged. Preserve the "Back always works" URL state (the scope must live in the viewer URL's query params so back/refresh keep the context).

## Tests

- Frontend: a subject tile link includes the subject scope (e.g. `?person=<uid>`); opening the viewer with that scope pages neighbors within the subject set (mock the neighbors call and assert it queries the scoped list, not the bare library).
- If a backend change is needed to make `GET /photos?person=<uid>` match `GET /subjects/{uid}/photos`, add an integration test asserting the two return the same photo set/order for a subject.
