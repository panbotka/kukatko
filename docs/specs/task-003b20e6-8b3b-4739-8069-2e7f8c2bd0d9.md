# Scoped import brings a photo's whole context

When one album is imported, a photo that also lives in three other albums currently arrives with only the album that was asked for. Migrating a slice must transfer each photo **complete**: every album it belongs to, and every label it carries, are created in Kukátko and attached — even when they were not the thing being imported.

## Do NOT import into the live catalogue

The user runs the real migration **by hand, together with the assistant** — no task imports production data. Write the code and test it; if you want a live check, point it at the **test database** (`KUKATKO_TEST_DATABASE_URL_HOST`, DB `kukatko_test`, safe to truncate) with a scratch storage dir, never at the live catalogue (which currently holds 247 photos, 17 of them from PhotoPrism). Reading from the production PhotoPrism is fine — it is read-only by design.

## Requirements

- For every photo a **scoped** run imports or updates (`--album`/`--label`/`--person`/`--year`, all four now exist — see commits 14ee9c7 and ccddda8), Kukátko resolves that photo's full context from the source and maps it:
  - **all albums** the photo belongs to (any album type), found-or-created by title, membership attached;
  - **all labels** the photo carries, found-or-created by name, attached with their source/uncertainty.
- People already arrive with the photo (`Files[].Markers[]` in the listing payload) — leave that path alone, but make sure a scoped run still seeds subjects from it.
- Idempotent: re-running a scoped import creates no duplicate album, label or membership row.
- The **full** import keeps its current catalogue-walk (`mapAlbums`/`mapLabels`) — do not make it fetch a detail per photo, it would turn one listing into 20k requests.
- Tests: a photo in three albums, imported via a scope naming only one, ends up in all three; a re-run changes nothing.

## Verified against the production instance

The photo **listing** payload carries neither `Albums` nor `Labels` — I checked. The **detail** endpoint does:

```
GET /api/v1/photos/{uid}  ->  { …, "Albums": [...], "Labels": [...], "Details": {...} }
```

So a scoped run needs one extra request per imported photo. That is the right trade for a slice (17 photos → 17 requests); it is exactly why the full run must not do it (20,670 photos in the source).

`internal/photoprism` has no get-photo-by-uid call yet — add one (`GetPhoto(ctx, uid)`), with the album and label relations typed, and fake it in the ppimport tests.

## Implementation Notes

- `internal/ppimport/organize.go` already has `mapOneAlbum`/`mapOneLabel` (find-or-create by title/name) and `attachAlbumMembers`/`attachLabelMembers`. Reuse them; what is new is the per-photo direction: photo → its albums/labels, instead of album → its photos.
- Album types matter: the source serves five (`photoprism.AlbumTypes`) and `/api/v1/albums` takes exactly one per request. A photo's detail names its albums directly, so the scoped path does not need to page them by type.
- Credentials for reading production are already in `.secrets/db.env` (`KUKATKO_IMPORT_PHOTOPRISM_BASE_URL` / `_TOKEN`). Never commit them.
- Docs: `docs/PACKAGES.md` (ppimport + photoprism client), `docs/API.md` only if an endpoint changes.
