# M4 — Maps backend (mapy.com proxy + GeoJSON)

Add a backend proxy for mapy.com map tiles and reverse geocoding (so the API key never reaches
the browser) and a GeoJSON endpoint of geotagged photos for the map view.

## Context
Read `docs/ARCHITECTURE.md` §12 (Maps). The mapy.com API key is in config as `maps.mapy_api_key`
(env `MAPY_API_KEY`). Tile URL: `https://api.mapy.com/v1/maptiles/{mapset}/256/{z}/{x}/{y}`;
reverse geocode: `https://api.mapy.com/v1/rgeocode?lon=&lat=&lang=cs`. Key sent via
`X-Mapy-Api-Key` header from the server. Free tier: 250k credits/mo, 1 tile=1 credit,
geocode=4 credits, rate 500 tiles/s.

## Requirements
- **Tile proxy**: `GET /api/v1/map/tiles/{mapset}/{z}/{x}/{y}` (auth) → fetch from mapy.com with
  the API key in the header, stream the tile back with appropriate cache headers. Restrict
  `mapset` to an allowlist (basic|outdoor|aerial|winter). Support retina (`@2x`) where mapy.com
  allows. Never expose the key.
- **Reverse geocode proxy**: `GET /api/v1/map/rgeocode?lat=&lng=` (auth) → call mapy.com rgeocode,
  return a simplified `{location, name, regionalStructure}`. Rate-limit/cache to conserve credits
  (geocode is 4 credits); call only on demand.
- **GeoJSON endpoint**: `GET /api/v1/map/photos` (auth) → GeoJSON FeatureCollection of photos
  with lat/lng (respecting the standard filters: date range, album/label scope, archived), each
  feature carrying photo uid + thumb info for the map markers/clustering.
- Handle mapy.com errors (401/403/404/429) and surface them sanely; do not leak the key in errors.

## Quality gate (mandatory)
- Use the **golang-developer** skill. `make check` MUST pass.
- Tests with a **fake mapy.com server** (httptest): tile proxy forwards with key header + mapset
  allowlist + streams bytes; rgeocode parsing + simplified response + caching; key never appears
  in client-facing responses/errors. Integration test (test DB): GeoJSON output for geotagged
  photos with filters.
