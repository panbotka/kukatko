# Fix PhotoPrism Import Gaps

`docs/MIGRATION_AUDIT.md` audits the PhotoPrism → Kukátko field mapping and marks every source field **MAPPED**, **WAIVED** (deliberately dropped, with a reason) or **GAP** (a real loss nobody decided on). Fix the GAPs on the PhotoPrism import path.

## Requirements

- Read `docs/MIGRATION_AUDIT.md`, section "PhotoPrism → Kukátko". **If that file or section does not exist, stop and report — do not guess what the gaps are.** This task's whole input is that document.
- Fix every row marked GAP whose recommended fix lands in `internal/ppimport` or `internal/photoprism`. Do NOT touch rows marked WAIVED — those are deliberate decisions, and re-litigating them is out of scope.
- A "fix" means the source value reaches the right Kukátko column on BOTH paths: the insert path (`ppimport/metadata.go` `buildPhoto`) and the incremental re-run path (`metadataUpdate`), for every column the import owns. `photos.Store.UpdateMetadata` (`internal/photos/store.go` ~line 268) overwrites its columns WHOLESALE — a field added to the insert path but not to the patch gets blanked on the next incremental run. That failure is invisible until someone re-imports.
- Anything you add to `metadataUpdate` must also be compared in `metadataUnchanged` (~line 184). That comparator must match exactly the fields `UpdateMetadata` writes; a missing comparison makes a real rewrite look like a skip and the import silently stops updating.
- The import must stay INCREMENTAL and IDEMPOTENT: re-running it must never clobber edits a user made in Kukátko. These existing tests assert it and must keep passing — `TestIntegration_detailsRerunIsNoop`, `TestIntegration_detailsNeverClobberLocalEdits`, `TestMetadataUpdate_emptyNeverClobbers`, `TestMetadataUpdate_preservesKukatkoFields`, `TestMetadataUpdate_preservesNotes`.
- If a fix needs a new column, add a migration in `internal/database/migrations` following the existing ascending numbering, and grep the WHOLE repo for other writers of that table — a new column that other packages do not fill will pass `make check` and still be wrong.
- If a GAP turns out to need a product decision rather than code (a PhotoPrism field with no sensible Kukátko home), do NOT invent a column. Leave the code alone and flip that row to WAIVED in the audit document with the reasoning.

## Scope guard

If the audit lists more gaps than fit in one session, fix them highest-impact first — silent per-run data loss beats a never-imported cosmetic field — and leave the remaining GAP rows in the document untouched so the rest stays visible. State plainly in your final report which gaps you fixed and which you left, and why.

## Verification

- Every fixed field needs a test asserting it survives BOTH a fresh import and a re-run. `psimport.TestBuildPhoto` is the cautionary example — it asserts 7 of ~23 mapped fields; do not copy that thinness into ppimport.
- Integration tests run against a real test DB — `KUKATKO_TEST_DATABASE_URL` in `.secrets/db.env` (database `kukatko_test`, safe to truncate).
- `make check` must pass.
- Update `docs/MIGRATION_AUDIT.md`: flip each fixed row from GAP to MAPPED. Update `docs/PACKAGES.md` if the import's documented behaviour changed.
