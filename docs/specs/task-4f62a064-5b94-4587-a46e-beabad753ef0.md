# Expand a collection — photos similar to an album / a label

Kukátko can find photos similar to **one** photo (`GET /photos/{uid}/similar`). It cannot find photos similar
to a **collection** — "show me photos that look like the ones already on the štítek *Ostatky*, so I can add
the ones I missed". That is the single most useful way to finish a half-tagged library.

Build the query. The page is a separate task.

## API

- `GET /api/v1/albums/{uid}/similar?threshold=&limit=` (RequireWrite)
- `GET /api/v1/labels/{uid}/similar?threshold=&limit=` (RequireWrite)

Both share one service — the only difference is how the source photo set is resolved.

**Note for whoever ports this from photo-sorter:** its version resolves album/label membership by calling
**PhotoPrism's API** — it is the last leftover PhotoPrism dependency in that feature. Kukátko owns its own
albums and labels (`internal/organize`); query them natively. Do not introduce a PhotoPrism call here.

## Algorithm

Per-photo kNN unioned with voting — **not** a mean-of-the-album vector. A collection like "Ostatky" is not one
visual concept, and averaging its embeddings produces a centroid that resembles nothing in it.

1. **Source set** — every photo in the album / carrying the label. Load their CLIP image embeddings from
   `internal/vectors`. Photos without an embedding are skipped, and the count is **reported** (the sidecar is
   often offline, so a collection can be half-embedded and the user must know that is why the results are thin).
2. **kNN per source photo** — `FindSimilar` with the caller's max distance, over-fetching so the filters below
   do not starve the result. Bounded concurrency.
3. **Union with voting** — merge candidates by photo. `match_count` = how many source photos returned it;
   `distance` = the **minimum** across them.
4. **Exclude photos already in the collection.** This is the "already a member" filter and it is the whole
   point — a result list full of photos already on the label is worthless.
5. **The vote rule** — a candidate must be returned by at least `min_match_count` source photos, scaled with
   the source-set size and the threshold, clamped to **1..5**. **Return `min_match_count` in the response** —
   the UI shows it. Without the vote rule this endpoint is a firehose.
6. **Rejections** — drop candidates the user has already rejected for this label (`internal/feedback`), and
   apply the **negative-exemplar rule**: a photo closer to a photo rejected for this label than to any photo
   carrying it is dropped. (Albums: same mechanism if album rejections exist; if the feedback package only
   models label rejections, apply it for labels and note the asymmetry rather than inventing a new table here.)
7. **Sort by `match_count` DESC, then distance ASC** — a photo that five source photos agree on beats a photo
   that one source photo agrees on very strongly. Truncate to the limit.

Response per candidate: the photo (in the shape the grid already consumes, with `thumb_url` stamped via
`internal/mediaurl`), the distance, the similarity (`1 - distance`), and the `match_count`. Plus a summary:
source photo count, source photos **with** embeddings, `min_match_count`, threshold, result count.

## Thresholds and config

New config keys (add to the `Config` struct, `setDefaults`, `config.example.yaml`, the tests **and**
`docs/OPERATIONS.md` at once). Suggested defaults from what works in photo-sorter: max distance **0.30**
(the UI shows this as 70 % similarity), limit **50**, cap the limit at something sane (200).

## Edge cases

- An **empty** album/label, or one where no photo has an embedding: return an empty result with the counts
  filled in, not an error. The UI must be able to explain *why* it is empty.
- A collection of **one** photo: works, and degenerates to the existing per-photo similarity — that is fine.
- A huge album (thousands of photos): do not run thousands of kNN queries. Cap the number of source photos
  used as query vectors (sample them — deterministically, e.g. the newest N, or spread across the album) and
  **report the cap in the response** so the truncation is visible rather than silent.

## Verification

- Integration tests with seeded embeddings: candidates that several source photos agree on rank above a single
  strong match; photos already in the collection never appear; a rejected photo never appears; the
  negative-exemplar rule drops what it should; an empty collection returns an empty, non-error result with the
  right counts; the source-set cap is applied and reported.
- `make check` must pass.
- Document: `docs/API.md` (both endpoints), `docs/PACKAGES.md`, `docs/OPERATIONS.md` (config keys), and a line
  in the `CLAUDE.md` package map if a new package appears.

## Out of scope

- Any frontend (separate task).
- Writing anything — this endpoint is read-only. Adding the found photos to the album/label goes through the
  existing bulk endpoint (`POST /photos/bulk`).
