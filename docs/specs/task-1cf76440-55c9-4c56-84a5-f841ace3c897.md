# Photo detail — surface the metadata that is already there

`web/src/components/photo/TechnicalDetails.tsx` (the collapsible read-only card on the photo detail page)
shows only camera, lens, aperture, exposure, focal length, ISO, file name, dimensions and uploader.
Kukátko **already stores and already ships to the client** a lot more than that — it is simply rendered
nowhere. Compared against PhotoPrism's detail dialog, this is most of what feels "missing".

Expand the card so everything the app knows about a photo is visible.

## Requirements

Render these in the card. Group them under sub-headings so the card stays readable —
**Fotografie / Photo**, **Soubor / File**, **Poloha / Location**, **Video** (video only), **Původ / Origin**.

**Already in the photo payload today — just render them:**

- `altitude` — nadmořská výška, in metres, only when non-null
- `file_size` — human-readable (kB / MB), with the exact byte count in a `title` tooltip
- `file_mime` — rendered as a short format label (e.g. `image/jpeg` → JPEG, `image/heic` → HEIC)
- `file_orientation` — the raw EXIF value 1–8 rendered as a human label (e.g. "Normální", "Otočeno 90° vpravo")
- **aspect ratio** — derived from `file_width`/`file_height`, rendered like `4 : 3` (reduce the fraction;
  fall back to a decimal like `1.50 : 1` when it does not reduce cleanly)
- **resolution in megapixels** — derived from `file_width × file_height`, one decimal (e.g. "12,2 Mpx")
- `taken_at_source` — where the capture date came from, as a human label (EXIF / z názvu souboru /
  ručně / neznámý)
- `file_hash` — SHA256, truncated with the full value in a `title` tooltip and a copy-to-clipboard action
- `created_at` / `updated_at` — added to the library / last modified
- `private` — a badge when the photo is private
- Video only (`media_type` is `video` or `live`): `duration_ms` as `m:ss`, `video_codec`, `audio_codec`,
  `has_audio`, `fps`

**Location — needs a small backend addition:** the photo detail payload does **not** currently include the
reverse-geocoded place. `photo_places` (country / region / city / place_name) is populated by the `places`
job and today only feeds library filters via `placesapi`. Include it in the photo detail response
(`internal/photoapi`) and render country / city / place name here, alongside the coordinates. This is
cached data — do **not** trigger an on-demand geocode from the detail page (that is what `PhotoLocation`
already does separately, and the credits are metered).

**New columns from the metadata task** — render them if they exist on the payload; if the columns are not
there yet, skip them cleanly rather than blocking on that task: `subject`, `artist`, `copyright`,
`license`, `keywords` (rendered as separate chips split on the comma), `software`, `scan` (a badge —
"Sken"), `color_profile`, `image_codec`, `camera_serial`, `original_name` (only when it differs from
`file_name`), `projection`.

**Origin:** `photoprism_uid` / `photosorter_uid` when set — shown as plain provenance text, so it is
obvious where an imported photo came from.

## UI rules

- Every field is **read-only** here. Editing happens in `MetadataPanel`, and this task changes nothing
  about it.
- **A field with no value is not rendered at all** — no "—", no empty rows. A photo with sparse metadata
  must not turn into a wall of blanks.
- The card keeps its existing collapsible chevron pattern and stays **collapsed by default**.
- The existing "regenerate thumbnail" action for editors stays where it is.
- Keep the layout scannable — a two-column definition-list style (label / value) that collapses to one
  column on narrow viewports. Long values (hash, colour profile, paths) must wrap or truncate, never
  force horizontal page scrolling.
- Numbers and dates use the existing i18n formatting conventions of the app (Czech is the default
  locale — decimal comma).

## i18n

Every new label goes into **both** `web/src/i18n/locales/cs/common.json` and `.../en/common.json`
(namespace `common`). Czech first — it is the default.

## Verification

- Vitest: a photo with a full payload renders each group and each field; a photo with a sparse payload
  renders no empty rows; a still photo shows no video group; aspect ratio and megapixels are computed
  correctly (cover a 4:3, a 16:9 and an odd ratio); the card is collapsed on first render.
- Backend: integration test that the photo detail response carries the cached place when `photo_places`
  has a row, and omits it cleanly when it does not — and that no geocode call is made.
- `make check` must pass.
- Update `docs/FRONTEND.md` (TechnicalDetails) and `docs/API.md` (the place block added to the photo
  detail payload).

## Out of scope

- Editing any of these fields.
- Triggering reverse geocoding from the detail page.
- The raw `exif` JSONB blob — it is already shipped to the client and rendering it whole is a separate
  question; do not dump it here.
