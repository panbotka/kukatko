# Pick a location by name, not by coordinates

Setting a photo's location today means typing coordinates or clicking a point on a map. For a scanned photo you
know was taken in *Veselí nad Moravou*, that is the wrong interaction: you know the name, not the numbers, and
finding the right spot on a map by panning is tedious.

Kukátko already talks to mapy.com server-side (`internal/mapy`, the tile proxy and reverse geocoding, with the API
key never leaving the server). Add **forward** geocoding — name → coordinates — and wire it into the location
editor.

## Backend

- `GET /api/v1/map/geocode?q=<query>&limit=N` (RequireAuth) — forward geocoding through `internal/mapy`.
  Returns ranked suggestions: the display name, the type (city / street / POI / region), the country/region for
  disambiguation, and the coordinates.
- **The API key stays server-side.** Same as the tile proxy and `rgeocode` — this is a hard project rule.
- **Cache and rate-limit it exactly as `rgeocode` already is.** mapy.com credits are metered and a typeahead fires
  a request per keystroke — that is precisely how you burn a quota in an afternoon. Debounce on the client
  **and** cache on the server (the same query returns the same answer; cache it), **and** apply the existing
  per-key rate limiter. Do not ship a typeahead with no throttle.
- Degrade gracefully: mapy.com unreachable or out of credits → a clean error the UI can show ("vyhledávání míst
  není dostupné"), never a 500, and never a broken edit form. The rest of the location editor (coordinates, map
  click) must keep working.

## Frontend

In the location editor on the photo detail (`MetadataPanel` / `PhotoLocation`):

- A **search field**: type a place name, get a dropdown of suggestions, pick one → the coordinates fill in and the
  map recentres on the point. The existing map-click and coordinate-paste paths stay — this is a third way in, not
  a replacement.
- Debounce the input (a few hundred ms) and cancel in-flight requests on a new keystroke. Do not fire a request
  per character.
- Show enough in each suggestion to disambiguate (there are several *Veselí*s) — name plus region/country.
- Keyboard: arrows to move through suggestions, Enter to pick, Esc to close. It is a form control; make it behave
  like one.
- Bulk: **not** in scope, but note that the existing bulk edit already has `set_location` — if wiring the same
  picker into `BulkEditModal` is trivial once the component exists, do it; if it is not, leave it.

## i18n

Every string into **both** `web/src/i18n/locales/cs/common.json` and `.../en/common.json` (Czech is the default —
and this feature is overwhelmingly going to be used with Czech place names, so make sure diacritics and the
mapy.com locale behave).

## Verification

- Backend: integration test against a **fake** mapy client (the project's rule — external dependencies behind an
  interface): results are returned and shaped correctly; the cache serves a repeat query without a second upstream
  call; the rate limiter kicks in; an upstream failure produces a clean typed error, not a 500.
- Vitest: typing debounces; suggestions render and are keyboard-navigable; picking one sets the coordinates and
  moves the map; an upstream error shows the message and leaves the rest of the editor usable.
- `make check` must pass.
- Document: `docs/API.md` (the endpoint), `docs/PACKAGES.md` (`internal/mapy`, `internal/mapsapi`),
  `docs/FRONTEND.md`, `docs/OPERATIONS.md` if a config key appears (cache TTL / rate limit).

## Out of scope

- Geocoding as a batch operation over many photos.
- Any other geocoding provider.
