# Re-partition HTTP endpoint gating to the maintainer/admin split

Apply the new role model to the API surface: operational endpoints require **maintainer**, governance endpoints require **admin**, and permanently emptying/purging trash now requires **admin** (it currently only requires editor/write). This task assumes the backend role model already provides a `maintainer` role and a `RequireMaintainer` middleware (from the role-model task). If `RequireMaintainer` is missing, add it following the existing `Require*` pattern in `internal/auth/middleware.go`.

Read `docs/API.md`, `docs/SECURITY_AUDIT.md`, and the auth section of `docs/PACKAGES.md` first. Endpoint gating is wired in `cmd/kukatko/serve.go` and the per-API `cmd/kukatko/{audit,backup,restore,jobs,system,maintenance,import}.go` files, which inject `authAPI.RequireAdmin` / `RequireImport` etc. into the `*api` packages.

## Target gating

**Maintainer-only** (operations) — change these from their current admin/import gate to `RequireMaintainer`:
- Imports — `internal/importapi` (`/import/*` triggers; currently `RequireImport`). Import run history that is currently always-mounted may stay readable, but the trigger endpoints require maintainer.
- Maintenance — `internal/maintenanceapi` (`/maintenance/*`).
- System status — `internal/systemapi` (`/system`).
- Backup — `internal/backupapi` (`/backup/*`).
- Restore — `internal/restoreapi` (`/restore/*`).
- Jobs — `internal/jobsapi` (`/jobs/*`).
- Processing backfills — `internal/processapi` (`/process/*`).

**Admin** (governance) — remain `RequireAdmin` (now satisfied by admin OR maintainer via the ladder):
- User management — `internal/auth` `/admin/users/*`.
- Audit log — `internal/auditapi` (`/audit`).

**Trash purge → admin** (tightening): permanently emptying the trash / purging archived photos currently requires write (editor). Move the destructive trash operations to `RequireAdmin`. In `internal/photoapi/http.go`, `POST /trash/empty` and the per-photo permanent purge endpoint must require admin. `GET /trash/info` (read) stays `RequireAuth`. Archiving a photo (soft delete, reversible) stays a write (editor) operation — do NOT gate archiving behind admin, only the permanent deletion.

## Requirements

- Update the `cmd/kukatko` wiring so each `buildXxxAPI` for the operational surfaces above receives `authAPI.RequireMaintainer` instead of `RequireAdmin`/`RequireImport`.
- Keep audit and user-management on `RequireAdmin`.
- Gate the destructive trash endpoints in `internal/photoapi` on admin (the auth middleware is injected into `photoapi` — thread `RequireAdmin` through to those specific routes; keep the rest of photoapi unchanged).
- Do not change read endpoints, write (editor) endpoints, or the MCP mount.

## Tests

- Integration/HTTP tests asserting the new matrix on representative endpoints:
  - An **editor** is 403 on an operations endpoint (e.g. `/maintenance`, `/backup`, `/import/photoprism`, `/system`, `/jobs`) and 403 on trash purge.
  - An **admin** is 403 on operations endpoints but authorized on audit, users, and trash purge.
  - A **maintainer** is authorized on operations endpoints AND on audit/users/trash (ladder).
  - Trash: editor can archive (soft delete) but is 403 on empty/purge; admin can purge.
