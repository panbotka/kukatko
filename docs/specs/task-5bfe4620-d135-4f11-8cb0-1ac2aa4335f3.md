# Decode every common upload format so thumbnails "just work"

Users must be able to upload essentially any photo they have and get a working thumbnail and
preview — jpg, png, webp, heic, gif, bmp, tiff, and RAW files from any camera. Today ingest
already **accepts** any file (there is no allow-list: `internal/ingest` stores and catalogues
whatever is uploaded and returns 201), but only a subset can be **decoded** into a thumbnail.
Everything else is silently catalogued with no thumbnail and no perceptual hash — the upload
"succeeds" but the photo shows a broken/placeholder tile. This task closes that gap for the
common raster formats and widens RAW coverage.

## Current state (for context, verify before changing)

- Decodable today (produce a thumbnail): JPEG, PNG, WebP (pure-Go); HEIC/HEIF (shell-out
  `heif-convert`); RAW via `exiftool` embedded-preview extraction for a fixed extension list
  (`.cr2 .cr3 .nef .arw .dng .raf .orf .rw2 .pef .srw`); video via `ffmpeg` poster frame.
- NOT decodable today → no thumbnail: **BMP, GIF, TIFF**, and less common RAW extensions.
- The decode dispatch lives in `internal/imgconvert` (`extFormats` extension map, `classifyMagic`
  content sniffing, `DetectFormat`, and the `EnsureDecodable` switch). Pure-Go decoders are
  registered by blank import in **both** `internal/ingest/ingest.go` and
  `internal/thumb/resize.go`. `CGO_ENABLED=0` is mandatory — only pure-Go decoders or shell-outs
  are allowed, never a cgo image library.

## Requirements

- **BMP, GIF and TIFF decode to a thumbnail** using pure-Go decoders (`golang.org/x/image/bmp`,
  `golang.org/x/image/tiff`, and the standard library `image/gif`). Add them to the extension
  map, register the decoders in every place decoders are registered, and route them through the
  direct-decode path. An animated GIF may thumbnail from its first frame.
- **Content-based detection still wins over a wrong extension.** A BMP/GIF/TIFF whose magic bytes
  identify it must decode even if the file was misnamed, mirroring the existing behaviour where
  magic is authoritative for directly-decodable formats and a RAW extension is only trusted when
  the magic matches nothing. Do NOT let TIFF magic hijack real camera RAW files (many RAWs are
  TIFF-based containers) — RAW files must still be detected by their RAW extension and go through
  the RAW preview path, exactly as they do now.
- **Widen RAW extension coverage** to the common remaining vendors so more cameras "just work":
  at least `.3fr .iiq .x3f .kdc .mrw .srf .mef .nrw` in addition to the current set (only add
  extensions the existing `exiftool` embedded-preview path can actually serve; skip any that
  genuinely cannot yield a preview and note why).
- **The file picker must not hide valid files.** The upload drop-zone input
  (`web/src/components/upload/DropZone.tsx`) currently hints `accept="image/*,video/*"`, which on
  some browsers/phones hides RAW/HEIC because they aren't labelled `image/*`. Broaden the accept
  hint (or drop the restriction) so RAW, HEIC and the newly supported formats are selectable, and
  confirm drag-drop still accepts everything.
- **No format is rejected at ingest** — that behaviour stays. The goal is that the common formats
  now also get a thumbnail instead of only being stored.
- A format that still cannot be decoded (an exotic RAW with no embedded preview, a corrupt file)
  must continue to be catalogued without failing the upload, and must not crash the pipeline.
