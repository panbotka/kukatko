# The media URL signature contract

Originals and thumbnails live in a private Cloudflare R2 bucket, served from the edge by a Cloudflare Worker that verifies a signature on every request. The Worker is **not** built in this repository. All Cloudflare infrastructure — the bucket, the Worker's source, its bindings and its hostname — is defined in the infra repository at `/home/pi/projects/infra`, in the `cloudflare-r2/` Terraform root module, and deployed by Terraform.

This repository owns exactly one half of that arrangement: it **mints** the signed URLs, in `internal/storage/sign.go`. The Worker **verifies** them. The two implementations live in different repositories, in different languages, and nothing in either build will notice if they drift apart. A one-byte change to the signed payload silently 403s every image in the library.

This task makes that drift impossible to introduce unnoticed, and corrects the documentation, which currently promises the Worker ships from here.

## Requirements

### Freeze the contract as golden test vectors

Add `internal/storage/testdata/url_signature_vectors.json`: a machine-readable set of fixtures, each naming a secret, an object key, an expiry and the expected signature. The file is consumed by the Worker's own tests in the infra repo, so it must be self-describing and stable — treat it as a published artifact, not as scratch test data.

Cover at least:

- a plain key such as `2024/03/<64 hex chars>.jpg`
- a key with several path segments, since real object keys are `YYYY/MM/<sha256>.<ext>` and contain slashes
- an expiry of `0`, and a large far-future expiry
- a signature produced with the **previous** secret, which verification must still accept
- a case whose signature is deliberately wrong, which verification must reject

Extend `internal/storage/sign_test.go` with a test that loads the vectors and asserts the signing and verification code agrees with every one of them. The point is that anyone who changes the algorithm is forced to regenerate the golden file, which makes the change visible in review and tells them to update the infra repo in step.

### Correct the documentation

The Worker is an external deployable. Say so wherever this repo currently implies otherwise:

- `docs/superpowers/specs/2026-07-09-s3-storage-design.md` — the design doc currently describes the Worker as "a second deployable, written in JavaScript … a separate artifact to version and deploy", which reads as if it ships from this repo. Rewrite that so it is unambiguous that the Worker lives in the infra repo and is applied by Terraform there. Update the task table row for the Worker to match. Leave the rest of the design intact: it is still correct.
- `docs/ARCHITECTURE.md` §4 — the signed URL form and two-secret rotation are already documented and stay. Add where the Worker is defined, and note that `internal/storage/testdata/url_signature_vectors.json` is the contract both sides are held to.
- `docs/OPERATIONS.md` — the `storage.r2.*` reference stays. Rotating `url_signing_secret` is now a procedure that spans two repositories: the same value must be configured here and as a `secret_text` binding on the Worker in the infra repo. Document the ordering that avoids breaking URLs already handed out — move the current value into `url_signing_secret_previous`, put the new value in `url_signing_secret`, deploy both sides, and drop the old value only once every outstanding URL has expired (the TTL is one hour by default).
- `config.example.yaml` — the comments on `media_base_url` and `url_signing_secret` should name the hostname the Worker is served on, `kukatko-media.panbotka.cz`, and point at the infra repo as the place the Worker is defined.

### What this task must not do

- Do not add a `worker/` directory, a `wrangler.toml`, a `wrangler.jsonc`, or any JavaScript outside `web/`.
- Do not add a `make` target for a Worker. `make check` gains nothing here beyond the new Go test.
- Do not change the signing algorithm, the query parameter names, the URL TTL, or anything else in `internal/storage/sign.go` beyond what the new test needs. The contract is already implemented and already correct; this task documents and pins it.
- Do not change how media is currently served. The Go binary still streams bytes through `internal/photoapi`; switching the serving path to signed URLs is separate work.
