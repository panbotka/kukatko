# Fix: the mobile navigation menu does not close after tapping an item

On a phone, opening the hamburger menu and tapping a destination navigates to the
page but leaves the menu expanded, covering the content. The user must manually tap
the hamburger again to close it. The menu must close automatically on any navigation.

## Current behavior (root cause)

- `web/src/components/Layout.tsx`: the `<Navbar expand="md" ... collapseOnSelect>`
  (around line 308) relies on react-bootstrap's `collapseOnSelect`, which only
  collapses the `Navbar.Collapse` when a nav item fires a select event. In this
  navbar that event does not reliably fire:
  - The top-level links are `<Nav.Link as={NavLink} to=...>` with NO `eventKey`
    (`renderLink`, ~lines 252-267).
  - The grouped menus (Procházet / Nástroje / Provoz / Správa) are built from a raw
    react-bootstrap `<Dropdown as={NavItem}>` with `<Dropdown.Item as={NavLink}>`
    (`renderGroup`, ~lines 275-303) — deliberately NOT `NavDropdown` (to keep the
    `title` tooltip). A raw `Dropdown`'s items do not propagate a select event to the
    Navbar's collapse handler, so tapping a grouped item never collapses the menu.
  - The user menu is a `NavDropdown` with items `as={Link}` (~lines 347-377).
- Net effect: on mobile the collapse stays open after selecting most items.

## Requirements

- After the user taps ANY navigating item in the mobile menu — a top-level link, an
  item inside any group dropdown (Procházet / Nástroje / Provoz / Správa), or an item
  in the user menu (Account, Help) — the collapsed menu must close automatically.
- The fix must be robust to item type, not dependent on each item firing a
  react-bootstrap select event. The recommended approach: make the `Navbar`
  controlled — hold an `expanded` state, pass `expanded` + `onToggle`, and close it
  (`setExpanded(false)`) whenever the route changes (an effect on the `pathname` from
  the already-present `useLocation`). This guarantees the menu closes on every
  navigation regardless of which control was tapped. Tapping the same route (no
  navigation) should still close the menu — closing on tap is also acceptable.
- Do not break desktop: on `md`+ the navbar is not collapsible, so the controlled
  state must be a no-op there (nothing visibly changes).
- The hamburger toggle must still open and manually close the menu as today.
- Logout (which calls a handler, not a route change) should also dismiss the menu.

## Notes

This is the concrete bug fix for the "menu won't close" complaint; the broader
"refine the mobile hamburger menu into a drawer" task is separate polish and must not
be a prerequisite for this fix.
