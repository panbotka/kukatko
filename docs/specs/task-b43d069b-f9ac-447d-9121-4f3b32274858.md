# Photo Detail — Edit Panel Beside the Photo

On the photo detail page, opening "Úpravy" must let the user edit the ORIGINAL photo shown at the top of the page, with the edit controls appearing in a panel on the right — the same way the faces panel appears.

## Current behaviour

- `web/src/pages/PhotoDetailPage.tsx` (route `/photos/:uid`). The "Úpravy" control (i18n `photo.edit.title`, cs "Úpravy") is a collapsible react-bootstrap `Card` at the very BOTTOM of the page (~line 611), gated on `canWrite`, state `const [editOpen, setEditOpen] = useState(false)` (~line 97).
- Page order today: header → `OrganizeBadges` → photo `Row` → `StackStrip` → Organize + Caption row → Technical details card → **Úpravy card** → `SimilarPhotos`. Opening Úpravy therefore scrolls the original photo out of view — which is the bug.
- `web/src/components/photo/EditPanel.tsx` renders ITS OWN second `<img>` as the preview (`thumbUrl(uid, 'fit_1280')`, `maxHeight: '40vh'`, ~line 91-97), controls stacked underneath. The card is deliberately unmounted until opened (`{editOpen && <Card.Body>`) precisely so the page carries only ONE copy of the photo on first render. Comments at `PhotoDetailPage.tsx:611-613` and `EditPanel.tsx:95-96` say so.

## The pattern to mirror — read carefully

The faces panel does NOT slide and is NOT an Offcanvas, despite `docs/FRONTEND.md` using the word "vysune". It is a plain Bootstrap grid reflow with ZERO animation (`PhotoDetailPage.tsx` ~469-545):

- The photo column is `<Col xs={12} lg={showFaces ? 8 : 12}>` — it shrinks from 12 to 8.
- The panel simply mounts as a sibling `<Col xs={12} lg={4}>` holding `FacesPanel` — a `<Card>` whose header carries a count and an `x-lg` close button, and whose `Card.Body` has `maxHeight: '80vh', overflowY: 'auto'`.
- Below `lg` the panel stacks under the photo (`xs={12}`).
- ONE boolean drives both the overlay and the panel so they can never disagree.

Copy this pattern literally. Do NOT add a slide transition — a real animation here would be inconsistent with the faces panel sitting right next to it. (react-bootstrap `Offcanvas` exists in the codebase but only in `components/library/FilterBar.tsx`; it is not the pattern here.)

## Requirements

- Opening "Úpravy" keeps the ORIGINAL photo at the top visible and applies the live edit preview to IT. The photo column shrinks to `lg={8}`, the edit panel mounts beside it at `lg={4}`, and below `lg` it stacks underneath.
- `EditPanel` must NOT render its own `<img>` any more. The main photo becomes the preview surface — `PhotoDetailPage` already applies `editPreviewStyle(edit)` to it (~line 396). This preserves the page's "exactly ONE copy of the photo" invariant, which is the entire reason the card is unmounted-until-opened today.
- Keep every existing control: rotation (clockwise quarter turns, `rotateRight`), brightness and contrast (`Form.Range`, −1..1 step 0.05), the crop checkbox + 4 normalised crop sliders (`crop_x/crop_y/crop_w/crop_h`, `DEFAULT_CROP` = centred 80% box), and the `photo.edit.save` ("Uložit úpravy") / `photo.edit.reset` ("Obnovit originál") buttons.
- The panel gets a close button in its header, mirroring `FacesPanel` (`aria-label` via i18n, no hard-coded string).
- Live preview stays CSS-only via `web/src/lib/photoEdit.ts` `editPreviewStyle()` (`filter: brightness(1+b) contrast(1+c)`, `transform: rotate(Ndeg)`, `clip-path: inset(...)`). Saving still goes through `saveEdit` → `PUT /api/v1/photos/{uid}/edit`. **Do not change the backend.**

## Edge cases

- **Faces and edits open at once.** Both panels want the same `lg={4}` slot beside a `lg={8}` photo. Decide and implement a coherent behaviour — the simplest is that opening one closes the other. The photo must never end up with two panels fighting for one column, and must never be squeezed into a third layout.
- **Crop/rotate vs the face overlay.** `FaceOverlay` positions boxes in PERCENTAGES of the photo wrapper (`className="position-relative d-inline-flex mw-100"`, which shrink-wraps the `<img>`). The preview's `clip-path: inset(...)` and `transform: rotate(...)` change the rendered geometry, so boxes drawn over a cropped or rotated preview will not sit on the faces. Handle it deliberately — the simplest correct answer is to hide the face overlay while the edit preview is non-identity (`isIdentityEdit` already exists in `photoEdit.ts`).
- **Do not let anything stretch the photo wrapper.** `align-items-start` on the `Row` and the shrink-wrapping wrapper are load-bearing: a `w-100` or `align-items: stretch` letterboxes the image inside its box and the face boxes drift off the faces. jsdom will not catch this.
