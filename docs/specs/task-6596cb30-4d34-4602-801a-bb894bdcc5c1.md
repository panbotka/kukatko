# User note field

Users already carry a display name (their real name). Add a free-text note so an administrator can record why an account exists or who the person is.

## Requirements

- Add a nullable `note` text column to the `users` table via a new numbered migration in `internal/database/migrations/`, following the existing convention (own transaction, recorded in `schema_migrations`).
- Extend the auth store so the note is read wherever a user is loaded and written wherever a user profile is updated.
- The admin user endpoints (list users, create user, update user) accept and return both the real name (display name) and the note.
- Both fields are optional. Creating a user without either stays valid and they default to empty.
- The note is only ever returned to admin callers. It must not appear in the session or current-user payload that any authenticated user can read.

## Edge cases

- A note longer than 1000 characters is rejected with HTTP 400 and a message naming the field.
- The update endpoint has partial-update semantics: omitting the note from the request leaves the stored note untouched, while sending an empty string clears it.

## Tests

- Integration tests for the admin user endpoints covering: create with a note, partial update that preserves the note, explicit clear, over-length rejection, and that a non-admin caller can never read the note.
