# New user role: "AI"

Add a fourth user role, `ai`, intended for an automated AI agent that authenticates via an API token (Bearer). The AI role may curate the catalog and trigger imports, but nothing else administrative.

## Permission boundary (exact)
The AI role **can**:
- Manage photos: edit metadata, curation, rating/favorites — everything an `editor` can do (i.e. it passes all write-guarded endpoints).
- Manage albums and labels.
- Trigger imports/migration runs (the import endpoints, which are currently admin-only).

The AI role **must NOT** be able to reach any of these (they must return 403 for `ai`):
- User administration (`/admin/users` — create/edit/disable users, change roles, manage other users' API tokens).
- Backups and restore.
- Jobs administration.
- Maintenance (scan/repair).
- Process backfills (embeddings/faces/clusters/places).
- Audit log.
- System status / any other admin-only surface.

In short: `ai` = `editor`'s write powers **plus** import, and nothing else that is admin-gated.

## Backend requirements (`internal/auth`)
- Add `RoleAI Role = "ai"` in `internal/auth/role.go`. Include it in `Role.Valid()`. Make `Role.CanWrite()` return true for `ai` (so it passes `RequireWrite`). `Role.IsAdmin()` must stay **false** for `ai`.
- Import currently uses the shared `RequireAdmin` guard (see `internal/importapi` and the `RequireAdmin` config field pattern used across admin modules). Introduce a dedicated authorization path so that **import** is reachable by `admin` OR `ai`, while every other admin-gated module keeps requiring `admin` only. Prefer adding a new requirement/predicate (e.g. a `RequireImport` middleware / `requireImport` case in `authorize`) and switching the import routes to it — do not broaden the generic `RequireAdmin`, or you will accidentally open backups/jobs/maintenance/etc. to `ai`.
- Add a new SQL migration (use the **next sequential migration number after the highest existing file** in `internal/database/migrations/` — check the directory; do not hardcode a number) that alters the `users.role` CHECK constraint (from `0002_auth.sql`) to include `'ai'`. The `sessions.role` column has no CHECK; confirm nothing else constrains the value.
- Role validation on create/update lives in `internal/auth/service_admin.go` (rejects `!Role.Valid()`), so adding `ai` to `Valid()` makes it assignable automatically — verify.

## Frontend requirements (`web/`)
- Add `'ai'` to the `Role` union and `ROLE_RANK` in `web/src/services/auth.ts`, and to the assignable `ROLES` list in `web/src/services/users.ts` (so an admin can assign the AI role in the Users admin UI). Give it a translated label (cs default + en).
- Ensure `canWrite` is true for `ai` so curation UI is enabled.
- Navigation/menu gating (`web/src/components/Layout.tsx`): an `ai` user should see the curation surfaces (library, albums, labels) and import, but must NOT see the admin-only entries it cannot use (users, backups, jobs, maintenance, processes, audit, system). Gate those entries so they are hidden for `ai` (they currently key off `isAdmin`). Import should be visible/usable for `ai`.

## Tests (mandatory)
- Unit tests for `Role.Valid()`/`CanWrite()`/`IsAdmin()` covering `ai`.
- Integration/HTTP tests asserting the boundary: an `ai`-role principal (token or session) gets **200/allowed** on a write endpoint (e.g. PATCH a photo, album/label mutation) and on an import trigger, and **403** on at least one endpoint from each blocked area (users, backups, jobs, maintenance, process backfill, audit, system).
- Migration applies cleanly against the test DB.
