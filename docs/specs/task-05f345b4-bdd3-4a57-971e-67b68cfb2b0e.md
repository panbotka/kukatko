# Approximate ("circa") date for historical photos

Scanned and inherited photos often have no reliable capture date — you know it is "somewhere in the
forties" or "around 1950", not a timestamp. Today Kukátko has a single `taken_at` and a
`taken_at_source` (exif/filename/manual/unknown), so the only options are to type a precise date that
lies, or leave it empty and lose the photo from the timeline.

Add a way to say **"this date is an estimate"** and to record, in the user's own words, what the estimate
is based on.

## Model

Two new columns on `photos` (new migration, use the **next free number** — check the directory, do not
assume one):

- `taken_at_estimated` BOOLEAN NOT NULL DEFAULT false — the date is a guess, not a fact
- `taken_at_note` TEXT NOT NULL DEFAULT '' — free text, e.g. "kolem roku 1950", "za války",
  "podle babičky někdy před svatbou"

`taken_at` keeps its current meaning and stays the single anchor for **sorting, the timeline, grouping
and date filters** — nothing about ordering changes. The estimate flag is presentation and truthfulness,
not a second date axis. Do not add a date range, do not change `taken_at_source`.

## Requirements

### Backend

- Migration + both columns on the `Photo` struct (snake_case json tags) + `MetadataUpdate`, so
  `Store.UpdateMetadata` writes them.
- `PATCH /api/v1/photos/{uid}` accepts `taken_at_estimated` (bool) and `taken_at_note` (string, trimmed,
  max 500 chars → 400 beyond that). Editor/admin RBAC as for every other metadata field.
- The fields ride along in the existing photo payload.
- **Grep the repo** for other packages that INSERT into or SELECT from `photos` and keep their column
  lists in sync — a stale list compiles fine and breaks at runtime.

### Frontend — editing

In `web/src/components/photo/MetadataPanel.tsx` (the existing editable form on the photo detail page,
gated on `canWrite`), next to the existing `taken_at` datetime field:

- A checkbox "Datum je odhad" / "Date is an estimate".
- A text input for the dating note, shown **only when the checkbox is on** — an empty note on a
  non-estimated photo is meaningless, so do not clutter the form with it.
- Both save through the same PATCH the form already does; no separate save button.

### Frontend — display

- On the photo detail page, wherever the capture date is currently shown, an estimated date renders with
  a clear "circa" marker — prefix the date with `cca` (cs) / `c.` (en) and, when a note exists, show the
  note next to it. It must be impossible to mistake an estimated date for a known one at a glance.
- Give the marker an accessible title/tooltip carrying the note, so the information is not colour- or
  glyph-only.
- The library grid / timeline keep sorting by `taken_at` exactly as today. If the grid tile already shows
  a date, it gets the same `cca` marker; if it shows none, do not add one.

### i18n

All new strings go into **both** `web/src/i18n/locales/cs/common.json` and `.../en/common.json`
(namespace `common`, Czech is the default).

## Edge cases

- `taken_at` is NULL and `taken_at_estimated` is true: allowed — the note carries the meaning
  ("někdy ve 40. letech"), and the photo behaves everywhere exactly as an undated photo does today.
- Unchecking "estimate" leaves the note in place in the form until saved, but a saved non-estimated
  photo must not keep a stale note — clear `taken_at_note` when `taken_at_estimated` goes false.
- Bulk edit is **not** in scope — do not extend `internal/bulk`.

## Verification

- Integration test: PATCH sets both fields, over-long note is rejected with 400, viewer gets 403,
  clearing the flag clears the note.
- Vitest in `web/src/pages/PhotoDetailPage.test.tsx` (and/or a MetadataPanel test): the note input appears
  only when the checkbox is on, an estimated date renders with the `cca` marker and the note, a normal
  date renders unchanged.
- `make check` must pass.
