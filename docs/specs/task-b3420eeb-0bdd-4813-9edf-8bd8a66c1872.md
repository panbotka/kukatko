# Admin Audit Log page

## Summary
Add an admin-only "Audit Log" page that displays the audit trail from the existing `GET /api/v1/audit` endpoint, with filtering and pagination.

## Background (already in place)
- The backend endpoint `GET /api/v1/audit` exists, is **admin-only**, and returns `{ entries, total, limit, offset, next_offset }`. Each entry: `id, actor_uid, action, target_type, target_uid, details, ip, user_agent, created_at`.
- Supported query params: `user` (actor UID), `entity_type`, `entity_uid`, `action`, `since`/`until` (RFC 3339), `limit` (≤500, default 100), `offset`.
- There is currently **no** frontend consuming it.

## Requirements
- A new admin-gated page (e.g. route `/audit`) reachable from the admin navigation, alongside existing admin pages (Users, System status, Maintenance). Non-admins must not see the nav entry or reach the page.
- The page lists audit entries **newest-first** in a table: when, who (actor — display a name where resolvable, else the UID), action, target (type + uid), and IP. The `details` JSON is viewable (e.g. expandable row or a details cell).
- Filters: actor (user), action, target type, target uid, and a date range (since/until), mapped to the endpoint's query params.
- Pagination via the endpoint's `limit`/`offset`/`next_offset` (load-more or pager). Show the total count.
- Filter/sort/page state lives in URL query params so browser Back works, per the project's "Zpět vždy funguje" convention.
- Empty state when no entries match; plus loading and error states.

## Implementation notes
- Add a service module under `web/src/services/` calling `GET /api/v1/audit` with filter params, returning the typed response.
- Add `web/src/pages/AuditPage.tsx`; wire the route in `web/src/App.tsx` behind the same admin guard the Users page uses; add the nav entry where the other admin links live.
- Actor name resolution: if the admin UI already has a user list (the Users page fetches users), reuse it to map actor UID → name; otherwise display the UID. Do not block page render on name resolution.
- Add cs + en i18n keys for title, column headers, filters, and empty/error states.

## Tests
- Frontend tests (Vitest): the page renders entries from a mocked service, applies filters to request params, paginates via `next_offset`, and reflects filter state in the URL. Follow existing admin-page test patterns.
