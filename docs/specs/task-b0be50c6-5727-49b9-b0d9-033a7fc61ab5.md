# Fix: batch Favorite / Archive flashes the whole list

Clicking Favorite or Archive in the multi-select batch action bar makes the entire photo list disappear into a loading skeleton and re-stream all thumbnails — a full-page blink. Read `docs/FRONTEND.md` (photo list / pagination / multi-select) first.

## Root cause (confirmed)

- Batch Favorite/Archive (`web/src/components/organize/BatchActionBar.tsx` ~140-158, 221-237) → on success calls `bulk.finish()` (`web/src/hooks/useBulkEdit.ts` ~75-82), which clears the selection AND calls `onEdited?.()`.
- `onEdited` is the page's `reload` (`web/src/hooks/useReloadKey.ts`), which bumps the reload key fed into `usePhotoLibrary` → `usePaginatedPhotos`.
- `web/src/hooks/usePaginatedPhotos.ts` (~141-152, 184-196): a reload-key change re-runs the fetch effect and immediately does `setData({ ...INITIAL, loading: true })` where `INITIAL` has `photos: []`. So `status` synchronously becomes `'loading'` before the network returns.
- The page (`web/src/pages/LibraryPage.tsx` ~226, 258) then swaps the mounted `<PhotoGrid>` for `<GridSkeleton>` (because `photos.length === 0` / `status==='loading'`) → the grid unmounts and every thumbnail re-loads. Contrast: single-photo favorite updates optimistically in place (`favOverrides`, ~79-90) with no refetch → no flash.
- This affects every list using `useBulkEdit({ onEdited: reload })`: library, album detail, label detail, people (subject), search, favorites, places.

## Requirements

- After a batch action, refresh the affected rows WITHOUT blanking the list to a skeleton. Two acceptable approaches (implementer's choice):
  - **Optimistic in-place update:** apply the batch change to the currently-loaded photos (patch favorite/rating in place; remove archived photos from the list) like the existing `favOverrides` path, without a full refetch; or
  - **Non-blanking reload:** make a *reload* (as opposed to a *query/params change*) refetch in the background while keeping the current `photos` visible — i.e. do not reset to `INITIAL` / `photos: []` and do not flip `status` to `'loading'` when it is only a reload-key bump with unchanged query params. Keep first-load and query-change loading states as they are.
- The fix must hold for all the list pages listed above (the shared `usePaginatedPhotos` / `useBulkEdit` path), not just the library — but must NOT regress the genuine loading skeleton on initial load or when filters/sort/search actually change.
- No full-grid unmount or thumbnail re-stream when doing a batch favorite/archive on an already-loaded list.

## Tests

Assert the initial-load and query-change skeleton still appears. Assert archived photos are removed / favorite reflects in place after the batch action.
