# Rework the photo detail into a clear edit-first hierarchy

The photo detail page (`web/src/pages/PhotoDetailPage.tsx`) already shows everything, but
editing is hard to find. Title, description, private notes and the AI caption are all edited
through a single **"Edit" mode button** buried at the bottom of the first info card
(`web/src/components/photo/MetadataPanel.tsx`), and that button only renders for editors, so
users report "I don't know where to edit the title/description/AI note — it disappeared."
Albums and labels edit inline in `OrganizePanel`, faces via `FaceAssignPanel`, technical facts
collapse in `TechnicalDetails`, and the non-destructive photo editor is a separate `EditPanel`
card. The information hierarchy does not match how the app is actually used: **organizing a
photo (albums, tags, people) is the everyday action and must be the most prominent and the
easiest to edit**; captions and location are next; camera facts are reference; pixel-level photo
editing is rare.

Restructure the detail page around that hierarchy. Keep every existing capability and endpoint —
this is a reorganization of prominence and editing affordances, not a rewrite of features.

## Information hierarchy (top → bottom of the side panel)

1. **Organize — Albums, Tags, People. The primary block. Always visible, always the first
   panel, always directly editable** (no expander, no separate "edit mode"). For an editor/admin
   the add/remove controls are visible immediately; a viewer sees the chips read-only.
   - Albums: existing add-autocomplete + removable chips (reuse `OrganizePanel`).
   - Tags (labels): existing add-autocomplete with create-on-type + removable chips.
   - People: the faces/people this photo contains, shown as named chips with the ability to
     assign/rename/remove a person, reusing the existing face-assign flow (`FaceAssignPanel`,
     `useFaces`). Assigning a face to a person and clearing it must remain possible from here.
   - Albums, tags and people should be visually distinct from each other (e.g. a colour or icon
     per kind); if the app already defines a colour convention for album/tag/person chips, reuse
     it, otherwise give the three kinds a clear visual distinction here.

2. **Caption & place — Title, Description, AI description, Location. Visible read-only, edited
   on demand.** Each of these shows its current value (or a muted "add…" placeholder when empty).
   An editor reveals the editing controls by clicking the section (an inline "edit"/expand
   affordance on the block) — NOT by a single hidden global "Edit" button at the bottom of the
   card. When expanded it exposes: title (text), description (textarea), AI description /
   `ai_note` (textarea), private notes (textarea), taken-at (datetime-local), and the location
   (coordinate text field + the existing Leaflet map picker). Save issues the existing
   `PATCH /api/v1/photos/{uid}`; cancel reverts. It must be obvious to an editor how to edit each
   of title, description and AI description — that discoverability is the whole point of this task.

3. **Technical details — camera, lens, aperture, exposure, focal length, ISO, dimensions, file
   name, file size, format, uploader. Reference only, collapsed by default** (reuse
   `TechnicalDetails`). Read-only.

4. **Photo editing — crop / rotate / brightness / contrast. Last, collapsed by default, editor
   only** (reuse `EditPanel`). This is used rarely; it must sit at the very bottom and never
   compete with the organize block for attention.

## Header

- The header row leads with the back control (returns to the originating list, preserving
  scope/order — keep `backHref`), immediately followed by the photo's **title**; when the photo
  has no title, fall back to its original **file name**. This title/filename must be clearly
  associated with the back arrow and must not be pushed off-screen or hidden on narrow widths.
- Keep the right-aligned header controls (rating, flag, favorite) and the on-image prev/next
  arrows and ←/→ keyboard navigation.

## Every editable attribute must have a control (completeness)

The whole point is that a photo is fully editable and every editable attribute is discoverable —
no attribute the API accepts should be missing a UI control. Cover the **entire** editable set:

- Via `PATCH /api/v1/photos/{uid}` (`PhotoMetadataUpdate`): **title, description, notes** (private
  capture notes), **ai_note** (AI description), **taken_at**, **location (lat/lng)**, and the
  **private / visibility flag**. `private` is currently editable by the API but has **no control in
  the UI** — add a clear "private / visibility" toggle for editors (place it sensibly — with the
  caption & place editing controls, or as a header control next to favorite — and wire it to the
  `private` field; there is already a "private" filter in the library, so this closes the loop).
- Plus the controls that already exist and must remain: **albums, tags, people** (organize block),
  **rating, flag, favorite** (header), **archive**, and the **non-destructive editor**
  (crop/rotate/brightness/contrast).
- Intrinsic/read-only facts (camera, lens, EXIF, dimensions, file name/size/format, uploader) stay
  read-only in Technical details — they are not user-editable and are out of scope for editing.

## Requirements

- The three edit tiers behave exactly as described: Organize is always-on inline editing;
  Caption & place is read-only-until-expanded per-section editing; Technical and Photo-editing
  are collapsed-by-default.
- No capability regresses. Title, description, AI note, private notes, taken-at, location,
  private/visibility, albums, tags, people, rating, flag, favorite, archive, download-original,
  similar-photos, prev/next, the fullscreen lightbox, and the non-destructive editor all still work.
- RBAC is preserved: viewers see everything read-only with no add/remove/edit controls; editors
  and admins get all editing affordances. The editing controls must be genuinely discoverable
  for an editor — the failure this task fixes is an editor not finding where to edit.
- Reuse the existing components (`OrganizePanel`, `MetadataPanel`, `TechnicalDetails`,
  `EditPanel`, `FaceAssignPanel`, `FaceOverlay`) and the existing services/endpoints. Refactor
  their composition and prominence; do not invent new endpoints (the `private` field already exists
  on the PATCH payload).
- Works on mobile: on a narrow viewport the panels stack under the image in the same priority
  order (Organize first), remain reachable, and the header title stays visible.
