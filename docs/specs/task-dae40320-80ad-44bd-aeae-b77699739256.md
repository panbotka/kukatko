# Resolve duplicates by merging into the best copy

The duplicates page detects near-duplicate groups but is **read-only** — there is no way to act on
them. Add a resolution flow that does not simply throw copies away: the user keeps the **best** copy
and the redundant copies are **merged into it** so nothing organizational is lost — the keeper ends
up in every album and with every tag the duplicates had, and tagged with every person that appeared
on any copy. Only then are the redundant copies archived.

## Current state (for context)

- `internal/duplicates` produces near-dup `Group`s (pHash LSH + embedding HNSW, union-find);
  `internal/duplicatesapi` exposes only `GET /duplicates` (behind `requireWrite`). `DuplicatesPage`
  renders the groups.
- Album/label membership ops exist in `internal/organize` (`AddPhoto`/`RemovePhoto`, audited
  `AddPhotosAudited`, `AttachLabelAudited`/`DetachLabelAudited`, `AlbumsForPhoto`). People are linked
  to photos via markers in `internal/people` (subjects + markers). Photos are soft-archived via
  `internal/photos` `ArchiveAudited` (originals retained until a later purge). Mutations are recorded
  in the audit trail in the same transaction.

## Behavior

- **Pick the keeper.** Within a group, auto-select the best copy — highest resolution (pixel count),
  tie-break by larger file size, then by richest metadata / earliest created. The user can override
  which member is the keeper before confirming.
- **Merge the rest into the keeper, in one transaction:**
  - **Albums:** the keeper becomes a member of every album that any copy in the group belonged to
    (union; skip albums it is already in).
  - **Tags/labels:** attach to the keeper every label carried by any copy (union).
  - **People:** the keeper ends up tagged with the union of every person (subject) tagged on any copy
    in the group. Note that a face marker's bounding box is pixel-specific to one file and cannot be
    copied meaningfully to a different image — so reflect the *people present* (ensure a
    subject↔keeper association/marker exists for each person found across the group), not the exact
    boxes; if the keeper already has that person, skip.
  - **Scalar metadata (fill gaps only):** if the keeper lacks a title, description, rating, favorite
    or flag and a copy has one, carry it over. Never overwrite a value the keeper already has.
  - **Archive the redundant copies** via the soft archive (`ArchiveAudited`) so they leave the
    library but their originals are retained until purge. Never hard-delete originals here.
  - Record the merge in the audit trail.
- Editors/admins only (the endpoint stays behind `requireWrite`); viewers cannot resolve.

## Requirements

- Add a backend endpoint (e.g. `POST /duplicates/merge` / `/resolve`) that takes the group and the
  chosen keeper UID and performs the transactional merge + archive above, reusing the existing
  organize/people/photos services rather than duplicating their SQL. It is idempotent enough that
  re-running on an already-resolved group is a safe no-op (the copies are already archived).
- **Frontend:** on `DuplicatesPage`, each group shows its members with the suggested keeper
  highlighted and selectable; a "Keep best & merge" action shows a short preview of what will move
  (e.g. "+3 albums, +2 tags, +1 person · 2 copies will be archived"), asks for confirmation, calls
  the endpoint, and on success removes the group from the list. Show errors clearly.

## Verification

`make check` must pass. Backend: test that merging unions album and label memberships and people
onto the keeper, fills only missing scalar fields, archives the non-keepers, writes an audit entry,
and enforces RBAC (viewer forbidden); test the keeper auto-selection heuristic. Frontend: test that
the page lets the user pick a keeper, previews the merge, calls the endpoint, and drops the resolved
group. Document the endpoint in `docs/API.md`, the flow in `docs/FRONTEND.md`, and update
`docs/PACKAGES.md` (duplicatesapi) / `docs/ARCHITECTURE.md` if the resolution model is described.
