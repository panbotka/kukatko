# Maps: send a configurable User-Agent to mapy.com

The mapy.com API key is currently restricted **by source IP** in the mapy.com console. That is brittle: this host's public IPv4 and its ISP-delegated IPv6 prefix both change, and when they do the key starts answering `403 Forbidden` and every map tile turns grey (this already happened — the IPv6 egress address was not on the allowlist).

The key is used **server-side only** (the backend proxy; it never reaches the browser), so the plan is to restrict the key **by User-Agent** instead of by IP. The mapy.com console allows only **one restriction type per key**, so this is a switch, not an addition. mapy.com requires an **exact** User-Agent match (no wildcards), and we control the string, so a random token inside it acts as a second secret: the `apikey` travels in the query string and is the thing most likely to leak into a log, while the User-Agent does not — a leaked key alone is then useless.

The client today sends Go's default `Go-http-client/2.0` (`internal/mapy/mapy.go:303` builds the request without touching headers).

## Requirements

- Every request the mapy.com client makes — tiles **and** geocode/rgeocode — carries a `User-Agent` header taken from configuration.
- New config key `maps.user_agent` (env `KUKATKO_MAPS_USER_AGENT`), a string, **default empty**. Empty = send no explicit header (Go's default), so nothing breaks for an instance that has not set it.
- The value is already provisioned in `.secrets/db.env` as `KUKATKO_MAPS_USER_AGENT` — read it from the environment, do not hardcode it, do not commit it, and **never log it** (it is a credential, exactly like `maps.mapy_api_key`; the same goes for the api key, which must never appear in a log line or an error message).
- Register the key everywhere the project rules require, in one change: the `Config` struct, `setDefaults`, `config.example.yaml` (documented with a **placeholder**, never the real token), the config tests, and `docs/OPERATIONS.md`.
- Unit test: the outgoing request carries the configured `User-Agent` when set, and no explicit one when the key is empty. Assert it against a fake upstream (`httptest`), for both the tile and the rgeocode path.
- Verify against the **live** mapy.com API that a tile request still returns a real PNG with the header in place. At this point the key is still IP-restricted, so it must keep working — the header is simply not checked by anyone yet.

## Do NOT

- Do not add a `Referer` header or ask for a referer restriction. mapy.com validates only the host+port of a header **we send ourselves** — with no browser in the loop it is self-attested and worth nothing for a server-side key.
- Do not force IPv4 (an earlier draft of this task did; the IPv6 address has since been allowlisted and the IP restriction is going away anyway).
- Do not touch the mapy.com console. Switching the key from the IP restriction to the User-Agent restriction is a **manual step the user performs after this ships and the instance is restarted** — the console permits only one restriction type at a time, so the switch is atomic and the code must be sending the header first.
