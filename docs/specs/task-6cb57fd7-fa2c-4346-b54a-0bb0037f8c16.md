# Fix: batch "Add to album" / "Add label" picker never loads

In the multi-select floating batch action bar, clicking **"Přidat do alba"** or **"Přidat štítek"** opens the modal frame but its body is stuck on the loading spinner forever — the album/label options never load, so the user can never pick anything. Root cause is a self-aborting effect. Read `docs/FRONTEND.md` (multi-select / batch bar) first.

## Root cause (confirmed)

`web/src/components/organize/BatchActionBar.tsx`, the option-loading `useEffect` (~lines 101-124):
- It calls `setOptions({ status: 'loading' })` inside the effect, and `options.status` is in the effect's dependency array.
- So setting status to `loading` re-triggers the effect: React runs the cleanup `controller.abort()` — aborting the `fetchAlbums`/`fetchLabels` request it just started — then the re-run early-returns on the `options.status !== 'idle'` guard, so no new fetch is issued.
- The aborted fetch rejects with an `AbortError` that the `.catch` swallows, leaving `options` stuck at `{ status: 'loading' }` permanently → the modal body only ever shows the spinner.
- Real `fetchAlbums`/`fetchLabels` pass the `AbortSignal` into `fetch` (`web/src/services/organize.ts`), so the abort actually cancels the network request — which is why it fails only in the running app.

The correct pattern already exists in `web/src/components/photo/OrganizePanel.tsx` (~lines 55-69) and `web/src/components/organize/BulkEditModal.tsx` — their effects key on the trigger (`show`/`canWrite`) and keep the loaded data OUT of the deps, so setting the result never aborts the in-flight fetch.

## Requirements

- Fix the effect so opening the album/label picker actually loads the options and renders the picker fields. Restructure so the effect does NOT depend on the state it writes: re-run on the picker trigger only (`picker`), and do the idle/once guard without putting `options.status` in the dependency array (mirror `OrganizePanel` / `BulkEditModal`). The in-flight fetch must not be aborted by the effect's own state write; the cleanup should still abort on genuine unmount / picker close.
- After the fix: opening "Add to album" loads the album list into the `MultiSelect`; opening "Add label" loads labels (add + remove); Apply becomes enabled once a selection is made and calls the existing `bulkUpdatePhotos` (`POST /photos/bulk`) path (which already works). Do not change the backend or the bulk service.
- Handle a genuine (non-abort) fetch error by showing the existing error state rather than an infinite spinner.

## Tests

- **Fix the misleading test:** `web/src/components/organize/BatchActionBar.test.tsx` ("opens the album picker") currently mocks `fetchAlbums`/`fetchLabels` with `mockResolvedValue` that IGNORE the `AbortSignal`, so it passes even though production aborts. Update the mock to respect the `AbortSignal` (reject with an `AbortError` when aborted) — reproducing the bug — then assert that opening the picker reaches the loaded/ready state and renders the options (the album `MultiSelect` / label fields), proving the fetch is not self-aborted.
- Add/adjust a test for the genuine-error path (a real fetch rejection shows the error state, not an infinite spinner).
