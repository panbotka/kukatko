# Album List Ordering

The album list must always be ordered from newest to oldest by the photos each album contains. Albums with no usable date go last.

## Current behaviour

- `internal/organize/albums.go` — `(*Store).ListAlbums(ctx)`, SQL in the const `listAlbumsSQL` (~line 195). Current `ORDER BY a.title, a.uid` — alphabetical.
- The query ALREADY selects the dates you need: `MIN(p.taken_at) AS taken_from` and `MAX(p.taken_at) AS taken_to`, computed over live photos only (`p.archived_at IS NULL AND (p.stack_uid IS NULL OR p.stack_primary)`). No new joins are needed — these are aggregates and are legal in ORDER BY after GROUP BY. They already surface as `TakenFrom`/`TakenTo` on `organize.AlbumSummary`.
- `photos.taken_at` is `TIMESTAMPTZ` and NULLable (migration `0003_photos.sql`). There is no `taken_at_local`. The `albums` table has no album-level date column — only `created_at`/`updated_at`.
- `GET /albums` (`internal/organizeapi/albums.go`, `handleAlbumList`) takes no query parameters; order is fixed server-side.
- The frontend does not sort: `web/src/pages/AlbumsPage.tsx` maps `state.albums` in server order.

## Requirements

- Order albums by their NEWEST photo — `MAX(p.taken_at)` — descending. The album holding the most recent photo comes first.
- Albums whose live photos all lack `taken_at`, and empty albums, have a NULL max and must sort LAST (`NULLS LAST`).
- Append `a.uid` as a tiebreaker so the order is total and stable.
- This ordering is ALWAYS applied. Do NOT add a `sort`/`order` query parameter and do NOT add a sort control to the UI — the order is not a user choice.
- Follow the repo's existing convention, `internal/photos/store_list.go` `orderClause` (~line 627): always `NULLS LAST` regardless of direction, always a `uid` tiebreaker. The same idiom already appears in this very file in the cover-photo LATERAL join (`ORDER BY p2.taken_at DESC NULLS LAST, p2.uid`) and in `internal/people/subjects.go`.
- Do NOT use `COALESCE(taken_at, created_at)` here. That fallback is right for single photos (where `created_at` is NOT NULL) but wrong for albums: it would give an undated album the album's upload time and float it to the TOP, the exact opposite of the requirement.

## Edge case worth checking

`AlbumsPage`'s create-modal `onSaved` callback appends a new album with `[...prev.albums, { ...album, photo_count: 0 }]`, so a fresh album lands at the END regardless of server order until the next refetch. Under the new ordering an empty album belongs at the end anyway (NULL date → last), so this is likely already correct — verify it, and if it disagrees with the server order, refetch instead of appending.
