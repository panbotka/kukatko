# Per-user review-decision audit (admin)

Give admins a way to inspect an individual user's sorting decisions ŌĆö which photos they confirmed/rejected as which person or label, and Ano vs Ne ŌĆö so a reviewer can spot someone who is sorting poorly. This builds on the durable audit log, where every decisive review answer is recorded with `actor_uid` and `details.via = 'review'`.

This task assumes review face-confirmations (face "yes") are already tagged `via:review` (done in the leaderboard/attribution task). If that tag is still missing, the view simply omits face confirmations ŌĆö do NOT re-implement the tagging here, but note the dependency.

Read `docs/API.md` (audit + review), `docs/FRONTEND.md` (AuditPage / audit service), and the `internal/auditapi` package docs first.

## Requirements

- **Backend ŌĆö filtered audit listing**: extend the audit read API (`internal/auditapi`, `GET /audit`, admin-only) so it can be filtered to one user's review decisions: a filter by `actor_uid` and a filter restricting to review activity (`details.via = 'review'`, i.e. the actions `face.assign`, `label.attach`, `face.reject`, `label.reject`). Support paging/limit consistent with the existing endpoint. Each returned record should carry enough to render a decision: the action (which maps to Ano for assign/attach, Ne for reject), the target (person/subject or label), the photo reference where present (`details.photo_uid`/`face_index`), and the timestamp. Keep it `RequireAdmin`.
- **Frontend ŌĆö per-user decision view**: a view showing one user's review decisions, reachable **from the leaderboard** (clicking a user on `/leaderboard` opens their decision history). This affordance is admin-only: the click-through/link is shown only to admin-or-higher; non-admins just see the standings. Reuse the existing audit UI infrastructure (`web/src/services/audit.ts`, `web/src/lib/auditView.ts`, `web/src/pages/AuditPage.tsx` patterns) rather than building a parallel stack.
  - Show, per decision: the photo thumbnail (where the record references a photo), whether it was Ano (confirm/attach) or Ne (reject), the person or label involved, and when. Make it filterable/groupable by Ano/Ne.
  - Include the user's name and their tallies at the top (can reuse the leaderboard counts).
  - Empty state when the user has no recorded decisions.
- Keep URL state (selected user, any Ano/Ne filter, paging) in query params.

## Behavior / edge cases

- Only admin-or-higher can reach this data (backend `RequireAdmin`; frontend hides the affordance otherwise). A non-admin hitting the endpoint directly gets 403.
- Deleted users (`actor_uid` NULL) have no attributable history ŌĆö handle gracefully.
- Skips are not recorded, so they never appear ŌĆö expected.

## Tests

- Backend integration: seed review + non-review audit rows for two users; assert the filtered endpoint returns only the target user's `via:review` rows, correctly split Ano/Ne, respects paging, and is 403 for a non-admin, 200 for admin.
- Frontend: the leaderboard shows the click-through only for admins; the decision view renders decisions from a mocked service with the Ano/Ne split and thumbnails, plus an empty state.
