# Publish the Migration Audit Report

Render the migration audit to a PDF, publish it, and email the link to the project owner.

## Requirements

- **Source:** `docs/MIGRATION_AUDIT.md` in this repo — the field-by-field audit of the PhotoPrism → Kukátko and photo-sorter → Kukátko mappings, where every row is marked MAPPED, WAIVED or GAP. **If that file does not exist, stop and report. Do not invent a report and do not send an email.**
- **Render** it to a PDF that is genuinely readable: wide mapping tables must not be cut off at the page edge (landscape orientation or a smaller table font is fine), and the document needs a title, a generation date and page numbers.
- **Publish** to `/opt/soubory/files/kukatko/migration-audit/`, which is served publicly at `https://soubory.panbotka.cz/kukatko/migration-audit/`. Put the date in the filename. Do not dump files into the root of the share.
- **Email the link to `kozak@talko.cz`** using the `botka-mailbox` skill. Subject and body in Czech. The body must summarise the findings — how many fields were audited, how many mapped, how many waived, and the list of real gaps with one line each — so the report can be triaged from the mail alone, without opening the PDF. Then the link.

## Rendering notes

This box is an ARM64 Raspberry Pi. **NEVER** run `npx playwright install`, download a browser, or attempt any browser-install step — they always fail here. Chromium IS already available at `/usr/local/bin/chromium`. A reliable path is markdown → styled HTML → `chromium --headless --no-sandbox --print-to-pdf=<out.pdf> <file.html>`. Use `pandoc` or `weasyprint` instead only if they are already installed — check first, do not install. Note `markitdown` is the wrong direction (it converts documents INTO markdown).

## Rules

- **The published PDF is on the public internet — anyone with the URL can read it.** Before publishing, read the report and confirm it contains no DSNs, no API keys, no passwords, no personal data, nothing from `.secrets/`. If it does, stop and report instead of publishing.
- **Verify the published URL actually serves the PDF** (HTTP 200, correct content type) BEFORE sending the email. Never email a link you have not fetched.
- Do not commit the PDF into the repository.
- This task changes no source code. If the working tree is clean at the end, that is the correct outcome and there is nothing to commit — the deliverable is the published PDF and the sent email, and your final report must state the public URL and confirm the mail went out.
