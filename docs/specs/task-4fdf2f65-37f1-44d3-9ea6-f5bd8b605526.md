# Timeline Buckets — API

Provide a date histogram of the photo library so the frontend can render a fast year/month scrubber and jump to a date. Buckets honor the same filters as the photo list. Backend only.

## Requirements
- `GET /api/v1/photos/timeline` (RequireAuth) in `internal/photoapi`: returns month-granularity date buckets ordered the same way as the default grid (by `taken_at`, newest-first, with NULLs handled consistently with the list). Each bucket: `{year, month, count, cumulative}` where `cumulative` is the number of photos that appear BEFORE this bucket in newest-first order (so the client can map a bucket to a scroll index). Include the overall `total`.
- The endpoint accepts the same filter query params as `GET /photos` (archived/private/has_gps/date range/camera/lens/uploader/album/label/country/city/favorite/`q` when present) via the existing `parseListParams`, and the aggregation must respect them — buckets must match exactly what the list would return in the same newest-first order. The `sort` param is irrelevant (always grouped by date); document that the scrubber assumes the default date sort.
- Add a photos store method `TimelineBuckets(ctx, ListParams)` that reuses the shared `buildWhere` so filters stay consistent with `List`/`Count`. Compute `cumulative` server-side.
- Integration test: bucket counts and cumulative correctness, archived excluded by default, and a filtered case (e.g. scoped to an album) matching the list.
