# Saved Searches (Smart Albums) — UI

Let users save the current library/search view (filters + sort + search query + mode) as a named saved search and re-open it later. Saved searches are owner-private. Backend: `GET/POST/GET/PATCH/DELETE /api/v1/saved-searches` with body `{name, params}` where `params` is opaque JSON (store the URL view-state object verbatim).

## Requirements
- `web/src/services/savedSearches.ts`: typed client `fetchSavedSearches()`, `createSavedSearch(name, params)`, `updateSavedSearch(uid, {name?, params?})`, `deleteSavedSearch(uid)`; type `SavedSearch`. `params` is the same object the app already serializes into the URL via `useUrlState`/`viewToParams` — store and restore it verbatim.
- On `LibraryPage` and `SearchPage`: a "Save view" (cs: "Uložit pohled") action that captures the current view state object and prompts for a name in a small modal, then calls `createSavedSearch`.
- A "My saved searches" (cs: "Moje uložená hledání") surface listing the current user's saved searches with name + rename + delete actions. Implement it as a dedicated `/saved` page AND a quick dropdown in the navbar search area (link the dropdown to the page for management). Opening a saved search navigates to `/library` — or `/search` if its params contain a search query/mode — with the saved `params` applied to the URL so filters/sort/query are restored exactly. Add the appropriate nav link in `Layout`.
- Optimistic create/delete with error handling; empty state ("Zatím žádná uložená hledání").
- i18n cs + en in BOTH locale files (Czech default). No hardcoded UI text.
- Vitest tests: saving captures the current view params; opening a saved search restores the URL state (navigates with the saved params).
