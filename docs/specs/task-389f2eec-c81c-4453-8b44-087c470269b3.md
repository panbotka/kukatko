# Documentation & repo hygiene sweep

Do a full pass over the repository's documentation and get it into a good, accurate, consistent state. This is a **hygiene/correctness sweep, not a redesign** — fix staleness, gaps, broken links, and inconsistencies; do not rewrite the architecture or invent new scope. Many feature tasks landed before this one, so the goal is to make the docs match the code as it actually is now.

## Method (verify against the code, don't just re-read prose)
For each area, cross-check the documentation against the real source of truth in the codebase (grep the packages, routes, config struct, make targets, migrations). A doc line that no longer matches the code is a bug to fix.

## Checklist

### 1. CLAUDE.md (rules + signpost + package map)
- The **Package map** section lists one line per `internal/*` (and `cmd/*`, `web/`) package. Reconcile it against the actual directories: every existing package has a line; no line points to a package that was removed/renamed; descriptions still match what the package does. Add/remove/fix lines as needed.
- Every document referenced in the "Where to find what" table and elsewhere (`docs/ARCHITECTURE.md`, `PACKAGES.md`, `API.md`, `FRONTEND.md`, `OPERATIONS.md`, `DEVELOPMENT.md`, `PERF.md`, `RESTORE.md`, `UX_AUDIT.md`) must exist and the link must resolve.
- `make docs-budget` enforces the 300-line limit on CLAUDE.md — if the file is over budget, move descriptive detail into the right `docs/` file (never delete rules to fit). Confirm `make docs-budget` passes.

### 2. docs/PACKAGES.md
- One entry per Go package; reconcile with `internal/*` and `cmd/*`. Fix descriptions that drifted, add missing packages, remove entries for packages that no longer exist.

### 3. docs/API.md
- Reconcile documented endpoints against the actual routes (grep the `*api` packages and `internal/server`/route wiring). Every `/api/v1` route should be documented with method, path, auth requirement, and payload shape; remove docs for routes that no longer exist; fix changed request/response fields.

### 4. docs/FRONTEND.md
- Reconcile documented components/hooks/pages/services against `web/src`. Fix renamed/removed items, add notable new ones. Don't try to document every file — keep it at the level the doc already targets, but make what's there correct.

### 5. docs/OPERATIONS.md + config.example.yaml
- Reconcile the config-key catalog against `internal/config` (the `Config` struct + `setDefaults`). Every config key exists in `config.example.yaml` with a sane documented default and is listed in `docs/OPERATIONS.md`; env-var mapping notes (`KUKATKO_` prefix, the `maps.mapy_api_key` ↔ `MAPY_API_KEY` exception) are correct.
- Reconcile documented CLI subcommands (`serve`/`migrate`/`import`/`backup`/`restore`/`maintenance`/`ctl`/`version` and any others) and `make` targets against the actual Cobra commands and the `Makefile`. Fix drift.

### 6. README.md
- Ensure it accurately reflects what the app is and its current user-visible feature set and setup/run instructions. Fix stale claims, dead links, and outdated commands. Keep the tone/scope it already has.

### 7. Cross-cutting hygiene
- Fix broken internal links between docs, obviously outdated statements, duplicated/contradictory info, and glaring typos in the docs (English per project rule).
- Do NOT touch application code behavior. Doc/markdown/comment edits only, plus `config.example.yaml` if a key is genuinely missing/mislabeled. If you discover a code-vs-doc mismatch where the **code** looks wrong (not the doc), do not fix the code — note it briefly in the commit message / a short `## Known mismatches` note rather than silently changing behavior.

## Constraints
- Stay within one session — if the repo is large, prioritize **correctness fixes** (wrong/stale/broken) over stylistic polish, and cover the highest-traffic docs first (CLAUDE.md package map, API.md, OPERATIONS.md/config, README) before the rest.
- Keep changes reviewable and scoped to documentation.

## Done when
- `make docs-budget` passes and the docs listed above are consistent with the current code.
- `make check` passes (docs-budget + fmt + lint + tests must stay green — this task shouldn't break any of them) and `make dev` still starts.
- Commit (summarize what was reconciled/fixed) and push per project rules.
