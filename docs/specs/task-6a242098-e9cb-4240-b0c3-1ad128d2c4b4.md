# The review game — backend

A new way to tidy the library: instead of hunting through pages, Kukátko asks **one question at a time** and you
answer yes / no / don't know. *"Je tohle Tomáš Kozák?"* over a photo with an unnamed face. *"Má tahle fotka mít
štítek Ostatky?"* over a photo that looks like the ones already on that label. Answer, next question, answer,
next question.

This task builds the queue and the answer endpoint. The page is a separate task.

**Prerequisites** — this composes existing pieces, it does not re-implement them. Confirm they exist:
`internal/feedback` (persisted rejections + the negative-exemplar rule), the unassigned-only face vector search,
the per-subject candidate service, and the album/label similarity service. If a prerequisite is missing, say so
rather than building a private copy of it.

## What makes a good question

The queue is the whole feature. A queue of obvious questions is boring and teaches the system nothing; a queue of
impossible ones is demoralising.

**Target the uncertainty band.** Rank candidates by how close they sit to the decision boundary — the ones where
the system genuinely does not know are the ones where a human answer buys the most. Concretely: for a face
candidate, the confidence against the subject; for a label candidate, the similarity to the label's photos. Pick
from a **middle band** (roughly 45–75 % confidence — make the band configurable, do not hard-code it), ordered so
the most informative question comes first.

Deliberately **exclude**:
- anything already answered — a rejection in `internal/feedback`, an existing assignment, an existing label;
- anything the negative-exemplar rule kills;
- faces below the minimum reviewable size (a 20-pixel face in a crowd is not a fair question);
- candidates so certain they are not worth asking about (those belong on the `/recognition` page, where you
  confirm them in bulk).

**Mix the two kinds.** A queue of forty face questions in a row is a grind. Interleave face and label questions —
roughly alternating, but skewed toward whichever kind actually has good candidates. Make the mix deterministic
given the same library state, so the queue is reproducible and testable (the scripting environment has no
randomness available in tests — do not reach for `rand` to shuffle).

## API

- `GET /api/v1/review/queue?limit=N` (RequireWrite) — returns a batch of questions (default ~20, so the UI can
  prefetch and stay instant between answers).

  Each question carries: a stable question id, its kind (`face` | `label`), the photo (with `thumb_url`), and the
  subject-under-question or label-under-question. Face questions additionally carry the face's bbox — pixel **and**
  display-relative, honouring EXIF orientation — plus what confirming would do (create a marker vs assign a person),
  so the answer endpoint has no ambiguity. Every question carries its confidence, because the UI shows it.

- `POST /api/v1/review/answer` (RequireWrite) — body: the question id and the answer (`yes` | `no` | `skip`).
  - **yes** on a face → assign, through the **existing** assign state machine (`POST /photos/{uid}/faces/assign`);
    do not open a second write path.
  - **yes** on a label → add the label, through the existing organize path.
  - **no** → record a rejection in `internal/feedback`. This is what makes the game converge: a "no" is permanent,
    the question never comes back, and the negative-exemplar rule uses it to kill similar candidates.
  - **skip** → record that it was skipped and **do not ask it again in this session**, but do not treat it as a
    rejection. "Don't know" is not "no". Whether a skip is durable across sessions is your call — but a skipped
    question must not immediately reappear at the top of the next batch, which is the obvious bug here.
  - Idempotent: answering the same question twice must not double-assign or error out.
  - Every write audited in the same transaction, per project rules.

- Return, alongside the answer, enough for the UI's session counter (answered so far, and how many candidates
  remain) — without recomputing the whole queue on every answer, which would be pathologically slow. An estimate is
  fine and better than an exact number that costs a second.

## Performance

The queue must be **fast** — the game is unplayable if each batch takes five seconds. The candidate searches are
expensive; batch them, bound the concurrency (this box is RAM-constrained), and consider caching a queue per user
for a short window rather than recomputing from scratch on every request. Do not hold the whole face table in Go
memory.

## Config

New keys for the uncertainty band and the queue size (`Config` struct + `setDefaults` + `config.example.yaml` +
tests + `docs/OPERATIONS.md`, all at once).

## Edge cases

- A library with **no named people and no labels**: return an empty queue with a reason the UI can show ("nejdřív
  pojmenuj pár lidí a založ štítky"), not an error.
- The embeddings sidecar being offline is irrelevant here — the vectors are already in Postgres.
- A question whose underlying photo/face/label was deleted between fetch and answer: fail that one answer
  gracefully (the UI moves on), do not 500.

## Verification

- Integration tests with a seeded library: the queue only contains candidates in the configured band; a rejected
  candidate never appears; an already-assigned face never appears; a skipped question does not lead the next
  batch; `yes` on a face actually assigns; `yes` on a label actually labels; `no` writes a rejection and the same
  question is gone from the next queue; answering twice is idempotent; the empty-library case returns a clean
  empty queue.
- A test that the queue is **deterministic** for a fixed library state.
- `make check` must pass.
- Document: `docs/API.md`, `docs/PACKAGES.md` (+ the `CLAUDE.md` package map line if a new package appears),
  `docs/OPERATIONS.md` + `config.example.yaml` for the config keys, and `docs/ARCHITECTURE.md` for the
  uncertainty-band design decision.
