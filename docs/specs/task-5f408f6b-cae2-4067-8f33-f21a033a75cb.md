# Backend: per-user sorting leaderboard

Add a leaderboard endpoint that ranks users by how many sorting decisions they have made in the review game, turning `/review` into a friendly competition. Read `docs/API.md` (Review game API), `docs/ARCHITECTURE.md` (review section), and the `internal/review` / `internal/reviewapi` / `internal/audit` package docs in `docs/PACKAGES.md` first.

## Background — how attribution works today

Every decisive review answer already writes a row to the durable audit log (`audit_log`) in the same transaction as the mutation, tagged with the acting user (`actor_uid`) and `details->>'via' = 'review'`:
- label "yes" → action `label.attach`, tagged `via:review`
- face "no" → action `face.reject`, tagged `via:review`
- label "no" → action `label.reject`, tagged `via:review`
- **face "yes" → action `face.assign` — NOT currently tagged `via:review`** (it routes through `internal/facematch` `Service.Apply`, which builds its own audit entry). This is the one gap.
- "skip"/"Nevím" writes nothing (in-memory session only) and must NOT count.

## Requirements

- **Close the attribution gap**: when the review game confirms a face (face "yes"), the resulting `face.assign` audit entry must be tagged `details.via = "review"`, consistent with the other three review actions. Thread the `via:review` marker from the review answer path (`internal/review/answer.go`) into the facematch apply call / audit metadata so review-originated face confirmations are distinguishable from ordinary recognition assignments. Do NOT change ordinary (non-review) face assignments — they stay untagged.
- **Leaderboard aggregation**: add a query that aggregates, per user, the count of review **decisions** — the `via:review` audit rows across `face.assign`, `label.attach`, `face.reject`, `label.reject` — grouped by `actor_uid`, joined to `users` for `display_name`/`username`. "Yes" decisions = `face.assign` + `label.attach`; "No" decisions = `face.reject` + `label.reject`. Skips are not recorded and are excluded by construction. Support **time windows**: all-time, last 7 days, and today (a `window` query parameter, default all-time; compute cutoffs from `created_at`). Exclude rows whose `actor_uid` is NULL (deleted users).
- **Endpoint**: `GET /review/leaderboard` (place it with the review API, `internal/reviewapi`), gated `RequireAuth` — visible to any logged-in user (it returns only aggregate counts + display names, no audit detail). Response: an ordered array (highest total first) of `{ user_uid, display_name, yes_count, no_count, total }`, plus the window that was applied, and enough for the frontend to identify the caller's own row (the caller's uid is available from context — include a flag or let the frontend match on uid).
- Wire the new endpoint in `cmd/kukatko` (the review API build), sharing the existing DB pool / audit store.

## Behavior / edge cases

- A user with zero decisions in the window need not appear (absent = 0). Returning only users with ≥1 decision is acceptable for v1; the frontend handles "not on the board yet".
- Break ties deterministically (total desc, then yes desc, then display_name).
- Keep it efficient — rely on the existing `audit_log` indexes on `actor_uid`, `created_at`, `action`. If a `details->>'via'` filter needs an index for acceptable performance, add it in a migration.

## Tests

- Integration test against the test DB: seed audit rows for several users across the four review actions and across time (today / within 7 days / older); assert the leaderboard counts, the yes/no split, ordering, and each window (all-time / 7d / today). Assert NULL-actor rows and non-review rows (no `via:review`) are excluded, and skips (no row) don't count.
- Test that a review face-"yes" now produces a `face.assign` audit row tagged `via:review`, while a non-review face assignment does not.
- Auth test: `RequireAuth` — a viewer can read the leaderboard; an unauthenticated request is 401.
