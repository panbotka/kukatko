# A real query language for search

KukĂˇtko's search is powerful but only reachable through structured parameters: `?album=`, `?label=`, `?person=`,
`?year=`, `?favorite=`. Everything the app knows â€” ISO, aperture, focal length, megapixels, altitude, codec,
camera, how many faces are in the shot â€” is stored and unqueryable. You cannot ask for "red photos with more than
three faces", or "everything shot on the 50 mm wide open", or "photos near this one".

Give the search box a query language, the way PhotoPrism has one. It is the single feature that turns a photo
library into a tool.

## The language

Free text plus `key:value` filters, mixed in one string:

`dovolenĂˇ camera:"Canon EOS R6" iso:100-400 faces:2`

**Filters** â€” implement the ones KukĂˇtko can actually answer from what it stores. Do not invent filters over data
that does not exist; a filter that silently matches nothing is worse than no filter.

- Text: `title:`, `description:`, `notes:`, `filename:`, `keywords:` / `subject:` (if the IPTC columns exist by
  then â€” if they do not, skip them cleanly rather than blocking)
- Organisation: `album:`, `label:`, `person:` / `subject:`, `favorite:`, `private:`, `archived:`, `rating:1-5`,
  `flag:pick|reject|eye`
- Time: `year:`, `month:`, `day:`, `taken:`, `before:`, `after:`, `added:`
- Place: `country:`, `city:`, `geo:yes|no`, `alt:300-500`, `near:<photo-uid>` (photos geographically near a given
  photo), `dist:` (km radius, pairs with `near:`)
- Camera / optics: `camera:`, `lens:`, `iso:200-400`, `f:2.8-4.5`, `mm:28-35`, `mp:3-6`
- Media: `type:image|video|live`, `codec:`, `portrait:`, `landscape:`, `square:`, `panorama:`
- Faces: `faces:yes|no|3` (a minimum count), `face:new` (has an unassigned face)

**Operators:**
- space between filters = **AND**
- `|` inside a value = **OR** (`label:cat|dog`)
- `!` = **NOT** (`label:!blurry`)
- `-` prefix on a term = NOT for free text
- ranges with `-` (`iso:200-400`), open ranges (`iso:800-`, `iso:-200`)
- `*` wildcard inside a text value
- quotes for values with spaces (`camera:"Canon EOS R6"`)
- `\` escapes a literal `|`, `!` or `-`

## Design rules

- **Parse into an AST, then compile to SQL.** Do not build a string-concatenating query builder â€” it is how
  injection bugs and unmaintainable filters happen. Everything goes through pgx parameters.
- **The parser is the thing to get right.** It is pure logic with no I/O, so cover it with a large table-driven
  test: every filter, every operator, quoting, escaping, ranges, malformed input. This is the part that will be
  extended for years; a weak parser here becomes a permanent tax.
- **Unknown key â†’ treat the whole token as free text**, do not error. `foo:bar` in a caption should still find the
  photo. But **do** return a machine-readable list of the tokens that were not understood, so the UI can show a
  gentle "nerozumĂ­m `foo:`" hint rather than silently returning nothing.
- **Free text keeps its current behaviour** â€” the existing Czech-aware full-text / semantic / hybrid search
  (RRF fusion) applies to the free-text part; the filters constrain the result set. A query with only filters and
  no free text is a pure filter query and must not go anywhere near the embedding sidecar.
- The existing structured query params **keep working** â€” this is additive. Do not break `?album=`, and do not
  break saved searches.
- **View state stays in the URL** (`?q=`), per the project's "back always works" rule.

## Frontend

- The search box accepts the language. Show the unrecognised-token hint inline.
- **Discoverability**: a small help affordance next to the box (a `?` popover or a link) listing the filters with
  one example each, in Czech and English. A query language nobody knows about is a query language nobody uses.
- Autocomplete for filter **keys** as you type (`ca` â†’ `camera:`, `city:`) is a big win and cheap â€” do it if it
  fits, but correctness of the parser comes first.

## Verification

- Exhaustive table-driven parser tests (this is the bulk of the work â€” treat it as such).
- Integration tests over a seeded library: each filter returns what it should; `|` widens; `!` excludes; ranges
  bound correctly; `near:` returns geographically close photos; `faces:3` counts markers; a filter-only query does
  not call the embedding sidecar; an unknown key degrades to free text and is reported.
- `make check` must pass.
- Document: `docs/API.md` (the `q=` grammar â€” this is a user-facing contract, write it out in full),
  `docs/FRONTEND.md`, `README.md` (headline feature), and the in-app help text (cs + en).

## Out of scope

- `color:` / `mono:` / `chroma:` â€” KukĂˇtko does not compute a colour palette; adding one is a separate task. Do
  not add a colour filter that cannot be answered.
- Changing the ranking of the free-text/semantic search.
