# Review Game Layout

The review ("Třídění") page at `/review` is visually broken: the photo overflows its container and collides with the question text. Fix the layout so the text sits above the photo and the whole view always fits the viewport without scrolling, in both portrait and landscape.

## Current behaviour and root cause

- `web/src/pages/ReviewPage.tsx` renders a fixed flex column, top to bottom: `header.review-game__top` → `div.review-game__progressbar` → `main.review-game__stage` (the photo) → `section.review-game__prompt` (question + face hint + confidence) → `footer.review-game__actions` (Ne / Nevím / Ano). The page sits outside the `Layout` shell (no navbar), like `/slideshow`.
- `web/src/components/review/review.css` sets `--review-chrome: 10rem` — a HARD-CODED ESTIMATE of the height of everything that is not the photo — and derives `--review-stage-h: calc(100dvh - var(--review-chrome))`.
- `web/src/components/review/ReviewPhoto.tsx` sizes the frame width-driven: `aspectRatio: ratio` plus `maxWidth: min(100%, calc(var(--review-stage-h) * ratio))`.
- There is NO absolute-positioned text overlay. The text is a normal sibling below the photo in flow. The overlap is a flex-overflow bug: when the real chrome exceeds the 10rem estimate, the photo's aspect-ratio-derived height exceeds the space `.review-game__stage` actually gets. The stage has `min-height: 0`, no `overflow: hidden`, and centres its content — so the photo bleeds up AND down, landing on the header and the prompt.
- Real chrome is taller than the guess because `@media (pointer: coarse)` in `app.css` bumps `.btn` to `min-height: 2.75rem`, the question uses `font-size: clamp(1.35rem, 3.5vw, 2.1rem)`, the actions are `size="lg"`, and `.review-game__alert` may appear.
- The only corrective query is `@media (max-width: 576px)` (chrome → 13rem). It is WIDTH-based, so a landscape phone — wide but short — never matches it and keeps the 10rem estimate while having the least vertical room. That is the sharpest edge of the bug. No `orientation:` media query exists anywhere in the app.
- `overflow-y: auto` on `.review-game` rarely saves it: a fixed-height flex column with a shrinkable stage prefers to shrink the stage rather than scroll.

## Requirements

- The question text moves ABOVE the photo. Order: header/progress → question (+ face hint + confidence) → photo → action buttons.
- The photo must never overlap any text, at any viewport size or orientation.
- The whole view fits the viewport: no vertical or horizontal scrolling, portrait and landscape. The user must never scroll to reach the Ne/Nevím/Ano buttons.
- The photo takes only the space remaining after text and buttons — it shrinks rather than pushing anything off-screen. On a short landscape viewport the photo may become small; that is correct. Text and buttons win.
- Do NOT keep a hard-coded chrome estimate. Either measure the real chrome (e.g. ResizeObserver over the non-photo elements, writing the measured value into the custom property) or restructure so the photo is bounded by a real `max-height` from the flex remainder and overflow is structurally impossible. Prefer the solution that cannot drift when text or button sizes change.
- Keep the face bounding-box overlay aligned. `web/src/lib/faceGeometry.ts` (`padBbox`, `faceBoxStyle`) positions boxes in normalised PERCENTAGES of the frame, and `ReviewPhoto.tsx` gives the frame the exact `aspect-ratio` so no pixel measurement is needed. Preserve that invariant — if the frame's box stops matching the rendered image, the face rectangles drift off the faces. `displayAspect()` computes the ratio in EXIF display space (orientations 5–8 swap w/h, fallback 1.5).
- Keep the image undistorted (`object-fit: contain` is the existing safety net).
