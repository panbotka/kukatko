# Grid density: 1–10 user-set columns, auto only seeds the initial value

Rework the photo-grid "images per row" density used by every photo list — Library, albums, labels, people, search, favorites, places. Today the control offers 1–8 columns plus a persistent **auto** mode (CSS `auto-fill`), which reads as a confusing "9th" option. Change it so the user always picks a concrete number of columns (scale **1–10**), and "auto" is used **only to seed the initial value from the screen width on first use** — after that it is whatever the user set, persisted in localStorage. Read `docs/FRONTEND.md` (the grid-density section) first.

## Current state

- Model `web/src/lib/gridDensity.ts`: `type GridDensity = 'auto' | number`, `GRID_COLUMNS_MIN=1`, `GRID_COLUMNS_MAX=8`, `GRID_TILE_MIN_PX=140`, `GRID_DENSITY_DEFAULT='auto'`, storage key `'kukatko.grid.density'`. Pure fns `sanitizeDensity`, `stepDensity`, `readStoredDensity`, `writeDensity`, `defaultDensityForViewport(isNarrow)` (returns `1` narrow / `'auto'` wide), `gridTemplateColumns(density, gapPx?)` (for `'auto'` emits `repeat(auto-fill, minmax(GRID_TILE_MIN_PX, 1fr))`).
- Hook `web/src/hooks/useGridDensity.ts`: `useSyncExternalStore` over localStorage; until the user picks a value the effective density comes from `defaultDensityForViewport` + `useIsNarrowViewport`. Returns `{ density, setDensity }`; also exports `setGridDensity()`.
- Control `web/src/components/library/GridDensityControl.tsx`: a −/+ stepper with a middle chip showing `A` (auto) or the number; clicking the chip resets to auto. Rendered inside `web/src/components/library/FilterBar.tsx`.
- Value consumers: `web/src/components/library/PhotoGrid.tsx` (VirtuosoGrid list `gridTemplateColumns` + a `data-density` attribute), `web/src/components/library/GridSkeleton.tsx`, `web/src/pages/SubjectPage.tsx`.

## Target behavior

- The density is **always a concrete integer number of columns in 1–10**. There is no persistent "auto" mode and no auto button/chip in the control.
- **First use (nothing stored in localStorage):** compute an initial column count from the current screen width — the same intent the old `'auto'` had, resolved to a concrete number (roughly how many tiles of about the min tile width fit across the available width), clamped to 1–10. Narrow phones still start at 1 (or 2). This initial value becomes the effective density and is persisted so it is stable thereafter.
- **After that:** the value is whatever the user set with the control, clamped to 1–10, persisted in localStorage under the existing key. Resizing the window must NOT override an existing value — auto is only the one-time seed.
- Keep the localStorage-backed, cross-tab-synced, per-device global behavior (a device preference, not URL state — unchanged).

## Requirements

- `web/src/lib/gridDensity.ts`: change `GridDensity` to a plain `number`; set `GRID_COLUMNS_MAX = 10`; remove the `'auto'` value and the `auto-fill` branch of `gridTemplateColumns` (it now always emits `repeat(n, 1fr)`); `sanitizeDensity` clamps to 1–10 and coerces any legacy stored `'auto'`/out-of-range value to a computed/clamped number; `stepDensity` steps within 1–10. Replace `defaultDensityForViewport(isNarrow)` with a width-based initial-columns function returning a concrete 1–10 count for a given viewport width (use `GRID_TILE_MIN_PX` as the target tile width). Update `GRID_DENSITY_DEFAULT` accordingly (a concrete fallback, or a helper that derives it).
- `web/src/hooks/useGridDensity.ts`: on first use with no stored value, derive the initial columns from the viewport width and persist it (seed localStorage) so it is stable; otherwise use the stored number. Remove the ongoing `'auto'` / `useIsNarrowViewport` dynamic recompute. Preserve `useSyncExternalStore` + cross-tab sync.
- `web/src/components/library/GridDensityControl.tsx`: remove the auto/reset affordance; the middle element just shows the current column count (1–10); −/+ step within 1–10 and disable at the bounds. Keep it prop-less, reading the shared hook.
- **Legacy migration:** an existing `'auto'` value already in a user's localStorage must resolve gracefully to a concrete number (compute from current width) on read rather than breaking.
- Update all consumers that special-cased `'auto'` (`PhotoGrid`, `GridSkeleton`, `SubjectPage`, and the `data-density` attribute PhotoGrid stamps for tests/keyboard nav — now always a number).
- i18n `library.density.*` in `web/src/i18n/locales/{cs,en}/common.json`: remove/replace the "Auto" label (keep the "columns per row" aria/tooltip strings, drop the auto-reset label). Czech default.

## Tests

- Vitest for `gridDensity.ts`: `sanitizeDensity` clamps to 1–10 and converts legacy `'auto'`/invalid to a number; `stepDensity` respects the 1–10 bounds; the width-based initial function returns sensible counts across representative widths (narrow → 1, very wide → up to 10); `gridTemplateColumns(n)` emits `repeat(n, 1fr)`.
- Vitest for `useGridDensity`: with empty storage the effective density is a concrete number seeded from width and persisted; a stored number is used verbatim; a stored legacy `'auto'` resolves to a number; no auto mode remains.
- Vitest for `GridDensityControl`: shows the current number, steps within 1–10, disables at the bounds, and no longer offers an auto/reset button.
