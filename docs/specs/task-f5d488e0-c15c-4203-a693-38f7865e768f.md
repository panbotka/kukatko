# Stacks — one shot, several files, one tile

Shoot RAW+JPEG and you get `IMG_1234.CR2` and `IMG_1234.jpg`. An iPhone Live Photo is `IMG_1234.HEIC` plus
`IMG_1234.MOV`. Export an edit from Lightroom and a second JPEG lands next to the original. Kukátko ingests each of
these as **its own photo**, so the same moment occupies two or three tiles in the grid, gets counted twice in every
album, and shows up twice in every search.

A stack groups those files into **one library item** with a **primary file** — the one you see — and the rest as
variants you can reach when you want them.

## The data model — grouping, not merging

There are two ways to do this and only one of them is safe.

**Do not merge the rows.** Folding the RAW's `photos` row into the JPEG's (moving its file into `photo_files` and
deleting the row) destroys information — the RAW's own dimensions, its EXIF, its embedding, its faces — and makes
unstacking impossible to do faithfully. `internal/dupmerge` does that deliberately for duplicates, where the loser
is genuinely redundant. Stack members are **not** redundant; you keep both on purpose.

**Group them instead.** Add to `photos` (new migration, use the next free number — check the directory):

- `stack_uid` — nullable; all members of a stack share it. NULL = not stacked.
- `stack_primary` — boolean; exactly one true per stack.

Every member keeps its own full row: its own metadata, embedding, faces, thumbnails. Stacking and unstacking then
become pure, reversible bookkeeping — set the column, clear the column. Nothing is ever lost, which is the property
that lets a user trust the feature enough to run the automatic detection over their whole library.

Note the existing `photo_files` table (roles `original` / `sidecar` / `edited`) is a **different** concept — the
files belonging to *one* photo. Do not conflate them.

## Detection

A job + a backfill (follow the `internal/processapi` `/process/*` pattern — admin-only, idempotent, resumable) that
groups existing photos by these rules, each **individually switchable in config**, because they have very different
false-positive rates:

1. **Same base filename, different extension** — `IMG_1234.CR2` + `IMG_1234.jpg`. The safest rule. On by default.
2. **Sequential / copy names** — `IMG_1234 (2).jpg`, `IMG_1234 copy.jpg`, `IMG_1234-edited.jpg`. On by default.
3. **Same EXIF unique id** — `ImageUniqueID` / XMP `InstanceID` where present. Very reliable when it exists.
4. **Same capture time to the second + same GPS** — the loosest rule, and the one that will wrongly stack burst
   shots taken in the same second. **Off by default**, and say why in the config comment.

Rules only ever group photos that plausibly *are* the same shot. When in doubt, do not stack — a wrongly stacked
photo is invisible, and an invisible photo is a lost photo. That asymmetry should drive every judgement call here.

**Primary selection:** prefer the file the user is most likely to want to look at — a JPEG/HEIC over a RAW, the
larger resolution over the smaller, the still over the video (for a Live Photo). Make the rule explicit and
documented, not emergent.

## Behaviour across the app

- **Grid, search, albums, labels, places, map**: only the **primary** of a stack appears. Non-primary members are
  hidden from the default view. This is the entire point — get the duplicates out of the grid.
- The tile of a stacked photo carries a **badge with the member count**, so it is obvious the item has variants.
- **Counts follow the same rule** — an album containing a 3-member stack shows one photo, not three. Audit every
  count in the app (album photo counts, label counts, subject photo counts, the year facet, the timeline, library
  stats) and make sure they agree. A count that disagrees with what the grid shows is a bug report waiting to
  happen; this is the fiddly part of the task, budget for it.
- **Photo detail**: a variants strip listing the stack's files (thumbnail, format, dimensions, size). Click one to
  view it. Two actions, editor-only: **set as primary** and **unstack** (remove that member from the stack — it
  becomes an ordinary standalone photo again).
- **Manual stacking**: from the grid selection, "stack selected" — for the cases the rules miss. And unstacking a
  whole stack at once.
- **Duplicates**: stack members must **not** be offered as near-duplicates of each other. A RAW and its JPEG are a
  textbook near-duplicate pair and the duplicates page would otherwise nag about every one of them forever. Exclude
  same-stack pairs from `internal/duplicates`.
- **Download**: downloading a stacked photo gives you the **primary** by default; the variants strip offers the
  others explicitly.
- **Live Photos**: Kukátko already has a `live` media type and a `LivePhoto` component — check how the still and
  the motion clip are paired today **before writing any code**. If they are already one photo, leave that path
  alone; if they are two rows, stacking is exactly the mechanism that fixes it, and they should stack
  automatically.

## Config

New keys (Config struct + `setDefaults` + `config.example.yaml` + tests + `docs/OPERATIONS.md`, all at once): a
master switch (default on), and one switch per detection rule with the defaults above.

## Verification

- Unit tests for each detection rule, including the ones that must **not** fire: two different photos that happen
  to share a second (rule 4 off), `IMG_1234.jpg` and `IMG_12345.jpg` (not a base-name match), a burst sequence.
- Integration: the backfill stacks a seeded RAW+JPEG pair and picks the JPEG as primary; the grid returns one row
  where there were two; **every count** (album, label, subject, year facet, library stats) drops accordingly;
  unstacking restores both rows fully, with their metadata, faces and embeddings intact; the duplicates scan does
  not offer the pair; re-running the backfill changes nothing.
- Vitest: the tile shows the member badge; the detail page lists the variants; set-primary and unstack work.
- `make check` must pass.
- Document: `docs/ARCHITECTURE.md` (the data model and — importantly — **why grouping rather than merging**, so a
  future iteration does not "simplify" it into a merge), `docs/API.md`, `docs/FRONTEND.md`, `docs/OPERATIONS.md` +
  `config.example.yaml`, `README.md`.

## Out of scope

- Any change to `internal/dupmerge` beyond excluding same-stack pairs from detection.
- Reorganising files on disk.
- Stacking across different capture events (bursts are not stacks — a burst is many real photos).
