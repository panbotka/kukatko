# Location estimates

Plenty of photos have no GPS — a camera without a receiver, a scan, a stripped export. But they were very often
taken on the same day, in the same place, as other photos that *do* have coordinates. PhotoPrism exploits this and
it is surprisingly effective: it fills the map, it makes the places hierarchy useful, and it costs nothing but a
query.

Do the same, honestly labelled.

## Behaviour

For a photo with a known `taken_at` but no `lat`/`lng`:

- Find photos **with GPS** taken close in time — the same day is the obvious window; a tighter window (a few
  hours) gives a better estimate. Make the window configurable.
- If those neighbours are **geographically coherent** (they cluster within a sane radius — a few kilometres), take
  their location. If the day's photos span Prague and Vienna, there is no honest estimate: **produce nothing**.
  A wrong location is worse than no location, because it silently poisons the map, the places hierarchy and any
  `near:` search built on it.
- Store the estimate in `lat`/`lng` **plus an explicit marker that it is an estimate** — a source field, mirroring
  the existing `taken_at_source` pattern (add a `location_source` column: `exif` / `manual` / `estimate` / empty).
  Never overwrite a location that came from EXIF or from the user.

**Skip photos where the estimate would be meaningless**: scans (the `scan` flag, if the metadata columns exist by
then), and photos whose date is itself an estimate — an estimated location derived from an estimated date is a
guess about a guess.

## Surfacing it — this is the part that must not be sloppy

An estimated location that looks identical to a real one is a lie the app tells the user.

- On the photo detail, an estimated location is **visibly marked as an estimate** (a label, not just a different
  shade), with a one-line explanation of where it came from ("odhad podle fotek z téhož dne").
- The user can **accept** it (promoting it to `manual`) or **clear** it, from the same place they edit the
  location today (`MetadataPanel`).
- On the map and in the places hierarchy, estimated locations are included by default (that is the point) but
  there is a way to see whether a pin is estimated.
- Bulk: not required.

## Mechanics

- A **backfill** over photos that lack a location, following the existing `internal/processapi` `/process/*`
  pattern (admin-only, idempotent, resumable). Re-running it must not re-estimate photos whose estimate was
  cleared by the user — a cleared estimate is a decision, and re-adding it every night would be maddening.
- Applying it on ingest is optional and probably not worth it (a freshly uploaded photo has no same-day
  neighbours yet). The backfill is the real path.
- The reverse-geocoding (`places` job) should run over newly estimated locations, so the places hierarchy picks
  them up — but mind the metered mapy.com credits: the existing rate limiting applies, do not blast the geocoder.

## Config

New keys (Config struct + `setDefaults` + `config.example.yaml` + tests + `docs/OPERATIONS.md`, all at once):
the time window, the coherence radius, and an on/off switch (default **on** — but it must be switchable off,
because someone will hate this feature).

## Verification

- Unit tests for the estimator: neighbours within the window and tightly clustered → an estimate is produced;
  neighbours 400 km apart → **no estimate**; no neighbours → no estimate; a photo with an existing manual location
  → untouched; a photo with an estimated date → skipped.
- Integration test: the backfill fills what it should, marks it as an estimate, is idempotent, and does not
  re-estimate a location the user has cleared.
- Vitest: the detail page marks an estimated location distinctly and offers accept/clear.
- `make check` must pass.
- Document: `docs/API.md` (the new field on the payload, the backfill endpoint), `docs/ARCHITECTURE.md` (the data
  model change and the honesty rule), `docs/OPERATIONS.md` + `config.example.yaml`, `docs/FRONTEND.md`,
  `README.md`.

## Out of scope

- Estimating a *date* from neighbouring photos (the inverse) — different feature, different risks.
- Any clustering heavier than "are these points close together".
