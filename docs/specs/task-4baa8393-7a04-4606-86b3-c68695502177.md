# Multi-album / multi-label filtering (AND)

Today the library/search filter lets the user pick at most one album and one label. Let the user progressively add several albums and several labels to the filter, combined with AND: a photo matches only if it is in **every** selected album **and** carries **every** selected label. (Example: "in album A AND album B, AND has label X AND label Y".)

## Backend (`internal/photos`, `internal/photoapi`)
- Change the album/label filter from scalar to multi-value:
  - `internal/photos/store_list.go`: `ListParams.AlbumUID string` → `AlbumUIDs []string`; `LabelUID string` → `LabelUIDs []string`.
  - `membershipClauses` (in `store_list.go`) currently emits one correlated `EXISTS` subquery for the single album and one for the single label. Change it to emit **one `EXISTS` per UID** — one per selected album, one per selected label. Because `buildWhere` joins all clauses with AND, N album-EXISTS + M label-EXISTS naturally yields the "member of all selected albums AND has all selected labels" semantics. Keep the correlated `album_photos` / `photo_labels` subquery shapes.
  - `internal/photoapi/params.go` (`applyFilters`): parse multiple values. Accept repeated query params (`?album=a&album=b&label=x&label=y`) via `q["album"]`/`q["label"]`. Preserve the existing behavior where setting an album forces chronological sort (apply it when at least one album is selected).
- This applies to both `GET /photos` (list) and `GET /search`, since both go through the same `ListParams`/`buildWhere` — verify search still scopes correctly.
- Keep backward compatibility: a single `album=`/`label=` value must still work.

## Frontend (`web/`)
- `web/src/lib/libraryView.ts`: `LibraryView.album`/`label` are single strings today. Change them to represent multiple selections. The URL-state layer (`web/src/lib/urlState.ts`) stores every key as a single string (`Record<string,string>`), so the least invasive encoding is a delimiter-joined list in the existing `album`/`label` keys (choose a delimiter that cannot appear in a UID, e.g. comma). Update `viewToParams` to emit the selected UIDs (either as repeated params or a form the backend parses), and `hasActiveFilters` to treat a non-empty list as active. Update `web/src/services/photos.ts` `buildPhotoQuery` / `PhotoListParams` to send multiple album/label values matching what the backend parses.
- `web/src/components/library/FilterBar.tsx` (`FacetRow`): the album and label controls use `SearchableSelect` (`web/src/components/library/SearchableSelect.tsx`), a single-choice combobox. Make selecting an album/label **add** to the current set rather than replace it (either make `SearchableSelect` multi-select, or keep it single-pick but append the chosen value to the list and show the already-selected ones as removable chips). Users must be able to add and individually remove each album/label.
- `web/src/components/library/filterChips.ts`: render **one chip per selected album** and **one per selected label**; clicking a chip's remove removes just that one from the list (not the whole facet).
- Facet option source is `web/src/hooks/useLibraryFacets.ts` (catalog-wide albums/labels) — reuse it.

## Edge cases
- Empty selection = no album/label filtering (unchanged).
- Removing the last chip clears that facet.
- "Back always works": the full multi-selection must round-trip through the URL query params so navigating back restores it.

## Tests (mandatory)

- `params.go` parsing test for repeated params.
- Frontend: FilterBar/libraryView tests for adding/removing multiple albums & labels and the URL round-trip.
