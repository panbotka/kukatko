# Add a "regenerate thumbnail" service action to the photo detail

Sometimes a photo's thumbnail is missing or stale — the ingest thumbnailing failed (e.g. the box
was offline, an unsupported format at the time, a transient error), and the tile shows a broken or
placeholder image with no way to recover short of re-uploading. Add a per-photo service action that
regenerates the thumbnail (and its perceptual hashes) on demand from the photo detail page.

## Current state (for context)

- Thumbnails and pHashes are produced by the `thumbnail` worker handler (`internal/thumbjob`),
  normally enqueued during ingest (`internal/ingest`). The persistent job queue is
  `internal/jobs`; admin backfills live in `internal/processapi` (`/process/*`).
- The photo detail page is `web/src/pages/PhotoDetailPage.tsx`; technical facts render in
  `web/src/components/photo/TechnicalDetails.tsx`.

## Requirements

- **Backend:** add an endpoint that (re)generates the thumbnail for a single photo by enqueuing (or
  running) a `thumbnail` job for that photo's UID, reusing the existing thumbnailer and job queue
  rather than duplicating logic. It must be idempotent (safe to click repeatedly), overwrite the
  existing thumbnail cache entries, and also refresh the perceptual hashes the thumbnail job
  computes. Return a clear result (job enqueued / done, or a typed error if the original is missing
  or the format is undecodable). Follow the existing API conventions under `/api/v1` and the
  project's RBAC — allow editors and admins (the same audience that can edit a photo); viewers
  cannot trigger it. Record it in the audit trail if that is how other mutations are recorded.
- **Frontend:** add a clearly-labelled service button on the photo detail — placed with the
  technical/maintenance details, not among the primary organize controls — that calls the endpoint.
  It shows a pending state while running, reports success or the returned error, and on success
  refreshes the displayed image/thumbnail (cache-bust so the new thumbnail actually shows without a
  hard reload). The button is only visible to editors/admins.
- If regeneration is asynchronous (enqueued job), the UI should communicate that it was scheduled
  and, where practical, reflect completion (e.g. re-fetch the photo / thumbnail after the job runs).
- Do not delete or alter the original file — this only rebuilds derived thumbnails/hashes.

## Verification

`make check` must pass. Add a backend test that the endpoint enqueues/runs a thumbnail job for the
given photo and enforces RBAC (viewer forbidden), and a frontend test that the button calls the
endpoint, shows pending/success/error states, and refreshes the thumbnail on success. Document the
new endpoint in `docs/API.md` and the button in `docs/FRONTEND.md`.
