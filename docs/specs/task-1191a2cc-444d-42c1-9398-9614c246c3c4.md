# Import photo-sorter's 1:1 embeddings + face vectors (no GPU recompute)

**Production reality (verified live against sorter.kotrzina.cz + fotky.kotrzina.cz):**
the photos, originals, files, albums, labels and metadata live in **PhotoPrism**;
**photo-sorter runs as a vector/faces LAYER on top of PhotoPrism** — its native
photos table is EMPTY (`GET /api/v1/photos` returns `total:0`) and it holds only
embeddings (~20,687) and faces (~112,806) keyed by the **PhotoPrism photo UID**
(confirmed: a UID from the sorter feed resolves 200 in PhotoPrism). So Kukátko must
import PHOTOS from PhotoPrism (existing `ppimport`) and only **enrich them with
photo-sorter's already-computed vectors** — NOT import photos from photo-sorter.

This eliminates the dependency on the often-offline GPU box for the whole migrated
library (the box would otherwise recompute all ~20k embeddings + ~112k faces).

## The photo-sorter migration feeds (verified working with the read-only psat_ token)
- `GET /api/v1/embeddings` — keyset-paginated (`limit`, `next_after`, `total`),
  items: `photo_uid`, `model` (`ViT-L-14`), `pretrained` (`laion2b_s32b_b82k`),
  `dim` (768), `created_at`, `embedding` (the vector). Model + dim MATCH Kukátko's
  CLIP ViT-L/14 768-dim → copy **verbatim, 1:1**.
- `GET /api/v1/faces` — items: `photo_uid`, `face_index`, `model`
  (`buffalo_l (ResNet100)`), `dim` (512), `bbox`, `det_score`, `marker_uid`,
  `subject_uid`, `subject_name`, `file_uid`, `photo_width/height`, `orientation`,
  `embedding` (512-dim vector).
- `GET /api/v1/stats` — totals (`total_photos`, `photos_with_embeddings`,
  `total_faces`, ...) for the completeness check.
- Auth: `Authorization: Bearer psat_...` (read-only). Config: wire
  `import.photosorter.base_url` + `import.photosorter.token`
  (env `KUKATKO_IMPORT_PHOTOSORTER_BASE_URL` / `KUKATKO_IMPORT_PHOTOSORTER_TOKEN`) —
  currently defined-but-unused; add to the config struct, defaults,
  `config.example.yaml`, docs.

## Requirements
- New photo-sorter FEEDS importer (HTTP client behind an interface, faked in tests):
  page `/embeddings` and `/faces`, and store each into Kukátko's `internal/vectors`
  (embeddings) and faces/markers/subjects, **attached to the Kukátko photo whose
  `photoprism_uid` == the feed's `photo_uid`**, copied 1:1 (no recompute).
- Carry the face markers + subject assignments from the faces feed (marker_uid /
  subject_uid / subject_name) so people/faces come across, not just raw vectors.
- Handle a feed entry whose photo has not (yet) been imported from PhotoPrism:
  skip-and-reconcile or defer, do not error the whole run.
- Idempotent, incremental, safe to run against an empty DB (part of the wipe + full
  reimport). External API behind an interface → fake in tests.
- This REPLACES the assumption that photo-sorter is imported as a photo source; the
  old direct-DB `psimport` path becomes irrelevant for this deployment (leave it, but
  it is not the migration path).
