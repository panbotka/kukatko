# /faces — find a person in untagged photos

The backend can now answer "where else does this person appear, unnamed?"
(`POST /api/v1/subjects/{uid}/candidates`). Build the page that makes it fast to work through the answer.

**Confirm the endpoint exists before starting.** If it does not, the backend task has not run yet — do not
build against a guess; implement the smallest honest version of it first or stop and say so.

## The page

New route **`/faces`** (editor-only, like `/people/clusters`), linked from the people section of the nav.

**Config panel at the top:**
- Person picker — typeahead over subjects (reuse the existing `AddAutocomplete` pattern), showing each
  person's photo count. This is the one required input.
- **Threshold slider** — the whole UI speaks **percent**, never cosine distance. Range 20–80 %, step 5,
  default 50 %. Convert to distance for the API (`distance = 1 - percent/100`). Label the live value and
  bookend the slider with **"Více výsledků" ↔ "Lepší shody"** — the trade-off must be legible without a
  manual.
- Limit — number input, 0 = no limit.
- A Search button. The search is explicit, not live-on-drag — it is an expensive query.

**Results:**
- A stats row: source photos, source faces, matches found, already done, and **the computed
  `min_match_count`** with a one-line explanation ("Fotka musí odpovídat alespoň {{n}} zdrojovým fotkám").
  Do not hide the filter that shaped the results.
- A grid of **large thumbnails with the candidate face drawn as a coloured rectangle over the photo** — not
  cropped face chips. You have to see the context to judge whether it is really them. Reuse the existing
  `FaceOverlay` geometry helpers (`web/src/lib/faceGeometry.ts`) rather than reinventing bbox maths.
- Colour-code by action and keep the code **consistent across badge, rectangle and legend**: needs a new
  marker / needs a person / already done. Show a legend.
- Filter tabs with counts (Vše / Nové / Přiřadit / Hotovo) that also scope what "Confirm all" applies to.
- Each card shows the match percentage and, when > 1, the number of source photos that voted for it.

## Confirming — this is the point of the page

- **Per card: ✓ and ✗.** ✓ confirms (calls the existing assign endpoint). ✗ **rejects and persists it** via
  the feedback endpoint (`POST /api/v1/feedback/face-rejections`) — this is the difference from photo-sorter,
  where a rejection evaporated and the same face came back on the next search. After a ✗ the card leaves the
  grid and never returns.
- **Optimistic, in place.** On ✓ the card flips to the "done" state where it stands — it does not vanish, the
  list does not reload, the scroll position does not jump. The stats counters update locally.
- **Keyboard.** Photo-sorter has no keyboard flow here and it is its biggest weakness — you approve hundreds
  of faces with a mouse. Kukátko already has a shortcut registry (`web/src/lib/shortcuts.ts`,
  `useKeyboardShortcuts`) and a grid navigation hook. Wire it up: arrows / `j k h l` move the focused card,
  **`y` or `Enter` = confirm**, **`n` = reject**, focus advances to the next actionable card automatically.
  Register the shortcuts in the existing `?` help overlay. Shortcuts must not fire while a text input is
  focused.
- **"Potvrdit vše (n)"** — confirms every actionable card in the current filter tab, sequentially, with the
  button itself showing live `current/total` progress. Filter tabs disable while it runs. It must be
  cancellable, and a failure partway through must leave the already-confirmed ones confirmed and report what
  failed — not roll back and not silently stop.

## States

- Nothing searched yet: an explanatory empty state, not a blank page.
- The subject has no faces yet: say so plainly — "nejdřív u téhle osoby pojmenuj pár obličejů".
- Zero matches: "žádné nové obličeje", with a hint that lowering the threshold widens the search.
- Loading: the search can take seconds on a big library — show progress, keep the config panel usable.

  ✓ calls assign and flips the card in place without a refetch; ✗ calls the rejection endpoint and removes
  the card; keyboard confirm/reject work and advance focus; "Confirm all" walks the filtered set and reports
  a partial failure; the empty/no-faces/zero-match states each render.
- `make check` must pass (ESLint strict, Prettier, Vitest, typecheck).
- Document: `docs/FRONTEND.md` (the page, its components and hooks) and `README.md` — this is a user-visible
