# Per-User Ratings & Flags — Data Layer

Add per-user star ratings (0–5) and pick/reject flags for photos, mirroring the existing per-user favorites design (`user_favorites`). This task covers ONLY the database schema, store methods, list filtering/sorting, and the `Photo` model annotation fields. The HTTP API/bulk and the frontend are separate tasks — do not implement them here.

## Requirements
- New SQL migration under `internal/database/migrations/` using the next free migration number. Table `user_ratings`: `user_uid` (FK users `ON DELETE CASCADE`), `photo_uid` (FK photos `ON DELETE CASCADE`), PRIMARY KEY `(user_uid, photo_uid)`, `rating SMALLINT NOT NULL DEFAULT 0 CHECK (rating BETWEEN 0 AND 5)`, `flag TEXT NOT NULL DEFAULT 'none' CHECK (flag IN ('none','pick','reject'))`, `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()`. A row exists only when a user has set a non-default rating or flag.
- Store methods in `internal/organize` (the same store that owns favorites):
  - `SetRating(ctx, userUID, photoUID, rating)` and `SetFlag(ctx, userUID, photoUID, flag)` — idempotent upsert. When both rating becomes 0 and flag becomes 'none', delete the row.
  - `GetRating(ctx, userUID, photoUID)` → rating + flag.
  - `RatingsAmong(ctx, userUID, photoUIDs)` → map `photo_uid → {rating, flag}` so a whole page can be annotated in one query (mirror `FavoritedAmong`). Missing rows default to rating 0 / flag 'none'.
  - Map FK violations to the existing not-found sentinels (photo/user) the favorites methods already use.
- `photos.ListParams`: add `RatedBy *string` (the current user uid, scopes annotation/filter), `MinRating *int` (filter rating ≥ n via correlated EXISTS on `user_ratings` for that user), `Flag *string` ('pick'/'reject'). Add a `rating` sort option (by that user's rating, NULLS LAST). All filters must go through the shared `buildWhere` so `Count` stays correct. Photos with no rating row count as rating 0 / flag 'none'.
- Annotate the `photos.Photo` model with per-user `Rating int` and `Flag string` fields (JSON `rating`/`flag`), analogous to `is_favorite`. Populating them in HTTP handlers is the API task — here just add the fields and the store support.
- Integration tests against the real test DB: store upsert/delete semantics, `RatingsAmong`, the `MinRating`/`Flag` filters, and the `rating` sort. Unit tests where logic is pure.
